## Preamble

Node 1.5 is the most mathematically dense document in SDA-1. Every agent in SDA-2 receives this feature object — a silent bug here propagates undetected through the entire diagnostic pipeline. Four issues are raised: two CRITICAL (wrong formula producing incorrect clinical measurements, unreliable signal processing on too-short a window), two REQUIRED (undefined behavior for edge cases that will be hit on real data), and one REQUIRED (unit inconsistency that will produce an implementation error).

## Issue 1 — Q-Wave Duration Formula Is Wrong; Will Produce False Pathological-Q Flags

**Severity: CRITICAL**

**Location:** Section 6.2, Q-wave analysis:

```
Q_duration_ms = (R_idx - QRSon_idx) / 500 * 1000  # approximation
```

**The flaw:**

This formula computes the time from QRS onset to the R peak — not the duration of the Q wave. In a normal QRS with a brief Q wave (10–15 ms) followed by a tall R wave (80–100 ms QRS width), `R_idx - QRSon_idx` will be ~80–100 samples = 160–200 ms. The Q-wave duration threshold is 40 ms. Almost every beat will be falsely flagged as `pathological_q = True`.

The actual Q-wave duration is the time from QRSon to the Q nadir:
```
Q_duration_ms = (Q_peak_idx - QRSon_idx) / 500 * 1000
```

Where `Q_peak_idx` is the sample index of the Q trough — the `Q` column (column 4) of the FPT. This value is already available in the FPT. The formula in the document confuses the Q-to-R interval (a different, clinically meaningless measurement) with Q-wave duration.

**Consequence:** The MR agent's pathological-Q detection (old infarction criterion) will fire on the majority of ECGs regardless of actual Q-wave width. The IT agent's inferior/anterior MI exclusion logic will also be corrupted.

**Required action:** Replace `(R_idx - QRSon_idx)` with `(Q_idx - QRSon_idx)` for Q-wave duration. Guard against `Q_idx == -1` (no Q wave detected in this lead for this beat → return `q_duration_ms = None`). The `is_pathological_q()` function should also guard: if `Q_idx == -1` and `R_amplitude_mv >= 0.1`, it is not a QS pattern, so return `False` (no Q wave present, not pathological).

## Issue 2 — fQRS Detection Applies `filtfilt` to a ~50-Sample Window; Edge Artifacts Dominate

**Severity: CRITICAL**

**Location:** Section 6.2, `detect_fragmented_qrs()`:

```python
b, a = butter(2, 40 / (fs/2), btype='high')
hf = filtfilt(b, a, qrs_segment)
peaks, props = find_peaks(np.abs(hf), height=0.1, distance=int(0.01 * fs))
return len(peaks) >= 2
```

**The flaw:**

At 500 Hz, a QRS segment spans approximately 70–120 ms = 35–60 samples. A Butterworth highpass at 40 Hz (`Wn = 40/250 = 0.16`) has a filter order of 2 (padlen = 3*(max(len(a)-1, len(b)-1)) = 9 samples minimum for `filtfilt`). For a 35–60 sample segment, the edge padding required by `filtfilt` is a substantial fraction of the signal. More critically, a 40 Hz highpass on such a short window is intrinsically unstable: the passband of interest (40–250 Hz notch content) spans only ~1–2 complete cycles in a 60-sample window. The resulting `hf` is dominated by filter startup/endpoint artifacts, not QRS notches.

Additionally, `find_peaks(np.abs(hf), height=0.1)` will reliably find the filter-artifact peaks at the segment edges regardless of whether any notch is present.

**Consequence:** fQRS will be detected on smooth QRS complexes wherever the filter edge artifact exceeds 0.1 mV — i.e., frequently. The MR agent uses fQRS as a marker of myocardial scar; false positives here corrupt its prior.

**Required action:** Replace the filtfilt approach with a direct time-domain notch counter: count local extrema (sign changes of the first derivative) within the QRS segment that exceed a height threshold (e.g., 0.05 mV above local baseline), excluding the primary R-peak itself. Specifically:

```python
def detect_fragmented_qrs(qrs_segment: np.ndarray) -> bool:
    """
    Counts directional reversals (notches) within QRS using sign changes of
    the derivative, excluding the primary deflection. Each notch must exceed
    0.05 mV in amplitude. fQRS = >= 2 notches.
    """
    if len(qrs_segment) < 5:
        return False
    d = np.diff(qrs_segment)
    sign_changes = np.where(np.diff(np.sign(d)))[0] + 1
    # Filter by amplitude: each turning point must exceed 0.05 mV relative to neighbors
    notch_count = 0
    for idx in sign_changes:
        local_amp = abs(qrs_segment[idx] - np.mean(qrs_segment[max(0,idx-3):idx+4]))
        if local_amp > 0.05:
            notch_count += 1
    # Exclude primary R peak (largest amplitude turning point)
    return (notch_count - 1) >= 2
```

This is implementable, edge-artifact-free, and matches the clinical definition (notches, not high-frequency content).

## Issue 3 — ST Isoelectric Window Can Reference Samples in the Filtfilt Edge-Transient Zone

**Severity: REQUIRED**

**Location:** Section 5.1:

```
isoelectric_reference_mv[beat, lead] = mean(morphology_signal[QRSon-10:QRSon, lead_idx])
```

**The flaw:**

`morphology_signal` is captured before any filter is applied (raw amplitude, 500 Hz grid — correctly specified in Node 1.2). The edge-transient concern does not apply to `morphology_signal` directly.

However, two real problems exist:

1. **Index underflow**: If the first beat's `QRSon` index is less than 10 samples from sample 0, `QRSon - 10` is negative. Python silently wraps negative slice indices, reading from the end of the array — returning a completely wrong baseline value for the most critical beat.

2. **Window falls within P-wave or T-wave of preceding beat**: For short PR intervals (< 120 ms, i.e., WPW), the 10-sample (20 ms) window immediately before QRSon may lie within the P wave, producing an elevated isoelectric reference that understates the true ST elevation.

**Required action:** (1) Add bounds check: `iso_start = max(0, QRSon_idx - 10)`. If `iso_start == QRSon_idx`, fall back to the TP segment of the preceding beat or return `None` for this beat's ST measurements. (2) Add PR-check: if `PR_interval_ms < 60` (WPW or extremely short PR), flag `isoelectric_reference_unreliable = True` for this beat, and downstream agents should not use ST measurements for this beat.

## Issue 4 — AFib State Leaves `heart_rate_atrial_bpm` and `av_ratio` Undefined; No Fallback Specified

**Severity: REQUIRED**

**Location:** Section 1.1 and 1.3.

When `afib_suspected = True`, Node 1.4's `_apply_condition_corrections()` sets all Ppeak columns to -1. Consequently, there are no P-P intervals, so `heart_rate_atrial_bpm` is undefined. Then `av_ratio = heart_rate_ventricular_bpm / heart_rate_atrial_bpm` produces a division-by-zero at runtime.

The document specifies the formulas but contains no guard for this state and no null sentinel for `heart_rate_atrial_bpm`. The feature object schema in Section 11 lists both fields without noting that they can be null.

**Required action:** Add explicit handling: when `afib_suspected = True` OR `count(Ppeak != -1) < 2`, set `heart_rate_atrial_bpm = None` and `av_ratio = None`. Document this in Section 11 as nullable fields. The IT agent must check nullability before using these for AV block classification.

## Issue 5 — P Terminal Force V1 Uses Inconsistent Units; Implementation Will Be Wrong by Factor of 1000

**Severity: REQUIRED**

**Location:** Section 6.1:

> P terminal force V1 = |amplitude of negative terminal deflection| * duration of negative deflection (ms)
> P terminal force > 40 μV·s (0.04 mV·s) = left atrial enlargement (Morris criterion)

**The flaw:**

The formula multiplies amplitude in mV by duration in milliseconds → result is in mV·ms.

The threshold is stated as 40 μV·s. Units: 40 μV·s = 0.04 mV·s = 40 mV·ms (since 1 s = 1000 ms, 0.04 mV·s = 40 mV·ms). The threshold in mV·ms is therefore 40 — not 0.04.

But the document states `> 0.04 mV·s` in one place, which a developer will read as 0.04 and compare to the mV·ms result (40). The comparison fails silently: P terminal force will never exceed 0.04 mV·ms in any patient; LAE will never be detected.

**Required action:** Standardize to a single unit system. Recommended: compute in mV·s (amplitude in mV × duration in seconds) → threshold is 0.04 mV·s. Remove the μV·s notation entirely. Add the unit explicitly to the feature schema field: `p_terminal_force_v1_mv_s` (not `mv_ms`).

Node 1.5 is the most complete ECG feature specification in the literature of this project — it covers every clinical criterion with mathematical precision.
