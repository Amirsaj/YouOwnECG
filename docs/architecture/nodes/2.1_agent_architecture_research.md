## Executive Summary

This document is the foundational research that determines the optimal agent architecture for the YouOwnECG diagnostic system. It synthesizes reasoning complexity analyses from all 13 STAT condition disease files, evaluates six candidate architectures through rigorous first-principles analysis, and recommends a single architecture based on a weighted decision matrix.

The recommended architecture is **Option E: Hybrid Parallel Specialists + Sequential Cross-Domain Reviewer**. This architecture achieves the best weighted score (8.00/10) across clinical accuracy, cross-domain reasoning, explainability, latency, robustness, scalability, and DeepSeek compatibility. It combines the parallelism and focused expertise of domain specialists with the cross-domain synthesis capability of a sequential reviewer layer, directly addressing the most critical finding from the disease file analysis: **11 of 13 STAT conditions require cross-domain reasoning**, and the hardest conditions (VT at 4.3, posterior STEMI at 3.8, de Winter at 3.7, Long QT/TdP at 3.7, Complete AVB at 3.7) demand integration across 3-5 reasoning domains simultaneously.

## 1. How Cardiologists Actually Read ECGs

### 1.1 The Systematic Approach

Expert cardiologists follow a structured, reproducible reading protocol. While individual preferences vary, the core sequence is remarkably consistent across training programs and international guidelines:

**Step 1: Rate**
- Calculate ventricular rate from RR interval
- Calculate atrial rate from PP interval if different
- Normal: 60-100 bpm. Tachycardia >100. Bradycardia <60.
- This is always first because rate immediately frames clinical urgency

**Step 2: Rhythm**
- Is it regular or irregular?
- If irregular: regularly irregular (e.g., Wenckebach, grouped beating) or irregularly irregular (AFib)?
- Is each P-wave followed by a QRS? Is each QRS preceded by a P-wave?
- Is there AV dissociation?
- P-wave morphology: sinus origin? Ectopic? Absent?

**Step 3: Axis**
- QRS axis determination from leads I and aVF (quick quadrant method)
- Normal: -30 to +90 degrees
- Left axis deviation: -30 to -90 (LAFB, LVH, inferior MI)
- Right axis deviation: +90 to +180 (LPFB, RVH, PE, lateral MI)
- Northwest axis: -90 to -180 (VT, severe conduction disease)

**Step 4: Intervals**
- PR interval: 120-200 ms (short → WPW or junctional; long → AV block)
- QRS duration: <120 ms normal; 120-150 ms incomplete BBB; >150 ms complete BBB
- QT/QTc interval: sex-specific thresholds; >500 ms = TdP risk

**Step 5: Morphology (lead-by-lead)**
- P-wave morphology: LAE (notched in II, deep negative in V1), RAE (peaked in II)
- QRS morphology: Q-waves (pathological?), R-wave progression (V1-V6), BBB patterns, voltage criteria for hypertrophy
- ST segment: elevation, depression, morphology (concave, convex, horizontal, upsloping, downsloping)
- T-wave: amplitude, symmetry, inversions, hyperacute features, peaked features
- U-waves: prominent? merged with T?

**Step 6: Synthesis**
- Integrate all findings into a coherent clinical picture
- Apply pattern recognition: does this combination of findings match a known syndrome?
- Consider differentials: what else could produce this pattern?
- Assign confidence levels
- Generate the clinical interpretation

### 1.2 Cognitive Process Analysis: Expert Mental Models

Cardiologists do not simply follow the 6-step protocol mechanically. Expert ECG reading involves several distinct cognitive operations:

**1. Pattern Matching (System 1 — Fast)**
Experienced readers recognize gestalt patterns in milliseconds. A "tombstone" STEMI, a classic AFib with RVR, or a complete heart block with wide escape are identified before the structured analysis begins. This is akin to a radiologist seeing a pneumothorax on a chest X-ray before formally reviewing each field.

Evidence from the disease files:
- Anterior STEMI (difficulty 2.5): "ST elevation in STEMI is a large signal (2-10+ mm) — detectable even in moderately noisy tracings" (stemi_anterior.md, Section 6.6)
- VF (difficulty 2.7): "The primary reasoning domain — recognizing the absence of organized cardiac complexes" (ventricular_fibrillation.md, Section 6.1)

**2. Territorial Mapping (Spatial Reasoning)**
Experts mentally map ECG lead groups to coronary territories:
- V1-V4 → LAD → anterior wall
- II, III, aVF → RCA/LCx → inferior wall
- I, aVL, V5, V6 → LCx/diagonal → lateral wall
- V1-V3 reciprocal → posterior wall

This spatial reasoning is essential for STEMI localization and appears in all 5 STEMI disease files.

**3. Conditional Branching (If-Then Reasoning)**
Experts apply conditional logic at multiple decision points:
- IF LBBB detected → THEN switch to Sgarbossa criteria for STEMI assessment
- IF peaked T-waves → THEN check if territorial (STEMI) or diffuse (hyperkalemia)
- IF wide QRS tachycardia → THEN apply Brugada criteria for VT vs SVT
- IF T-wave inversions in V2-V3 → THEN check if symmetric (Wellens) or asymmetric (LVH strain)

Evidence: Every disease file documents sequential reasoning chains with explicit conditional branches.

**4. Negative Evidence Weighting**
Expert readers assign diagnostic weight to the ABSENCE of features:
- No reciprocal changes in setting of ST elevation → less likely STEMI, consider pericarditis
- No Q-waves in setting of deep T-wave inversions → supports Wellens over completed MI
- No P-waves with regular QRS → consider complete AVB, sinoventricular rhythm, or junctional rhythm

Evidence: Wellens syndrome (wellens_syndrome.md, Section 6.1): "Must confirm ABSENCE of ST elevation, ABSENCE of pathological Q waves, ABSENCE of significant troponin elevation. These negative findings are as important as the positive T-wave findings."

**5. Cross-Domain Integration (The Expert Leap)**
The most challenging cognitive task is integrating findings across domains. This is where novices and algorithms fail:

- Seeing LBBB → automatically reassessing all ST/T findings through the Sgarbossa lens
- Detecting inferior STEMI → immediately looking for RV extension (V1, V4R) and posterior extension (V1-V3 depression)
- Finding peaked T-waves → checking if distribution is territorial (ischemia) or diffuse (metabolic) → then checking QRS width (hyperkalemia progression)
- Recognizing wide QRS tachycardia → simultaneously assessing morphology (VT vs SVT+aberrancy), AV relationship, axis, and concordance

This cross-domain integration is the most computationally demanding aspect of ECG interpretation and the primary driver of the architecture decision.

### 1.3 Where Experts Make Cross-Domain Leaps

The disease files reveal specific cross-domain leap points that the architecture must handle:

| Trigger Finding | Cross-Domain Leap | Disease File Evidence |
|----------------|-------------------|----------------------|
| LBBB detected | Switch STEMI assessment to Sgarbossa criteria | stemi_anterior.md: "IF LBBB is present, THEN switch to Sgarbossa criteria rather than standard STEMI criteria" |
| Inferior STEMI detected | Check for RV extension (V1 elevation) + posterior extension (V1-V3 depression) + AV block risk (PR interval) | stemi_inferior.md, stemi_right_ventricular.md, stemi_posterior.md |
| Peaked T-waves | Territorial vs diffuse distribution → STEMI vs hyperkalemia branching | hyperkalemia.md: "If peaked T is territorial (V1-V4 only) → consider anterior STEMI" |
| Wide QRS tachycardia | Morphology + AV relationship + axis + concordance integration | ventricular_tachycardia.md: "Rate + rhythm + morphology + axis + AV relationship + algorithmic criteria must be integrated" |
| ST depression V1-V3 | Anterior subendocardial ischemia vs posterior STEMI (mirror image reasoning) | stemi_posterior.md: "Must understand that ST depression in V1-V3 can represent EITHER anterior subendocardial ischemia OR posterior STEMI" |
| T-wave inversions V2-V3 | Symmetric (Wellens) vs asymmetric (LVH strain) → ischemia domain vs structural domain | wellens_syndrome.md: "Requires BOTH morphology expertise AND ischemia territory knowledge" |
| Coved ST V1-V3 | Brugada vs RV STEMI vs RBBB vs hyperkalemia pseudo-Brugada | brugada_type1.md: "Must differentiate Brugada Type 1 from RBBB, ARVC, early repolarization, hyperkalemia pseudo-Brugada" |
| Prolonged QT + PVCs | Assess TdP initiation risk via short-long-short sequences | long_qt_tdp.md: "PVCs in the setting of long QT should trigger a STAT alert even if the PVCs themselves would otherwise be benign" |
| Complete AVB | Ischemic context changes prognosis: inferior MI → transient; anterior MI → permanent | complete_av_block.md: "Complete AVB + inferior STEMI → likely transient; complete AVB + anterior STEMI → likely permanent, much higher mortality" |

These cross-domain leaps define the minimum communication requirements between any agents in the system.

### 2.2 Conditions Clustered by Reasoning Similarity

**Cluster 1: Territorial Ischemia / STEMI Group**
- Anterior STEMI, Inferior STEMI, Lateral STEMI, Posterior STEMI, RV STEMI
- Shared domains: D5 (ST analysis), D7 (lead-group correlation), D11 (amplitude measurement), D10 (differential), D6 (T-wave)
- Shared reasoning pattern: Detect ST changes → verify territorial distribution → identify reciprocal changes → exclude mimics → classify territory and culprit vessel
- Key differentiator: posterior STEMI uses "mirror image" reasoning (D5 inverted logic)

**Cluster 2: STEMI Equivalents / Subtle Ischemia**
- Wellens Syndrome, de Winter T-waves
- Shared domains: D6 (T-wave morphology — primary), D7 (lead-group), D12 (negative evidence), D5 (ST — for exclusion)
- Shared reasoning pattern: Detect non-standard ischemia pattern → verify specific morphology → confirm negative criteria → flag as STEMI equivalent
- Key differentiator: these conditions are diagnosed by what is NOT present (no ST elevation in Wellens, no ST elevation in de Winter despite LAD occlusion)

**Cluster 3: Arrhythmia / Electrical Instability**
- Ventricular Tachycardia, Ventricular Fibrillation, Long QT with TdP
- Shared domains: D1 (rate), D2 (rhythm regularity), D4 (QRS morphology), D9 (temporal)
- Shared reasoning pattern: Detect rhythm abnormality → classify mechanism → assess hemodynamic significance → identify substrate
- Key differentiator: VT requires the most cross-domain reasoning (D1+D2+D3+D4+D7+D9+D10); VF is primarily pattern recognition; Long QT adds interval precision (D8) and beat-to-beat temporal analysis (D9)

**Cluster 4: Conduction Disease**
- Complete (3rd Degree) AV Block
- Domains: D1 (rate), D2 (rhythm regularity), D3 (AV relationship — primary), D4 (QRS morphology), D9 (temporal — per-beat PR analysis)
- Unique reasoning: requires proving a negative relationship (no fixed PR interval) which demands analysis of EVERY beat

**Cluster 5: Metabolic / Channelopathy**
- Hyperkalemia, Brugada Type 1
- Shared domains: D6 (T-wave/morphology), D8 (intervals), D7 (lead-group — diffuse vs localized)
- Shared reasoning pattern: Detect characteristic morphology → assess spatial distribution → exclude ischemic mimics → stage severity
- Key differentiator: hyperkalemia is DIFFUSE (whole heart); Brugada is LOCALIZED (V1-V3 only). They can mimic each other (pseudo-Brugada in hyperkalemia)

## 3. Six Architecture Alternatives (Full Analysis)

### 3.1 Option A: Domain-Specialist Parallel Agents

**Description:**
5-7 specialist agents run simultaneously, each focused on a single reasoning domain. Each agent receives the full feature set from SDA-1 but is prompted to focus on its domain. Results are collected and merged by a lightweight aggregator.

**Proposed Specialists:**
1. Rhythm & Rate Agent (D1, D2)
2. Conduction Agent (D3, D4, D8)
3. Ischemia / ST Agent (D5, D7, D11)
4. Morphology / T-wave Agent (D6, D11, D12)
5. Metabolic / Interval Agent (D8, D6)
6. Temporal Analysis Agent (D9)
7. Aggregator (merges findings, generates final report)

```
                    ┌──────────────┐
                    │   SDA-1      │
                    │  Features    │
                    └──────┬───────┘
                           │ (broadcast to all)
           ┌───────┬───────┼───────┬───────┬───────┐
           ▼       ▼       ▼       ▼       ▼       ▼
        ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐
        │Rhythm│ │Cond.│ │Isch.│ │Morph│ │Metab│ │Temp.│
        │Agent │ │Agent│ │Agent│ │Agent│ │Agent│ │Agent│
        └──┬──┘ └──┬──┘ └──┬──┘ └──┬──┘ └──┬──┘ └──┬──┘
           │       │       │       │       │       │
           └───────┴───────┴───┬───┴───────┴───────┘
                               ▼
                        ┌────────────┐
                        │ Aggregator │
                        │  (merge)   │
                        └────────────┘
```

**Pros:**
- Maximum parallelism: all specialists run simultaneously → lowest latency baseline
- Clear domain ownership: each agent has a well-defined scope
- Scalable: adding a new domain = adding a new agent
- Each agent can have a focused, shorter prompt → better DeepSeek performance within each domain
- Easy to unit-test each agent independently

**Cons:**
- **Critical weakness: cross-domain reasoning.** The aggregator must handle all cross-domain conditions, but it only sees FINDINGS, not raw features. When the Ischemia Agent detects "ST depression in V1-V3" it does not know whether the QRS is wide (which the Conduction Agent knows). The aggregator must perform the cross-domain synthesis, but it lacks the chain-of-thought reasoning that produced each finding.
- The aggregator becomes a bottleneck: it must resolve conflicts, apply conditional logic (LBBB → Sgarbossa), and integrate across domains. This is the hardest reasoning task in the entire system, yet it receives only summarized outputs.
- 11 of 13 STAT conditions require cross-domain reasoning. This architecture handles the 2 easiest conditions well (VF, anterior STEMI in isolation) but struggles with the 11 that matter most.
- Evidence from VT (difficulty 4.3): "Rate + rhythm + morphology + axis + AV relationship + algorithmic criteria must be integrated" — this requires 6 domains simultaneously, not sequentially merged.
- Evidence from posterior STEMI (difficulty 3.8): "Must understand that ST depression in V1-V3 can represent EITHER anterior subendocardial ischemia OR posterior STEMI" — this disambiguation requires the ischemia agent to have access to morphology analysis (R-wave in V1) that belongs to a different agent.

**Cross-domain handling:** Poor. The aggregator must reconstruct reasoning chains from fragmented findings. Example: LBBB+STEMI requires the Conduction Agent to detect LBBB AND the Ischemia Agent to know about it and switch to Sgarbossa. Without direct communication, the Ischemia Agent may report "ST elevation in V1-V4" without applying Sgarbossa criteria.

**Estimated API calls:** 7 parallel + 1 aggregator = 8 DeepSeek calls per ECG
**Estimated latency:** ~3-5 seconds (parallel specialist round) + ~2-3 seconds (aggregator) = **5-8 seconds total**

### 3.2 Option B: Sequential Pipeline

**Description:**
Each agent runs sequentially, building on the findings of all previous agents. The first agent performs basic assessment, and each subsequent agent has access to both raw features AND all prior agents' findings.

**Proposed Pipeline:**
1. Rate & Rhythm Agent → findings
2. Conduction & Intervals Agent → + prior findings
3. Morphology Agent → + prior findings
4. Ischemia & Territory Agent → + prior findings
5. Synthesis & Differential Agent → + all prior findings → final report

```
  ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
  │  Rate &   │───▶│ Conduct. │───▶│  Morpho. │───▶│ Ischemia │───▶│ Synthesis│
  │  Rhythm   │    │ Intervals│    │  Agent   │    │ Territory│    │ & Diff   │
  └──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
       ▲               ▲               ▲               ▲               ▲
       │               │               │               │               │
       └───────────────┴───────────────┴───────────────┴───────────────┘
                           SDA-1 Features (available to all)
```

**Pros:**
- Natural information flow: later agents can use earlier findings (e.g., Ischemia Agent knows LBBB status from Conduction Agent)
- Cross-domain reasoning is built into the pipeline ordering (conduction before ischemia)
- Each agent's prompt can include prior findings as context → enables conditional branching
- Mirrors the cardiologist's sequential approach (rate → rhythm → intervals → morphology → synthesis)
- The Synthesis Agent has all prior findings for final integration

**Cons:**
- **Sequential latency:** 5 agents x 2-3 seconds each = 10-15 seconds total. This is 2-3x slower than parallel approaches.
- **Ordering bias:** The pipeline order determines what information is available to each agent. If the order is wrong for a specific condition, that condition suffers. Example: if morphology runs before ischemia, the morphology agent cannot know about ST changes; if ischemia runs before morphology, it cannot know about T-wave symmetry (needed for Wellens vs LVH).
- **Single point of failure:** If the Rate & Rhythm Agent makes an error (e.g., misclassifies VT as SVT), all downstream agents inherit that error.
- **No parallelism:** Cannot exploit the independence of many reasoning domains. Rate analysis and ST analysis are fundamentally independent for most conditions.
- **Context window growth:** Each successive agent receives a larger context (raw features + all prior findings). The final Synthesis Agent may face context window pressure.

**Cross-domain handling:** Good for conditions where the pipeline order matches the reasoning chain (e.g., LBBB detected by Conduction Agent → Ischemia Agent uses Sgarbossa). Poor for conditions where the natural reasoning order doesn't match the pipeline (e.g., detecting hyperkalemia requires T-wave analysis before QRS analysis, but the pipeline puts QRS before T-wave).

**Estimated API calls:** 5 sequential calls
**Estimated latency:** ~10-15 seconds total (sequential)

### 3.3 Option C: Hierarchical (Specialist → Supervisor → Orchestrator)

**Description:**
Three-tier architecture. Tier 1: domain specialists run in parallel (like Option A). Tier 2: supervisor agents handle cross-domain integration for specific condition clusters. Tier 3: a single orchestrator synthesizes all supervisor findings into the final report.

**Proposed Hierarchy:**
```
Tier 1 (Specialists, parallel):
  - Rate/Rhythm Agent
  - Conduction Agent
  - ST/Ischemia Agent
  - Morphology Agent
  - Interval Agent
  - Temporal Agent

Tier 2 (Supervisors, parallel after Tier 1):
  - STEMI Supervisor (integrates ST, Morphology, Conduction findings for all STEMI variants)
  - Arrhythmia Supervisor (integrates Rate, Rhythm, Conduction, Morphology for VT/VF/AVB)
  - Metabolic Supervisor (integrates Morphology, Intervals, Lead-group for hyperK/Brugada/LongQT)

Tier 3 (Orchestrator):
  - Final Synthesis Agent (integrates all supervisor outputs, resolves conflicts, generates report)
```

```
                    ┌──────────────┐
                    │   SDA-1      │
                    │  Features    │
                    └──────┬───────┘
                           │
        ┌──────┬───────┬───┴───┬───────┬──────┐
        ▼      ▼       ▼       ▼       ▼      ▼
      ┌────┐ ┌────┐ ┌────┐ ┌────┐ ┌────┐ ┌────┐   Tier 1
      │Rate│ │Cond│ │ ST │ │Morp│ │Intv│ │Temp│   Specialists
      └─┬──┘ └─┬──┘ └─┬──┘ └─┬──┘ └─┬──┘ └─┬──┘
        │      │      │      │      │      │
        └──┬───┘      └──┬───┘      └──┬───┘
           ▼              ▼              ▼
      ┌─────────┐   ┌─────────┐   ┌─────────┐      Tier 2
      │Arrhythmia│  │  STEMI  │   │Metabolic│      Supervisors
      │Supervisor│  │Supervisor│  │Supervisor│
      └────┬────┘  └────┬────┘   └────┬────┘
           │            │              │
           └────────────┼──────────────┘
                        ▼
                 ┌─────────────┐                     Tier 3
                 │ Orchestrator│                     Final
                 └─────────────┘
```

**Pros:**
- Handles cross-domain reasoning at the supervisor level with full context from relevant specialists
- Supervisors are focused on specific condition clusters → better prompt engineering per cluster
- Tier 1 runs in parallel → latency for Tier 1 is similar to Option A
- Tier 2 can also run in parallel (the three supervisors are independent)
- Most closely mirrors clinical practice: specialists → subspecialty consultation → attending synthesis
- Conflict resolution is explicit: the orchestrator has final authority

**Cons:**
- **Most API calls:** 6 (Tier 1) + 3 (Tier 2) + 1 (Tier 3) = 10 DeepSeek calls per ECG
- **Three latency rounds:** ~3-5s + ~3-5s + ~2-3s = **8-13 seconds total**
- **Supervisor boundary problems:** Some conditions span multiple supervisors. Hyperkalemia mimicking Brugada requires the Metabolic Supervisor and the STEMI Supervisor to coordinate — but they are at the same tier and cannot directly communicate.
- **Complexity:** Most complex architecture to implement, debug, and maintain
- **Prompt overhead:** Each supervisor must receive and parse the findings from multiple Tier 1 agents, consuming context window space

**Cross-domain handling:** Very good within supervisor scope. Mediocre across supervisors (same problem as Option A's aggregator, just one level up). The orchestrator must handle cross-supervisor conflicts.

Evidence from disease files: The STEMI Supervisor would handle the 5 STEMI conditions well, but posterior STEMI (cross-domain score 5/5) requires morphology analysis (R-wave in V1) that might be in the Morphology Agent's output, which feeds into the STEMI Supervisor. This works IF the information flow is properly designed.

**Estimated API calls:** 10 calls (6+3+1)
**Estimated latency:** ~8-13 seconds (three sequential rounds)

### 3.4 Option D: Single Comprehensive Agent

**Description:**
One DeepSeek reasoner agent receives all features from SDA-1 plus the entire RAG knowledge base for all relevant conditions. It performs a structured, step-by-step analysis following the cardiologist's systematic approach, producing a complete interpretation in a single call.

```
    ┌──────────────┐     ┌──────────────┐
    │   SDA-1      │     │  RAG Context │
    │  Features    │     │  (retrieved) │
    └──────┬───────┘     └──────┬───────┘
           │                    │
           └────────┬───────────┘
                    ▼
           ┌────────────────┐
           │  Single Agent  │
           │  (structured   │
           │   reasoning)   │
           │                │
           │ Step 1: Rate   │
           │ Step 2: Rhythm │
           │ Step 3: Axis   │
           │ Step 4: Ints   │
           │ Step 5: Morph  │
           │ Step 6: Synth  │
           └────────────────┘
```

**Pros:**
- **Simplest architecture:** No coordination overhead, no communication protocol, no conflict resolution
- **Full context:** The agent sees everything at once — features, RAG context, and its own reasoning. Cross-domain reasoning happens naturally within the chain-of-thought.
- **Natural for DeepSeek reasoner:** The reasoner mode is designed for long chain-of-thought reasoning. A single comprehensive prompt may produce better reasoning than fragmented multi-agent prompts.
- **Lowest API cost:** 1 DeepSeek call per ECG
- **No information loss:** No summarization or compression between agents

**Cons:**
- **Context window pressure:** Features for all 12 leads (~2000 tokens) + RAG context for up to 95 conditions (~10,000-50,000 tokens) + system prompt (~3,000 tokens) + reasoning space (potentially 10,000+ tokens) may exceed even DeepSeek's 64K context window. At 95 conditions, the RAG context alone could be 20,000+ tokens.
- **No parallelism:** Cannot exploit any independence between reasoning domains. One call, one latency.
- **Reasoning quality degrades with prompt length:** Research consistently shows that LLM reasoning quality decreases as context window utilization increases. A prompt using 80%+ of context window produces worse reasoning than one using 30%.
- **Less explainable:** The reasoning is inside a single monolithic chain-of-thought. Debugging which step went wrong is harder than debugging which agent made an error.
- **Single point of failure:** One hallucination in the reasoning chain corrupts the entire output.
- **Scalability concern:** Adding conditions means adding RAG context, which increases context pressure. At 95 conditions, this approach may not be viable without aggressive RAG filtering.

**Cross-domain handling:** Excellent in theory — the agent can naturally reason across domains because it sees everything. In practice, context window limits may force RAG truncation that removes the cross-domain knowledge the agent needs.

Evidence: VT (difficulty 4.3) requires 9 reasoning domains. A single agent with full context can integrate all 9 naturally. But the prompt would need to include VT detection criteria, Brugada algorithm, Vereckei algorithm, SVT+aberrancy criteria, AND all potential confounders — potentially 5,000+ tokens of RAG just for VT alone.

**Estimated API calls:** 1 call
**Estimated latency:** ~5-10 seconds (single long reasoning chain)

### 3.5 Option E: Hybrid (Parallel Specialists + Sequential Cross-Domain Reviewer)

**Description:**
Two-phase architecture. Phase 1: domain specialists run in parallel (focused, fast, domain-specific). Phase 2: a cross-domain reviewer agent receives ALL Phase 1 findings plus the raw features and performs cross-domain reasoning, conflict resolution, differential diagnosis, and synthesis.

The key insight: **Phase 1 agents do the focused domain work. Phase 2 does the expert integration.**

**Proposed Architecture:**
```
Phase 1 (Parallel Domain Analysis):
  - Rhythm & Rate Agent (D1, D2, D3)
  - Conduction & QRS Agent (D4, D8)
  - Ischemia & Territory Agent (D5, D7, D11)
  - Morphology & Repolarization Agent (D6, D11, D12)

Phase 2 (Sequential Cross-Domain Review):
  - Cross-Domain Synthesis Agent (D10, D13 — uses all Phase 1 findings + raw features)
```

```
                    ┌──────────────┐
                    │   SDA-1      │
                    │  Features    │
                    └──────┬───────┘
                           │ (broadcast to all Phase 1 agents)
           ┌───────────────┼───────────────┐
           ▼               ▼               ▼
    ┌─────────────┐ ┌─────────────┐ ┌─────────────┐
    │  Rhythm &   │ │ Ischemia &  │ │ Morphology  │
    │  Rate &     │ │ Territory   │ │ & Repolar.  │
    │  Conduction │ │   Agent     │ │   Agent     │
    └──────┬──────┘ └──────┬──────┘ └──────┬──────┘
           │               │               │
           └───────────────┼───────────────┘
                           ▼
                  ┌──────────────────┐
                  │  Cross-Domain    │
                  │  Synthesis Agent │     Phase 2
                  │                  │
                  │ - Receives ALL   │
                  │   Phase 1 output │
                  │ - Has raw feats  │
                  │ - Cross-domain   │
                  │   reasoning      │
                  │ - Diff diagnosis │
                  │ - Final report   │
                  └──────────────────┘
```

**Why 3 Phase 1 agents instead of 6:**
The disease file analysis reveals that many domains are tightly coupled:
- Rate, rhythm, and AV relationship (D1, D2, D3) are inseparable for conditions like complete AVB and VT
- ST analysis and lead-group correlation (D5, D7) always co-occur in ischemia assessment
- T-wave morphology and amplitude measurement (D6, D11) are used together in 10/13 conditions

Splitting tightly coupled domains across agents creates artificial boundaries that hurt reasoning. Merging them into 3 broader specialists keeps each agent's scope manageable while preserving natural reasoning connections.

**Pros:**
- **Best balance of parallelism and cross-domain reasoning:** Phase 1 runs in parallel (3s), Phase 2 runs sequentially (3-5s) = 6-8s total
- **Cross-Domain Synthesis Agent has full context:** It receives structured findings from all Phase 1 agents PLUS raw features. It can re-examine raw data when Phase 1 findings conflict or when cross-domain conditions are detected.
- **Handles the hardest conditions:** VT (4.3) — Phase 1 agents provide rhythm analysis, QRS morphology, and AV relationship findings. The Synthesis Agent integrates these with concordance, Brugada criteria, and Vereckei algorithm. Posterior STEMI (3.8) — Ischemia Agent detects V1-V3 depression; Morphology Agent detects tall R in V1; Synthesis Agent recognizes the mirror-image pattern and reclassifies as posterior STEMI.
- **Explainable:** Phase 1 findings provide transparent domain-specific assessments. Phase 2 reasoning is explicit about cross-domain integration.
- **Moderate API cost:** 3 parallel + 1 sequential = 4 DeepSeek calls per ECG
- **Scalable:** New conditions map to existing Phase 1 agents; complex cross-domain conditions are handled by the Synthesis Agent.

**Cons:**
- **Phase 2 is a complex agent:** The Synthesis Agent must handle all cross-domain conditions, all differentials, and all conflict resolution. Its prompt will be large and complex.
- **Two sequential phases:** Faster than pure sequential (Option B) but slower than pure parallel (Option A)
- **Phase 2 depends on Phase 1 quality:** If a Phase 1 agent misses a finding, the Synthesis Agent may not catch it (though it does have access to raw features as a safety net)

**Cross-domain handling:** Excellent. The Synthesis Agent is purpose-built for cross-domain reasoning. It sees both structured findings AND raw features, enabling it to: (1) verify Phase 1 findings against raw data, (2) apply conditional logic (LBBB detected by Conduction → Ischemia findings reinterpreted with Sgarbossa), (3) detect patterns that span agents (hyperkalemia pseudo-Brugada).

Evidence from disease files:
- LBBB+STEMI: Conduction Agent detects LBBB, Ischemia Agent reports ST changes, Synthesis Agent applies Sgarbossa criteria
- Posterior STEMI: Ischemia Agent reports "ST depression V1-V3", Morphology Agent reports "R/S > 1 in V1", Synthesis Agent recognizes posterior STEMI pattern
- VT vs SVT: Rhythm Agent reports "wide complex tachycardia, rate 180", Conduction Agent reports "QRS 160ms LBBB pattern", Morphology Agent reports "concordance in V1-V6", Synthesis Agent applies Brugada criteria → VT diagnosis

**Estimated API calls:** 4 calls (3 parallel + 1 sequential)
**Estimated latency:** ~3-5s (Phase 1 parallel) + ~3-5s (Phase 2) = **6-10 seconds total**

### 3.6 Option F: Condition-Triggered (Initial Screening → Activate Relevant Agents)

**Description:**
A lightweight screening agent performs initial triage on the feature set, identifying which conditions are plausible. Only agents relevant to those conditions are activated. This minimizes unnecessary computation.

```
                    ┌──────────────┐
                    │   SDA-1      │
                    │  Features    │
                    └──────┬───────┘
                           │
                    ┌──────▼──────┐
                    │  Screening  │
                    │   Agent     │
                    │ (lightweight)│
                    └──────┬──────┘
                           │ (which conditions are plausible?)
                    ┌──────▼──────┐
                    │ Condition   │
                    │  Router     │
                    └──┬──┬──┬───┘
                       │  │  │
             ┌─────────┘  │  └─────────┐
             ▼            ▼            ▼
        ┌─────────┐  ┌─────────┐  ┌─────────┐
        │STEMI    │  │  VT     │  │ HyperK  │
        │Expert   │  │ Expert  │  │ Expert  │
        │Agent    │  │ Agent   │  │ Agent   │
        └────┬────┘  └────┬────┘  └────┬────┘
             │            │            │
             └────────────┼────────────┘
                          ▼
                   ┌────────────┐
                   │  Synthesis │
                   └────────────┘
```

**Pros:**
- **Most efficient for clear-cut cases:** If the screening agent correctly identifies "obvious anterior STEMI", only the STEMI Expert Agent is activated → 2 calls total
- **Reduced cost for routine ECGs:** Normal ECGs might need only the screening agent → 1 call
- **Each condition-expert agent can be highly specialized** with focused RAG and deep domain knowledge

**Cons:**
- **Chicken-and-egg problem:** To know which conditions are plausible, you need to analyze the ECG. But the screening agent must be simple (otherwise it IS the comprehensive agent). A simple screener may miss subtle conditions — exactly the ones that matter most.
- **Misses unexpected findings:** If the screener doesn't activate the hyperkalemia pathway, peaked T-waves with QRS widening will be missed.
- **Handling cross-condition interactions:** If the screener activates "STEMI Expert" and "LBBB Expert" separately, who integrates them? Another agent is needed.
- **Not robust to atypical presentations:** Posterior STEMI (difficulty 3.8) presents as "ST depression in V1-V3" which a screener might route to "subendocardial ischemia" rather than "posterior STEMI."
- Evidence from de Winter (difficulty 3.7): "Automated STEMI detection algorithms look for ST ELEVATION. The de Winter pattern has ST DEPRESSION, so automated algorithms will classify this as 'non-STEMI' or 'normal', causing catastrophic delays." — A condition-triggered approach has exactly this vulnerability.
- **VT vs SVT ambiguity:** The screening agent sees "wide QRS tachycardia" and must decide whether to activate the VT Expert or the SVT Expert. This is the entire diagnostic question — the screening agent would need to solve it to route it.

**Cross-domain handling:** Poor. Cross-domain conditions require multiple condition-experts to be activated AND coordinated. The routing logic for cross-domain conditions is itself a complex reasoning task.

**Estimated API calls:** 1 (screener) + 1-4 (condition experts) + 1 (synthesis) = 3-6 calls, variable
**Estimated latency:** ~2-3s (screener) + ~2-5s (expert, variable) + ~2-3s (synthesis) = **6-11 seconds total**, highly variable

### 6.2 Reasoner Mode Implications for Prompt Design

All agents use DeepSeek reasoner mode. This has specific implications:

**1. Chain-of-Thought is mandatory and automatic.**
DeepSeek reasoner generates extended reasoning before producing the final output. The system prompt must guide this reasoning without constraining it. Prompt design should:
- Provide the reasoning STRUCTURE (what to check, in what order) but not the ANSWER
- Include explicit decision trees for complex conditions (Brugada criteria, Vereckei algorithm)
- Use "think step by step" framing with domain-specific steps

**2. Structured output requires careful formatting.**
Reasoner mode's CoT may produce free-form text. The final output must be parsed into a structured schema (JSON). The prompt should include the exact output schema with examples.

**3. RAG context must be high-signal, low-noise.**
The reasoner processes the entire context. Including irrelevant RAG content wastes reasoning capacity and may introduce hallucination. RAG retrieval must be precision-optimized:
- Phase 1 agents: retrieve only the domain-specific criteria and thresholds
- Phase 2 agent: retrieve cross-domain interaction rules based on Phase 1 findings (dynamic RAG)

**4. Temperature should be 0 for medical diagnosis.**
Deterministic output is essential. No creative sampling. Every run should produce the same reasoning for the same input.

**5. System prompt length matters.**
Research shows that DeepSeek reasoner performs best when the system prompt is concise but complete. Overly long system prompts (>5,000 tokens) can degrade reasoning quality. Each Phase 1 agent's system prompt should be under 3,000 tokens. The Phase 2 agent can be up to 4,000 tokens because its task is more complex.

## 7. Recommended Architecture

### 7.1 Recommendation: Option E — Hybrid (Parallel Specialists + Cross-Domain Synthesis)

Based on the decision matrix (score 8.00, 0.90 points ahead of the next best option), the cross-domain dependency analysis, and the DeepSeek context budget analysis, **Option E is the recommended architecture.**

### 7.2 Why Option E Won

**1. It directly addresses the primary finding from the disease files.**
The disease file analysis reveals that 11/13 STAT conditions require cross-domain reasoning, with a mean cross-domain difficulty of 3.7/5. Option E is the only architecture that simultaneously provides:
- Focused domain expertise (Phase 1 parallel specialists)
- Comprehensive cross-domain synthesis (Phase 2 with full context)
- A raw-feature safety net (Phase 2 can verify Phase 1 findings)

**2. It handles the hardest conditions correctly.**
The three hardest cross-domain cases and how Option E handles them:

**Case 1: VT vs SVT with Aberrancy (difficulty 4.3)**
- Phase 1 Rhythm Agent: detects wide QRS tachycardia, measures rate, assesses regularity, searches for AV dissociation
- Phase 1 Conduction Agent: classifies QRS morphology, measures QRS duration, determines axis
- Phase 1 Ischemia Agent: assesses concordance in V1-V6
- Phase 2 Synthesis: receives all findings. Applies Brugada criteria stepwise: (1) concordance? (from Ischemia), (2) RS >100ms? (from Conduction), (3) AV dissociation? (from Rhythm), (4) morphology criteria? (from Conduction). Also applies Vereckei aVR algorithm. Integrates all into VT vs SVT+aberrancy determination with explicit reasoning chain.

**Case 2: STEMI in LBBB (cross-condition, multiple STEMIs)**
- Phase 1 Conduction Agent: detects LBBB (QRS >120ms, typical morphology)
- Phase 1 Ischemia Agent: detects ST changes but notes "LBBB present — standard criteria may be invalid"
- Phase 2 Synthesis: receives "LBBB detected" + "ST changes present." Applies Sgarbossa criteria: concordant ST elevation ≥1mm? Concordant ST depression V1-V3? Excessively discordant STE with ST/S ratio ≥0.25? Each criterion checked against raw features. Produces Sgarbossa score and STEMI-in-LBBB determination.

**Case 3: Posterior MI Diagnosis (difficulty 3.8)**
- Phase 1 Ischemia Agent: detects "ST depression in V1-V3, horizontal/downsloping"
- Phase 1 Morphology Agent: detects "R/S ratio > 1 in V1, upright T-wave in V1"
- Phase 2 Synthesis: receives both findings. Recognizes the mirror-image pattern: V1-V3 depression + tall R in V1 + upright T = posterior STEMI, not anterior subendocardial ischemia. Checks for concurrent inferior STEMI (from Ischemia Agent). Determines culprit vessel (RCA if III > II, LCx if II ≥ III). Recommends V7-V9. Flags STAT.

**3. It scales to 95 conditions without collapsing.**
As conditions grow from 13 to 95:
- Phase 1 agents' domain scope remains the same. The Ischemia Agent handles all ischemia-related conditions regardless of count. Its prompt and RAG grow linearly but within its domain.
- Phase 2 Synthesis Agent receives structured findings (not raw features for every condition). The input grows in structured-finding count but the format is consistent and parseable.
- RAG retrieval for Phase 2 is DYNAMIC: based on Phase 1 findings, only relevant cross-domain interaction rules are retrieved. If no ischemia findings are present, LBBB+STEMI interaction rules are not loaded.

### 7.4 Agent Communication Protocol

**Phase 1 → Phase 2 Communication:**
Each Phase 1 agent produces a structured JSON output following a common schema:
```json
{
  "agent_id": "RRC|IT|MR",
  "confidence": 0.0-1.0,
  "findings": [
    {
      "finding_id": "uuid",
      "category": "rhythm|conduction|ischemia|morphology|interval|temporal",
      "description": "string",
      "evidence": {"leads": [], "measurements": {}, "criteria_met": []},
      "confidence": 0.0-1.0,
      "stat_flag": true|false,
      "cross_domain_hint": "string or null"
    }
  ],
  "raw_feature_flags": ["features needing Phase 2 verification"]
}
```

The `cross_domain_hint` field is critical: each Phase 1 agent flags findings that it believes require cross-domain integration. Examples:
- RRC Agent: `"cross_domain_hint": "LBBB detected — STEMI criteria require Sgarbossa pathway"`
- IT Agent: `"cross_domain_hint": "V1-V3 ST depression — check for posterior STEMI (need R-wave morphology from MR Agent)"`
- MR Agent: `"cross_domain_hint": "Diffuse peaked T-waves — territorial vs metabolic distinction needed (check ST analysis from IT Agent)"`

**Conflict Resolution:**
When Phase 1 agents produce conflicting findings (e.g., IT Agent says "ST changes consistent with ischemia" while MR Agent says "T-wave morphology consistent with hyperkalemia"), the Phase 2 CDS Agent:
1. Identifies the conflict from structured outputs
2. Re-examines raw features relevant to both findings
3. Applies cross-domain reasoning rules (territorial vs diffuse distribution, QRS width, P-wave presence)
4. Determines the most likely diagnosis with explicit evidence weighting
5. Documents the conflict and resolution in the reasoning chain

## 9. Agentic vs ML vs Hybrid Comparison

### 9.1 Traditional ML Approach

**Description:** Train deep learning models (CNNs, Transformers) on large ECG datasets (PTB-XL, CODE-15%, Chapman-Shaoxing) to classify ECG abnormalities directly from raw signals.

**Strengths:**
- Fast inference (milliseconds per ECG once deployed)
- No per-inference API cost (one-time training cost)
- Proven accuracy for common conditions (AFib, STEMI, LBBB) in published benchmarks
- Can process raw waveform data without SDA-1 feature extraction

**Weaknesses for THIS project:**
- **Rare conditions:** PTB-XL has <1% representation for 8/13 STAT conditions (de Winter, Brugada, Complete AVB, VF, RV STEMI, posterior STEMI, Wellens, hyperkalemia). ML models cannot learn what they cannot see. Evidence: de Winter (difficulty 3.7) "Very rare — approximately 2% of acute LAD occlusions; PTB-XL likely contains very few or zero labeled examples" (de_winter_t_waves.md, Section 6.6). Brugada Type 1 (difficulty 3.5) "Few if any validated examples may exist in the dataset" (brugada_type1.md, Section 6.6).
- **No explainability:** A CNN that flags "STEMI" cannot explain WHY — which leads show elevation, what the morphology is, what differentials were considered. Clinical adoption requires explanation.
- **Cross-domain reasoning:** ML models learn statistical correlations, not reasoning chains. The conditional branching required (LBBB → Sgarbossa) is difficult to encode in a standard classifier.
- **Updating knowledge:** When guidelines change (e.g., 2025 AHA updates to STEMI criteria), ML models must be retrained. Agentic systems update a RAG document.
- **No differential diagnosis:** ML classifiers produce labels, not differential diagnoses. "87% probability of anterior STEMI" is less useful than "anterior STEMI with convex morphology in V1-V4, reciprocal depression in III/aVF, differential includes BER (excluded: ST/T ratio >0.25) and LVH (excluded: no voltage criteria met)."

### 9.2 Agentic Approach (This Project)

**Strengths:**
- **Interpretable reasoning:** Every diagnosis comes with an explicit reasoning chain that clinicians can verify
- **Handles rare conditions:** RAG-augmented agents can reason about conditions never seen in training data (de Winter, Brugada) using textbook knowledge
- **Cross-domain reasoning:** The architecture is specifically designed for conditions that span reasoning domains (11/13 STAT conditions)
- **Dynamic knowledge updates:** New guidelines or conditions are added by updating RAG documents, not retraining models
- **Differential diagnosis:** Agents naturally produce differentials with evidence for and against each possibility
- **Flexible output:** Can produce ER nurse explanations, cardiologist-level analysis, or dashboard annotations — all from the same reasoning

**Weaknesses:**
- **Hallucination risk:** LLMs can generate plausible-sounding but incorrect findings. Mitigations: structured feature input (the agent reasons FROM computed features, not raw signals), confidence thresholds, multi-agent verification (Phase 2 verifies Phase 1).
- **Latency:** 6-10 seconds per ECG (vs milliseconds for ML). Acceptable for clinical use but not for real-time monitoring.
- **Cost:** $0.015-0.03 per ECG. Low in absolute terms but nonzero, unlike local ML inference.
- **Dependent on feature extraction quality:** If SDA-1 computes incorrect features, the agents inherit those errors. ML models that work on raw signals don't have this dependency.

### 9.3 Hybrid Possibility

A hybrid approach could combine the strengths of both:

**ML Pre-Screening + Agentic Deep Analysis:**
1. ML model performs fast classification on raw ECG signal
2. If ML confidence is HIGH (>0.95) for a COMMON condition → output ML result with standard explanation
3. If ML confidence is LOW, condition is RARE, or MULTIPLE conditions detected → invoke agentic pipeline for deep analysis

**Pros:** Faster for obvious cases, lower cost for routine ECGs, full agentic reasoning for complex cases.
**Cons:** Two systems to maintain, ML model still needs training data for rare conditions, the "routing" decision (ML vs agentic) is itself a judgment call.

### 9.4 Why Agentic for YouOwnECG

The decision to use an agentic approach for this project is based on four specific requirements:

**1. Interpretability is non-negotiable.**
YouOwnECG is a clinical decision support system. Clinicians will not trust (and regulators may not approve) a system that produces labels without reasoning. The 13 STAT disease files explicitly specify dashboard visualization requirements with lead-specific annotations, arrows, and clinician explanations at multiple levels (ER nurse, cardiologist). Only an agentic approach can generate these naturally.

**2. Rare conditions cannot be left out.**
8 of 13 STAT conditions are rare in PTB-XL. These conditions are life-threatening — they ARE the reason this system exists. An ML model that handles only the common 5 (anterior STEMI, inferior STEMI, lateral STEMI, AFib, VT) would miss the exact conditions that kill patients (de Winter, posterior STEMI, Brugada, Wellens, severe hyperkalemia). The agentic approach handles rare conditions through RAG knowledge, not training examples.

**3. Cross-domain reasoning is the core challenge.**
The disease file analysis demonstrates that 11/13 STAT conditions require cross-domain reasoning. ML models excel at pattern matching within a domain but struggle with conditional logic across domains. The LBBB → Sgarbossa pathway, the posterior STEMI mirror-image reasoning, and the VT Brugada criteria are all explicitly conditional, multi-domain reasoning tasks.

**4. The system must evolve with guidelines.**
Medical knowledge changes. The 2025 AHA/ESC guidelines updated STEMI criteria, Sgarbossa thresholds, and QTc risk stratification. The agentic system updates by modifying RAG documents. An ML system would require collecting new labeled data and retraining.

### 9.5 Risk Mitigations for the Agentic Approach

| Risk | Mitigation |
|------|-----------|
| Hallucination (fabricating findings) | Agents reason FROM computed features, not free text. Every finding must reference specific feature values. Phase 2 verifies Phase 1 findings against raw features. Output schema requires evidence for every finding. |
| Inconsistency (different results for same input) | Temperature = 0 for all agents. Deterministic output. System prompt caching ensures identical prompts. Regression testing on the 10 pilot scenarios with every prompt change. |
| Missed findings (false negatives) | Phase 2 has access to raw features as safety net. Multi-agent architecture means a finding can be caught by any of 4 agents. Sensitivity-optimized thresholds (flag liberally, let CDS resolve). |
| Cost escalation at scale | Prompt caching reduces input cost by ~75%. Phase 1 agents run in parallel (no redundant computation). At 10,000 ECGs/day, cost is ~$150-300/day — comparable to a single technician's salary. |
| Latency exceeding clinical acceptability | 6-10 second target is within 15-second threshold. Phase 1 parallelism is the key optimization. If needed, Phase 1 can be reduced to 2 agents. |
| DeepSeek model changes or deprecation | Architecture is model-agnostic. Prompt engineering targets structured reasoning, not model-specific quirks. Can migrate to GPT-4, Claude, or Llama with prompt adaptation. |

| Hyperkalemia → masks STEMI | hyperkalemia.md | Diffuse peaked T obscures territorial hyperacute T | CDS checks T-wave distribution from MR Agent: diffuse = hyperK, territorial = STEMI |
| VT → prevents all other assessment | ventricular_tachycardia.md | During VT, STEMI/hyperK/etc cannot be assessed | CDS flags "post-cardioversion reassessment required" |
| Long QT + PVCs → TdP risk | long_qt_tdp.md | PVCs in long QT setting enable SLS initiation | CDS receives prolonged QTc from MR Agent + ectopy from RRC Agent → flags TdP risk |
| Wellens ↔ de Winter (same patient, different times) | wellens_syndrome.md | Both indicate critical LAD disease; mutually exclusive at one time point | CDS verifies: de Winter has upsloping ST depression + peaked T (active occlusion); Wellens has T inversions + isoelectric ST (post-reperfusion) |
| Complete AVB + inferior STEMI vs anterior STEMI | complete_av_block.md | Prognostic implications differ by STEMI territory | CDS receives CHB from RRC Agent + STEMI territory from IT Agent → assigns appropriate prognosis |
| AFib → complicates QTc measurement | long_qt_tdp.md | Variable RR makes single-beat QTc unreliable | CDS receives AFib from RRC Agent + QTc from MR Agent → applies multi-beat averaging correction |
| AFib → complicates ST measurement | stemi_lateral.md | Varying baseline makes ST measurement harder | CDS flags "ST measurements in AFib setting — verify on beats with similar RR intervals" |
