## Adjudication by Issue

### Issue 1 (REQUIRED) — Async HTTP Client Specification

**RRC finding:** `asyncio.as_completed()` is correct but the implementation must use a native async HTTP client. Synchronous clients wrapped in `asyncio.to_thread()` delay STAT fast-path processing.

**EPM ruling: FIXED IN DOCUMENT.**

Section 11.1 requirement 6 now specifies: `call_agent()` must use a native async HTTP client (`httpx.AsyncClient`). Synchronous clients are explicitly prohibited. This is a Node 2.6 implementation requirement, not a design constraint, but documenting it here prevents the error at implementation time.

**Status:** Closed.

---

### Issue 1.2 (REQUIRED) — Multi-Agent STAT Race Condition

**RRC finding:** If two Phase 1 agents both return `stat_flag = true` near-simultaneously, the behavior was undefined — both fire? First wins? The report's `stat_alert` was a single object, not an array.

**EPM ruling: FIXED IN DOCUMENT.**

Section 3.3.1 added: "all-fire" behavior — all STAT conditions fire independently. The report `stat_alert` field has been changed to `stat_alerts[]` (an array). Each alert transitions independently through its lifecycle. Section 11.3 requirement 1 updated to require UI support for multiple simultaneous STAT alerts. Test ORC-11 added to validate this behavior.

**Status:** Closed.

---

### Issue 2 (CRITICAL) — `STAT_DOWNGRADED` Confidence Floor

**RRC finding:** A CDS confidence of 0.51 for a mimic diagnosis could autonomously retract a Phase 1 STEMI alert at 0.95 confidence. No confidence threshold for downgrade was specified.

**EPM ruling: FIXED IN DOCUMENT.**

Section 3.3 now defines: `STAT_DOWNGRADED` requires CDS mimic diagnosis confidence ≥ 0.80. If CDS confidence < 0.80, the orchestrator emits `STAT_UNRESOLVED` instead ("Conflicting assessments — manual review required"). Clinical governance is explicitly documented: autonomous downgrade is accepted at ≥0.80 threshold on clinical risk/alert-fatigue grounds; UI must require clinician acknowledgment of any `STAT_DOWNGRADED` event regardless. Section 11.3 requirement 2 captures the UI obligation. Test ORC-10b added for the sub-threshold case.

**Status:** Closed.

---

### Issue 3 (REQUIRED) — Retry Logic for Slow 5xx Responses

**RRC finding:** "Retry once (immediate, no backoff)" on 5xx does not distinguish fast failures (transient, retry makes sense) from slow failures (server timed out internally — immediate retry will also fail or timeout, adding 15s to Phase 1 latency).

**EPM ruling: FIXED IN DOCUMENT.**

Section 4.2 now distinguishes two rows: (a) 5xx received within <2s → immediate retry once; (b) 5xx received after ≥2s → treat as timeout equivalent, no retry. Section 11.1 requirement 3 adds `response_latency_s` to `AgentAPIError` to enable this discrimination.

**Status:** Closed.

---

### Issue 4 (CRITICAL) — `raw_feature_flags` Path Format and Lookup Semantics Undefined

**RRC finding:** Path format for `raw_feature_flags` strings was unspecified (JSONPath? dot notation?). Array path resolution semantics were undefined. Lookup failure handling was unspecified. Token cost of array paths was not accounted for in budget enforcement.

**EPM ruling: FIXED IN DOCUMENT.**

Section 5.2 now defines:
- Path format: dot notation with `[*]` wildcard for array traversal
- Resolution semantics: scalar paths → single value; `beats[*].field` → array of N values; `beats[*].leads[*].field` → N×12 2D array
- Token cost estimation before inclusion: N×1 for beat arrays, N×12 for per-lead arrays
- Budget priority: STAT-flagged paths first, corroborated paths second, scalar paths before array paths
- Lookup failure: log + add to `raw_feature_fields_dropped`, continue

This fix covers all four sub-items (4a-4d) from the RRC review.

**Status:** Closed.

---

### Issue 5 (REQUIRED) — Phase 1 Output Token Estimate Underestimated

**RRC finding:** Phase 1 output estimated at ~1,200 tokens per agent. For complex ECGs (VT + complete AVB + fascicular block), realistic estimate is 2,500–3,500 tokens per agent, putting CDS input at 12,000–15,000 tokens.

**EPM ruling: PRE-IMPLEMENTATION VERIFICATION TASK — does not block design gate.**

Rationale: 12,000–15,000 tokens is still within the 64K context window and does not change the architecture. The practical consequence is cost projection revision and potential Phase 1 output truncation rules if P95 exceeds 2,000 tokens per agent.

Section 6.2 updated to show both simple and complex estimates. VT-2.3-01 added to the verification task table: measure actual Phase 1 output sizes across 10 PTB-XL ECGs before implementation. If P95 exceeds 2,000 tokens, add truncation rules.

**Status:** Recorded as VT-2.3-01.

### Issue 6 (RECOMMENDED) — Clinical Urgency Rank

**RRC finding:** Flat confidence thresholds (≥0.8 = critical) may order a high-confidence LVH finding above a slightly-lower-confidence de Winter finding — clinically backwards.

**EPM ruling: ACCEPTED AND FIXED.**

`clinical_urgency_rank` field added to CDS `primary_findings` schema in Node 2.2 Section 5.7. CDS assigns ranks 1..N (1 = most urgent). Section 7.2 updated: orchestrator uses `clinical_urgency_rank` as primary sort key; flat confidence tiers serve as fallback only when CDS failed. This delegates clinical ordering judgment to CDS where it belongs.

**Status:** Closed (Node 2.2 amendment applied).

---

### Issue 7 (REQUIRED) — STAT Deduplication on Pipeline Retry

**RRC finding:** If a pipeline is retried (e.g., CDS timeout) for the same `ecg_id`, Phase 1 agents will likely re-fire STAT alerts — the UI receives duplicate STAT notifications.

**EPM ruling: FIXED IN DOCUMENT.**

Section 3.3.1 defines: orchestrator interface accepts `prior_stat_fired: bool` parameter. If `True`, fast-path STAT emission is suppressed on the retry run; Phase 1 STAT findings are still recorded in the report. The caller (SDA-1→SDA-2 interface layer) tracks this state per `ecg_id`. Test ORC-13 added.

**Status:** Closed.

---

### Issue 8a (REQUIRED) — `recording_duration_sec = 0` Passes Phase 0 Validation

**EPM ruling: FIXED IN DOCUMENT.**

Phase 0 validation now explicitly requires `recording_duration_sec > 0`. Section 11.2 requirement 3 updated to match. Test ORC-15 added. The TWA `null`-for-short-recordings rule (Node 2.2) depends on having a valid positive duration.

**Status:** Closed.

---

### Issue 8b (RECOMMENDED) — Normal ECG with Empty Findings

**RRC finding:** CDS behavior for normal ECG (empty Phase 1 findings) was undefined. Does CDS still run? Does it produce an affirmative normal conclusion?

**EPM ruling: FIXED IN DOCUMENTS.**

- Node 2.2: `no_significant_findings: boolean` added to CDS output schema. CDS runs regardless of Phase 1 findings; for normal ECGs, sets this field to `true` and includes reasoning in `reasoning_chain_summary`.
- Node 2.3 Section 11.3 requirement 6: UI must render affirmative normal conclusion when `no_significant_findings = true`.

**Status:** Closed.

---

### Issue 8c (REQUIRED) — CDS-Only STAT Alert Lifecycle Undefined

**RRC finding:** When CDS synthesizes a new STAT finding (e.g., left main stenosis) not detectable from any single Phase 1 agent, the alert lifecycle was undefined — there was no provisional alert, so the `STAT_PROVISIONAL → STAT_CONFIRMED` flow does not apply.

**EPM ruling: FIXED IN DOCUMENT.**

Section 3.3 now defines: when `stat_fast_path_fired = false` and CDS returns `stat_flag = true`, orchestrator fires `STAT_CONFIRMED` directly (no provisional phase). The latency for this alert is 6–10s (full pipeline), not the fast-path latency. Test ORC-12 added.

**Status:** Closed.

---

### Issue 8d (REQUIRED) — All-Noise Beat Class Not Caught in Phase 0

**RRC finding:** If all beats are `beat_class = 4` (noise), the feature object is semantically empty. Agents would produce findings on garbage data.

**EPM ruling: FIXED IN DOCUMENT.**

Phase 0 validation now checks that at least one beat has `beat_class` in {0, 1, 2, 3}. If all beats are class 4, return `PIPELINE_FAILED("NO_ANALYZABLE_BEATS")`. Section 11.2 requirement 4 added. Test ORC-14 added.

**Status:** Closed.

---

### Issue 9 (CRITICAL) — CDS Base Fields Don't Match Feature Schema

**RRC finding:** Six of the twelve base CDS fields in Section 5.2 had wrong field names or referenced fields that don't exist in Node 2.2 Section 1.3 (`aggregate.lbbb_present`, `aggregate.rbbb_present`, `aggregate.dominant_rhythm` do not exist; `aggregate.heart_rate_bpm` → `heart_rate_ventricular_bpm`; etc.). Agent key casing inconsistency (`"RRC"` vs `"rrc"`).

**EPM ruling: FIXED IN DOCUMENT.**

Section 5.2 base fields corrected to exact Node 2.2 Section 1.3 names:
- `aggregate.heart_rate_ventricular_bpm` (was: `aggregate.heart_rate_bpm`)
- `aggregate.rhythm_regularity_coefficient` (was: `aggregate.rr_variability_ms`)
- `aggregate.qrs_axis_degrees` (was: `aggregate.mean_qrs_axis_degrees`)
- `aggregate.av_ratio` (new — replaces removed `dominant_rhythm`)
- Removed `aggregate.lbbb_present`, `aggregate.rbbb_present`, `aggregate.dominant_rhythm`
- Clarification added: BBB status comes from RRC Phase 1 output, not feature schema

Agent key casing standardized to lowercase throughout Node 2.3 (`"rrc"`, `"it"`, `"mr"`, `"cds"`), matching Node 2.2 Section 6.3. Test ORC-16 added as regression test.

**Status:** Closed.
