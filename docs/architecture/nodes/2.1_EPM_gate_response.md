## Preamble

The RRC review (2.1_RRC_review.md) identified five gate-blocking issues and five non-blocking implementation prerequisites. The EPM has reviewed each issue. The gate decision is explained below.

**Core EPM principle applied here:** An RRC issue is gate-blocking only if it changes the architecture selection decision (which option to build). It is NOT gate-blocking if it affects implementation parameters (cost, latency) that will be validated empirically before commitment to production scale. Architecture decisions made now can survive empirical refinement of implementation parameters. Architecture decisions cannot survive a fundamental logic flaw in the selection reasoning.

## EPM Adjudication: Gate-Blocking Issues

### Issue 1 — Cost estimate likely wrong by 5x (reasoning token billing)

**RRC finding:** If reasoning tokens are billed as output tokens, actual cost is ~$0.14/ECG, not $0.03/ECG.

**EPM ruling: NOT gate-blocking for architecture selection.**

Rationale: The cost question is about DeepSeek billing mechanics, not about which architecture to build. All six candidate architectures use DeepSeek agents and all are equally affected by reasoning token billing. Option A (pipeline) and Option B (monolithic) would also incur higher costs if reasoning tokens are billed — the relative cost ranking between architectures is unchanged. Furthermore, $0.14/ECG remains commercially viable for a clinical decision support tool: comparable ML-based clinical diagnostics tools cost $0.50–$5.00 per analysis at commercial scale. The RRC's recomputed figure of $1,400/day for 10,000 ECGs is a plausible operating cost for a hospital-grade diagnostic service.

**Action required (pre-implementation, not pre-gate):** Verify DeepSeek reasoning token billing policy before the Node 2.6 (DeepSeek API Integration) implementation begins. Assign to Node 2.6. If cost exceeds $0.20/ECG, a cost reduction strategy (prompt length reduction, selective reasoner mode, lighter-weight model for Phase 1) must be evaluated at that node.

**Architecture document update:** Add a note to Section 6.3 acknowledging the reasoning token billing uncertainty and flagging it for Node 2.6 verification. The executive summary cost figure ($0.03/ECG) is identified as an estimate that excludes reasoning tokens pending billing policy confirmation. *(To be added as an addendum to Section 6.3 — not required for gate passage.)*

### Issue 2 — Latency estimate has no empirical basis

**RRC finding:** The 3-5 second per-agent figure is inconsistent with reasoning token counts at published DeepSeek throughput rates. Actual latency may exceed the 15-second clinical threshold.

**EPM ruling: NOT gate-blocking for architecture selection.**

Rationale: This is an empirical measurement gap, not a reasoning flaw. The 6-10 second estimate in the document is explicitly labelled as an estimate requiring pilot validation (Section 8.1). All candidate multi-agent architectures (B, C, D, E) have latency profiles that depend on the same DeepSeek inference speed — none has an empirically established advantage at this stage. If latency measurements reveal that Phase 1 agents take 15-25 seconds each, the architectural response is to reduce reasoning depth, shorten prompts, or use non-reasoner mode for straightforward Phase 1 tasks — not to abandon Option E for a different architecture.

The one case where a latency finding WOULD change the architecture: if DeepSeek's latency in reasoner mode proves so high that even a single-agent (Option B) call exceeds 60 seconds, the entire LLM-based architecture is untenable, and a different technology approach would be needed. This is an existential risk applicable to all options equally.

**Action required (pre-implementation):** Empirical latency benchmark is now a required deliverable for Node 2.1's pilot validation (Section 8). It must be completed before Node 2.2 (Individual Agent Design) begins prompt engineering. The SDA-2 contract should be updated to make the benchmark a gate for Node 2.2 activation, not Node 2.1.

### Issue 4 — Data residency risk not addressed (HIPAA/GDPR and DeepSeek as Chinese operator)

**RRC finding:** Transmitting patient-derived ECG features to a Chinese-operated API may be incompatible with HIPAA, GDPR, or applicable medical data laws. Not addressed anywhere in the document.

**EPM ruling: REAL RISK, but NOT architecture-selection-blocking.**

Rationale: The data residency concern is 100% valid and must be addressed. However, it does not change the selection between Option A–F. All six options use DeepSeek. If DeepSeek is found to be non-compliant for the target deployment context, the architectural response is to replace DeepSeek with a HIPAA-compliant LLM API (e.g., Azure OpenAI with Business Associate Agreement, AWS Bedrock, or a self-hosted model) — the Option E architecture itself is provider-agnostic. The document already states: "Architecture is model-agnostic. Can migrate to GPT-4, Claude, or Llama with prompt adaptation."

**The RRC is correct that this should not be in Section 9.5 as a minor bullet — it deserves its own treatment.**

**Action required (pre-deployment, tracked separately):** A data residency assessment must be completed before any integration with hospital infrastructure or PHI-containing systems. This assessment is out of scope for Node 2.1 (architecture selection) and belongs in the Master Charter's legal/compliance section. A new dynamic branch will be spawned: **Branch 4.5.2 — Regulatory Compliance & LLM Provider Assessment**. This branch will evaluate HIPAA BAA availability, GDPR data processing agreements, and alternative compliant LLM providers. The Safety Layer node (4.5) is the natural home.

**The architecture gate is not held pending this assessment** because: (a) the architecture is provider-agnostic, and (b) holding all architecture work pending a legal/compliance assessment of indeterminate duration would stall the entire project. The risk is logged and tracked.

### Issue 5 — Context window assumption may be outdated (64K vs 128K for DeepSeek R1)

**RRC finding:** DeepSeek R1 may have a 128K context window, not 64K. If true, Option D (Single Comprehensive Agent) becomes more viable and its scalability score should be revised.

**EPM ruling: NOT gate-blocking — margin is robust.**

The RRC's own weight reversal analysis showed Option E wins under all tested weight configurations. The relevant question is: does a 128K context window for Option D close the gap enough to reverse the decision?

EPM analysis:
- Option D's lowest score is Scalability (3/10) — this reflects 95-condition context saturation, not 13-condition context. Even at 128K, a system prompt + all 95 condition RAG contexts would overflow. The scalability knock is structural, not a window-size artifact.
- At 13 conditions specifically: yes, Option D at 128K is more viable than scored. The RRC's estimate of ~37,500 tokens for a selective RAG approach is credible. If the context window is 128K, Option D's scalability score for the 13-condition phase should be 7 (not 3), making Option D a closer competitor.
- Even with Option D's scalability rescored to 7: `0.25(9) + 0.20(9) + 0.15(7) + 0.10(7) + 0.10(6) + 0.10(7) + 0.10(7) = 2.25 + 1.80 + 1.05 + 0.70 + 0.60 + 0.70 + 0.70 = 7.80`. Option E (8.00) still wins by 0.20 — a smaller margin.

The RRC's recommendation to run Option D in parallel during the 10-scenario pilot is accepted and is now a required deliverable. If the pilot shows Option D matching or exceeding Option E's diagnostic quality, the architecture decision should be formally reviewed before Phase 2 agent design begins.

**Action required:** Verify DeepSeek R1 context window size. Update Section 6.1 with the confirmed figure and recompute Option D's scalability score accordingly. Document the revised margin. Add Option D as a parallel condition in the pilot validation (Section 8). Assign to Node 2.2 preparation tasks.
