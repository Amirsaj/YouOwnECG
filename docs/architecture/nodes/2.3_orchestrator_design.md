**Node:** 2.3
**SDA:** SDA-2 (Diagnosis & Agentic Core)
**Status:** Research Complete
**Type:** FD-FPRE (First-Principles Discovery)

**Dependencies:** Node 2.1 (Gate Passed 2026-03-27), Node 2.2 (Gate Passed 2026-03-28)

## Executive Summary

Node 2.3 defines the **runtime orchestrator**: the Python layer that coordinates agent execution in the Option E architecture (ADR-001). The orchestrator is not an LLM — it is the deterministic control plane that dispatches agents, collects outputs, detects STAT conditions, assembles the CDS input package, and routes the final report.

The central design question this node resolves is the **STAT fast-path timing decision**: STAT conditions (STEMI, VF, complete AVB, sustained VT, critical hyperkalemia, Brugada Type 1) must surface to the ER clinician within ~3-4 seconds — before the CDS agent completes. The orchestrator achieves this by monitoring Phase 1 agent outputs as they arrive, firing STAT alerts immediately when any Phase 1 agent sets `stat_flag = true`, and running CDS in the background to fill in context and confidence.

The second major design decision is **graceful degradation under partial failure**. In a 3-parallel-agent setup, a single agent timeout or API error must not silently produce an incomplete clinical assessment. This document defines exact fallback behavior for every failure mode: which findings are downgraded, which STAT conditions still fire (conservatively), and how the UI communicates partial analysis to the clinician.

What this document does NOT specify: the Python framework / async library implementation details (those are implementation-phase decisions), the exact API call structure (Node 2.6), or the report rendering format (SDA-3). Section 8 lists requirements this design places on those downstream nodes.

## 1. Orchestrator Role vs CDS Agent

The Option E architecture has two synthesis layers that must be precisely distinguished:

| | **CDS Agent** (Node 2.2) | **Orchestrator** (this node) |
|---|---|---|
| **Type** | LLM (DeepSeek R1 reasoner) | Python runtime (deterministic code) |
| **Role** | Clinical reasoning synthesis — resolve cross-domain interactions, calibrate confidence, produce unified diagnostic output | Runtime coordination — dispatch, collect, monitor, route |
| **Conflict resolution** | Clinical conflicts (e.g., Wellens + LBBB, concordant hint disagreement) | Runtime conflicts (timeout, schema validation failure, partial output) |
| **STAT detection** | Validates Phase 1 STAT flags, adds cross-domain context, may upgrade or downgrade | **Primary STAT fast-path**: fires immediately when Phase 1 STAT flag arrives |
| **Input** | Phase 1 structured outputs + raw feature selection (assembled by orchestrator) | Feature object from SDA-1 |
| **Output** | Unified clinical report JSON | Routing decisions, STAT alerts, assembled CDS input |

**Key rule:** The orchestrator performs zero clinical reasoning. All interpretation is deferred to the agents. The orchestrator's only clinical awareness is the binary `stat_flag` field in Phase 1 outputs — it does not inspect finding content, diagnosis codes, or confidence values to make routing decisions.

## 2. Pipeline Execution Model

### 2.1 Execution Phases

```
PHASE 0 — INGRESS (synchronous, <50ms)
  Receive feature object from SDA-1
  Validate feature_schema_version matches expected ("1.0")
  Validate required top-level fields present: ecg_id, recording_duration_sec, total_beats_analyzed, beats[], aggregate, signal_quality_per_lead
  Validate recording_duration_sec > 0 (reject malformed zero-duration recording with PIPELINE_ERROR)
  Validate beats[] contains at least one beat with beat_class in {0, 1, 2, 3} (class 4 = noise); if all beats are class 4, return PIPELINE_FAILED("NO_ANALYZABLE_BEATS")
  Validate all 12 CDS base fields (Section 5.2) resolve to non-null values in the feature object
  If any validation fails: return PIPELINE_ERROR with specific field and reason (do not invoke agents)

PHASE 1 — PARALLEL AGENT DISPATCH (async, ~3-5s wall time)
  Dispatch RRC, IT, MR agents concurrently
  Each agent call: POST to DeepSeek API with agent-specific system prompt + feature JSON
  Timeout per agent: 15 seconds (hard)
  Monitor Phase 1 outputs as they arrive:
    → On any output arrival: run STAT fast-path check (Section 3)
    → On STAT fast-path trigger: fire STAT alert to UI immediately (do not wait for Phase 1 completion)
  Await all three Phase 1 agents (with timeout)
  Validate each output schema (Section 4.1)

PHASE 1.5 — CDS INPUT ASSEMBLY (synchronous, <100ms)
  Assemble CDS input package from Phase 1 outputs + raw feature selection (Section 5)
  Apply token budget check (Section 6.2)

PHASE 2 — CDS DISPATCH (sequential, ~3-5s wall time)
  Dispatch CDS agent with assembled input
  Timeout: 20 seconds (hard)
  Receive CDS output
  Validate CDS output schema

PHASE 3 — REPORT ASSEMBLY (synchronous, <50ms)
  Merge Phase 1 outputs + CDS output into unified report structure (Section 7)
  Emit report JSON to Report node (SDA-3)
  Update STAT alert with full CDS context if STAT was already fired
```

### 2.2 Concurrency Model

Phase 1 agents run concurrently using `asyncio` with three concurrent API calls. The Phase 1 wall-clock latency is the latency of the **slowest** agent, not the sum. Expected:

| Agent | Estimated API latency | Notes |
|-------|----------------------|-------|
| RRC | 2.5–4.0s | Rhythm analysis; moderate reasoning |
| IT | 3.0–5.0s | Territory mapping; most reasoning for STEMI cases |
| MR | 2.5–4.0s | Morphology survey; similar to RRC |
| **Phase 1 total** | **3.0–5.0s** | Bounded by slowest agent |
| CDS | 3.0–5.0s | Cross-domain synthesis |
| **Pipeline total** | **6.0–10.0s** | Phase 1 + Phase 2 serial |

The STAT fast-path fires when the *first* Phase 1 agent to detect a STAT condition returns, which may be as early as 2.5 seconds.

## 3. STAT Fast-Path Design

### 3.1 Trigger Condition

The STAT fast-path fires when ANY Phase 1 agent returns an output containing `stat_flag = true` in ANY finding. The orchestrator does not read `stat_reason` or finding content — it fires the alert on the binary flag alone.

```python
def check_stat_fast_path(phase1_output: dict) -> bool:
    for finding in phase1_output.get("findings", []):
        if finding.get("stat_flag") is True:
            return True
    return False
```

### 3.2 STAT Alert Payload (Immediate)

When the fast-path fires, the orchestrator emits a STAT alert to the UI with the following payload. This is the **provisional** alert — it is updated when CDS completes.

```json
{
  "alert_type": "STAT_PROVISIONAL",
  "ecg_id": "string",
  "fired_by_agent": "RRC|IT|MR",
  "stat_findings": [
    {
      "agent": "string",
      "finding_description": "string",
      "confidence": "float",
      "stat_reason": "string"
    }
  ],
  "cds_status": "pending",
  "alert_timestamp_ms": "integer",
  "pipeline_phase": "phase1_partial|phase1_complete"
}
```

### 3.3 STAT Alert Update

When CDS completes, the orchestrator emits one of the following updates:

- **`STAT_CONFIRMED`**: CDS assessment agrees with Phase 1 STAT finding(s). Alert remains active.
- **`STAT_CONTEXTUAL`**: CDS adds important cross-domain context (e.g., "STAT STEMI pattern confirmed — however, LBBB present — Sgarbossa evaluation in progress"). Alert remains active at STAT level.
- **`STAT_DOWNGRADED`**: CDS resolves the STAT finding as a known mimic (e.g., Brugada Type 1 morphology in RBBB). The UI must clearly communicate the downgrade reason to the clinician.
- **`STAT_UNRESOLVED`**: CDS is uncertain — conflicting evidence, mimic diagnosis confidence below threshold, or the cross-domain interaction was inconclusive. Alert remains active. UI displays "Conflicting assessments — manual review required."

**`STAT_DOWNGRADED` confidence threshold:** A `STAT_DOWNGRADED` transition requires CDS to provide a mimic diagnosis with confidence ≥ 0.80 in the downgraded finding. If CDS confidence for the mimic diagnosis is < 0.80, the orchestrator emits `STAT_UNRESOLVED` instead. The rationale: a 95%-confidence STEMI should not be retracted by a 55%-confidence Brugada mimic assessment. This rule is enforced by the orchestrator, not by CDS.

**`STAT_DOWNGRADED` clinical governance:** The orchestrator executes the downgrade autonomously once the confidence threshold is met. The system does NOT require human confirmation of the downgrade before the UI updates. This is an explicit clinical risk acceptance: the system will occasionally autonomously retract a false-positive STAT alert. The risk is considered lower than the alert fatigue risk of requiring clinician acknowledgment before every STAT resolution. However, the SDA-3 UI must display `STAT_DOWNGRADED` with the original STAT finding and the downgrade reason side-by-side; the clinician must acknowledge the downgrade event (the alert is not simply removed). See Section 11.3.

**CDS-only STAT (no prior fast-path):** When `stat_fast_path_fired = false` and CDS output contains `stat_flag = true` in any finding (cross-domain STAT not detectable from single Phase 1 agent — see Section 3.5), the orchestrator fires a `STAT_CONFIRMED` event directly — skipping the provisional state. The latency for this alert is Phase 1 + Phase 2 (~6-10s), not the fast-path latency. This is acceptable for cross-domain STAT conditions that structurally require synthesis before detection.

**Clinical safety rule:** The orchestrator must NEVER suppress a Phase 1 STAT alert before CDS output arrives. If CDS times out, the Phase 1 STAT alert remains active. Under-alerting on a STEMI while waiting for CDS is unacceptable.

### 3.3.1 Multi-Agent STAT Behavior

If two Phase 1 agents both return `stat_flag = true` (e.g., RRC detects complete AVB and IT detects STEMI near-simultaneously), the orchestrator fires **all STAT conditions independently** — it does not suppress subsequent STAT alerts after the first one fires.

**Behavior:** The `stat_alert` object in the report uses an `alerts[]` array (not a single alert field) to accommodate multiple simultaneous STAT conditions. Each element has the same structure as the `STAT_PROVISIONAL` payload in Section 3.2. The `cds_status` field in each alert is updated independently when CDS completes.

**UI implication:** SDA-3 must render multiple simultaneous STAT alerts. The ordering follows the finding priority in Section 7.2 (ischemia > rhythm > conduction > ...). See Section 11.3 requirement 5.

**STAT deduplication on pipeline retry:** If the pipeline is retried for the same `ecg_id` (e.g., because CDS timed out and the caller retries), the orchestrator may re-fire STAT alerts from Phase 1 agents. To prevent duplicate UI alerts for the same ECG, the orchestrator interface accepts a `prior_stat_fired: bool` parameter. If `prior_stat_fired = true`, the fast-path STAT alert emission is suppressed on the retry run — Phase 1 STAT conditions are still recorded in the report but the UI is not re-notified. The caller (SDA-1→SDA-2 interface layer) is responsible for tracking `prior_stat_fired` state per `ecg_id`.

### 3.4 STAT Conditions by Owning Agent

| STAT Condition | Owning Agent | CDS Required? |
|---------------|-------------|---------------|
| VF / pulseless VT | RRC | No (CDS adds context only) |
| Sustained VT (Brugada criteria ≥6 beats) | RRC | Recommended (rule out VT vs SVT+aberrancy) |
| Complete AVB (3rd degree) | RRC | No |
| STEMI (any territory) | IT | Recommended (rule out mimics, confirm territory) |
| de Winter T-waves | IT | Recommended |
| Wellens syndrome | IT | Recommended (rule out LBBB interaction) |
| Brugada Type 1 coved | MR | Required (rule out RBBB mimic) |
| QTc ≥500ms + T-wave alternans | MR | Required (rule out drug QT, LBBB artifact) |
| Critical hyperkalemia (sine wave morphology) | MR | Recommended |
| Paced rhythm + STEMI equivalent (Sgarbossa+) | IT + RRC | Required (multi-agent) |

### 3.5 No Fast-Path for Cross-Domain STAT Conditions

Certain STAT conditions are CDS-only outputs — they cannot be reliably detected from a single Phase 1 agent's output alone:
- Left main stenosis (diffuse ST depression + aVR elevation) — requires IT + CDS
- AFib + pre-excitation (WPW + rapid conduction) — requires RRC + CDS
- VT vs SVT+aberrancy in borderline cases — requires RRC + MR + CDS

For these, the fast-path does NOT fire from Phase 1 alone. If any Phase 1 agent emits a constituent cross_domain_hint (e.g., `AFIB_DETECTED` + `WPW_DETECTED` in same RRC output), the orchestrator upgrades to a `STAT_PROVISIONAL` with `cds_status = "required"` and sends a warning to the UI: "Potentially critical combination detected — waiting for full synthesis."

## 4. Phase 1 Output Collection and Validation

### 4.1 Schema Validation

Each Phase 1 output is validated on arrival before STAT check:

**Required top-level fields:**
- `agent_id`: must be `"RRC"`, `"IT"`, or `"MR"`
- `schema_version`: must be `"1.0"`
- `processing_time_ms`: float
- `confidence_overall`: float in [0.0, 1.0]
- `findings`: array (may be empty)

**Finding validation:**
- Each finding must have `finding_id` (UUID), `category` (from controlled vocabulary), `confidence` (float), `stat_flag` (boolean)
- `cross_domain_hint` must have `hint_type` from the 17-type controlled vocabulary if present

If validation fails for any required field:
- Log the schema violation
- Mark the agent output as `PARTIAL` with `validation_error` field
- Proceed with degraded mode (Section 4.2)

### 4.2 Partial Failure Handling

| Failure Mode | Orchestrator Response |
|-------------|----------------------|
| Agent timeout (>15s) | Mark agent as `TIMED_OUT`; use empty output with `confidence_overall = 0.0`; continue to CDS with note |
| API error (5xx) received within <2s of call | Retry once (immediate, no backoff) — fast failure is likely transient; if retry fails, mark `API_FAILED` |
| API error (5xx) received after ≥2s of processing | Treat as equivalent to timeout — mark `API_FAILED` immediately (no retry); a slow 5xx means the server processed the request and failed, not a transient rejection; retrying will likely also fail or timeout |
| Schema validation failure | Mark `SCHEMA_INVALID`; flag all findings from that agent as `unverified`; continue |
| All 3 agents fail | Abort CDS; return `PIPELINE_FAILED` report with no findings |
| 2 agents fail | Attempt CDS with single Phase 1 output; CDS must mark all cross-domain inferences as `low_confidence` |
| CDS timeout (>20s) | Return Phase 1 findings only; mark CDS status as `incomplete`; STAT flags from Phase 1 remain active |
| CDS schema invalid | Return Phase 1 findings only; mark CDS status as `synthesis_failed` |

**Degraded report flag:** Any report produced with partial pipeline output must have:
```json
{
  "pipeline_status": "complete|partial|phase1_only|failed",
  "failed_agents": [],
  "clinical_confidence_note": "string describing what analysis is missing"
}
```

The UI (SDA-3) must display the degraded status prominently. A report missing IT Agent output is clinically incomplete for ischemia assessment — the UI must not present it as a full analysis.

### 4.3 Output Collection Ordering

Phase 1 agents are dispatched concurrently and collected as they complete (not in a fixed order). The orchestrator uses `asyncio.as_completed()` semantics — processing each result as it arrives for STAT fast-path, then waiting for all three before proceeding to Phase 1.5.

The completion order is non-deterministic and depends on DeepSeek API latency per call. This is expected and correct — CDS receives all three Phase 1 outputs regardless of arrival order.

## 5. CDS Input Assembly Protocol

### 5.1 Assembly Rules

The CDS input package (assembled in Phase 1.5) contains:

1. **Phase 1 structured outputs** — all three agents' full JSON outputs (or error stubs for failed agents)
2. **Raw feature selection** — the minimal feature subset assembled per Node 2.2 Section 5.2 protocol
3. **Orchestrator metadata** — pipeline timing, agent status, any fast-path alerts already fired

### 5.2 Raw Feature Selection

Per Node 2.2 design, CDS receives:

**Base fields (always included — 12 fields, exact names from Node 2.2 Section 1.3):**
- Top-level: `ecg_id`, `feature_schema_version`, `patient_sex`, `patient_age_years`, `recording_duration_sec`, `total_beats_analyzed`
- Aggregate: `aggregate.heart_rate_ventricular_bpm`, `aggregate.rhythm_regularity_coefficient` (dimensionless; <0.05 = regular), `aggregate.qrs_axis_degrees`, `aggregate.qtc_bazett_ms`, `aggregate.qtc_fridericia_ms`, `aggregate.av_ratio`

**Note:** LBBB/RBBB status is NOT in the SDA-1 feature schema — it is an agent output (RRC `conduction.bbb_type`). CDS obtains BBB status from the RRC Phase 1 output directly. The base feature fields contain only SDA-1-computed values.

**`raw_feature_flags` path format:**

Paths use **dot notation with array wildcard support**:
- Scalar path: `"aggregate.heart_rate_ventricular_bpm"` → resolves to a single value
- Array path with wildcard: `"beats[*].leads.V2.st_j60_mv"` → resolves to an array of N values (one per beat)
- Per-beat-per-lead wildcard: `"beats[*].leads[*].st_j60_mv"` → resolves to an N×12 array (all beats, all leads)

**Resolution semantics for array paths:**
- `beats[*].field` → include as a JSON array of N floats (preserves temporal sequence for agents)
- `beats[*].leads.V2.field` → include as JSON array of N floats for the specified lead
- `beats[*].leads[*].field` → include as a compact 2D JSON array (N beats × 12 leads); **this counts as ~N×12 tokens** — budget accordingly

**Lookup failure handling:** If a path does not resolve to a valid key in the feature object (malformed path or hallucinated field name), log the unresolved path, add it to `raw_feature_fields_dropped`, and continue.

**Conditionally added fields (per raw_feature_flags from Phase 1 agents):**
- Orchestrator collects all `raw_feature_flags` path strings from all Phase 1 findings
- Deduplicates paths
- Resolves each path using the rules above
- **Token budget cap: 2,000 tokens** for the raw feature section (base fields excluded from cap)

**Token budget enforcement:** Array paths are estimated before inclusion: N×1 tokens for beat arrays, N×12 tokens for per-lead arrays. The orchestrator pre-estimates and applies priority ordering if the budget would be exceeded:
1. Paths from STAT-flagged findings first
2. Paths flagged by multiple agents (corroborated)
3. Scalar paths before array paths (lower token cost per path)
4. Alphabetical within each tier

If still over budget: truncate lower-priority paths and log which fields were dropped in `raw_feature_fields_dropped`.

### 5.3 Cross-Domain Hint Aggregation

The orchestrator collects all `cross_domain_hint` objects from all Phase 1 findings and assembles them into a flat hint array for CDS:

```json
{
  "cross_domain_hints": [
    {
      "source_agent": "RRC|IT|MR",
      "hint_type": "enum from 17-type vocabulary",
      "payload": {},
      "description": "string"
    }
  ]
}
```

This is the primary communication channel from Phase 1 to CDS. The orchestrator does not filter, prioritize, or interpret hints — CDS receives all of them and applies its priority queue (Node 2.2 Section 5.4).

### 5.4 CDS Input Package Structure

```json
{
  "package_version": "1.0",
  "ecg_id": "string",
  "pipeline_run_id": "UUID",
  "phase1_outputs": {
    "rrc": { /* full RRC output or error stub */ },
    "it": { /* full IT output or error stub */ },
    "mr": { /* full MR output or error stub */ }
  },
  "phase1_agent_status": {
    "rrc": "complete|timed_out|api_failed|schema_invalid",
    "it": "complete|timed_out|api_failed|schema_invalid",
    "mr": "complete|timed_out|api_failed|schema_invalid"
  },
  "cross_domain_hints": [ /* aggregated from all Phase 1 findings */ ],
  "raw_features": { /* base + flagged fields, ≤2,000 tokens */ },
  "raw_feature_fields_dropped": [ /* fields excluded due to token budget */ ],
  "stat_fast_path_fired": "boolean",
  "stat_fast_path_agents": [ "RRC|IT|MR" ]
}
```

## 6. Token and Latency Budget

### 6.1 Per-Agent Token Budget

| Component | Token Estimate |
|-----------|---------------|
| RRC system prompt (Node 2.2 Section 2.2) | ~2,200 tokens |
| IT system prompt (Node 2.2 Section 3.2) | ~2,550 tokens |
| MR system prompt (Node 2.2 Section 4.2) | ~2,400 tokens |
| Feature object (full, per VT-2.2-01 measurement) | 3,500–5,000 tokens |
| **Phase 1 input per agent** | **~5,700–7,550 tokens** |
| Phase 1 reasoning tokens (DeepSeek R1, typical) | ~2,000–5,000 tokens |
| Phase 1 output JSON per agent | ~1,000–1,500 tokens |

**Note:** VT-2.2-01 (empirical token measurement of feature object) must be completed before implementation. If the feature object exceeds 5,000 tokens, Phase 1 input may approach 10,000 tokens, which is acceptable within a 64K context window but must be reflected in cost projections.

### 6.2 CDS Token Budget

| Component | Token Estimate |
|-----------|---------------|
| CDS system prompt (Node 2.2 Section 5.2) | ~3,000 tokens |
| Phase 1 structured outputs (3 agents × ~1,200 tokens simple / ~3,000 tokens complex) | ~3,600–9,000 tokens |
| Raw feature selection (base + flagged) | ≤2,000 tokens |
| Cross-domain hints array | ~300–600 tokens |
| Orchestrator metadata | ~100 tokens |
| **CDS input total (simple ECG)** | **~9,000–9,300 tokens** |
| **CDS input total (complex ECG — VT + STEMI + multiple findings)** | **~12,000–15,000 tokens** |
| CDS reasoning tokens | ~3,000–8,000 tokens |
| CDS output JSON | ~2,000–3,000 tokens |

Total CDS context: ~14,000–23,000 tokens for complex cases — well within 64K context window. **VT-2.3-01** (see Section 12) requires empirical measurement of Phase 1 output sizes across representative ECGs to validate and update these estimates before implementation.

### 6.3 Latency Budget

| Phase | Target | Hard Limit |
|-------|--------|-----------|
| Phase 0 (validation) | <50ms | 200ms |
| Phase 1 (parallel dispatch, slowest agent) | 3–5s | 15s (per agent timeout) |
| STAT fast-path fire (first agent with STAT) | 2.5–4s | 15s |
| Phase 1.5 (CDS input assembly) | <100ms | 500ms |
| Phase 2 (CDS) | 3–5s | 20s |
| Phase 3 (report assembly) | <50ms | 200ms |
| **Pipeline total (target)** | **6–10s** | **36s (sum of all hard limits)** |

The 36-second absolute hard limit is conservative — the 15+20 second timeouts are independently applied per agent. In practice, hard limit scenarios only occur when both Phase 1 and CDS time out (extreme API degradation).

## 7. Report Assembly

### 7.1 Unified Report Structure

The orchestrator assembles the final report JSON from Phase 1 outputs + CDS output. The report is the input to SDA-3 (UI rendering) and is also persisted to the database.

```json
{
  "report_version": "1.0",
  "ecg_id": "string",
  "pipeline_run_id": "UUID",
  "pipeline_status": "complete|partial|phase1_only|failed",
  "pipeline_timing": {
    "phase0_ms": "integer",
    "phase1_ms": "integer",
    "phase1_5_ms": "integer",
    "phase2_ms": "integer",
    "phase3_ms": "integer",
    "total_ms": "integer"
  },
  "agent_status": {
    "rrc": "complete|timed_out|api_failed|schema_invalid",
    "it": "complete|timed_out|api_failed|schema_invalid",
    "mr": "complete|timed_out|api_failed|schema_invalid",
    "cds": "complete|timed_out|api_failed|schema_invalid|skipped"
  },
  "stat_alerts": [
    {
      "fired_by_agent": "rrc|it|mr|cds",
      "status": "STAT_CONFIRMED|STAT_CONTEXTUAL|STAT_DOWNGRADED|STAT_UNRESOLVED|pending|none",
      "conditions": [ /* STAT finding summaries */ ],
      "fast_path_fired_at_ms": "integer | null",
      "cds_update_at_ms": "integer | null",
      "downgrade_confidence": "float | null"
    }
  ],
  "stat_fast_path_fired": "boolean",
  "phase1_outputs": {
    "rrc": { /* full RRC output JSON or null */ },
    "it": { /* full IT output JSON or null */ },
    "mr": { /* full MR output JSON or null */ }
  },
  "synthesis": { /* CDS output JSON or null */ },
  "clinical_confidence_note": "string | null"
}
```

### 7.2 Finding Priority Ordering

The report assembles findings in priority order for the UI. The **primary sort key** is `clinical_urgency_rank` (integer 1..N, assigned by CDS — 1 = most urgent). This delegates clinical priority ordering to CDS, which understands clinical context. The orchestrator does not reinterpret clinical urgency.

If CDS did not produce a `clinical_urgency_rank` (e.g., CDS failed and Phase 1-only report is used), the fallback ordering is:

1. **STAT-flagged findings** — ordered by confidence descending
2. **High-confidence non-STAT** (confidence ≥0.8)
3. **Moderate-confidence** (confidence 0.5–0.8)
4. **Low-confidence** (confidence <0.5)

Within each fallback tier, findings are ordered by clinical category priority: ischemia > rhythm > conduction > morphology > structural > metabolic > normal variants.

This ordering is performed by the orchestrator, not by SDA-3. SDA-3 receives the pre-ordered finding list and renders it.

**Note:** This requires a `clinical_urgency_rank` field in the CDS output schema (Node 2.2 amendment — see Section 8 cross-document changes).

## 8. Cross-Domain Hint Deduplication

A clinical situation may generate the same hint type from multiple agents. For example, both IT (`LBBB_DETECTED` from observing LBBB-morphology ST patterns) and RRC (`LBBB_DETECTED` from QRS duration + morphology classification) could emit `LBBB_DETECTED`. The orchestrator deduplicates hints by `hint_type`:

- If the same `hint_type` appears multiple times: keep all instances but tag as `corroborated = true` (multiple agents agree)
- CDS receives both instances and should treat corroborated hints with higher confidence
- If contradictory hints appear (e.g., RRC emits `LBBB_DETECTED` but MR emits `BRUGADA_MORPHOLOGY_CONFIRMED` for the same V1 morphology): both are passed to CDS; the orchestrator adds a `hint_conflict: true` flag — this triggers CDS to apply explicit conflict resolution (Node 2.2 Section 5.5 interaction 1)

## 9. Failure Mode Catalog

### 9.1 SDA-1 Feature Object Invalid

**Trigger:** Feature schema version mismatch or required field missing in Phase 0 validation.

**Response:** Return `PIPELINE_FAILED` immediately. Log specific validation error. Do not invoke any agents. The SDA-1 pipeline has a bug — this must be investigated before retrying.

**Clinical implication:** No analysis produced. The UI must display "Analysis unavailable — signal quality or processing error" with the specific validation error (for technical staff only, not displayed to clinicians).

### 9.2 All Phase 1 Agents Fail

**Trigger:** All three of RRC, IT, MR time out or return API errors.

**Response:** Abort CDS. Return `PIPELINE_FAILED`. Log all three errors with API response codes.

**Clinical implication:** This is an API availability failure, not a clinical failure. The system should queue the ECG for retry when the API recovers.

### 9.3 Single Phase 1 Agent Fails (IT Agent)

**Trigger:** IT Agent times out or returns schema-invalid output.

**Response:** Continue with RRC + MR outputs only. Assemble CDS input with `IT: null` and `phase1_agent_status.IT = "timed_out"`. CDS must explicitly note that ischemia assessment is incomplete.

**Clinical implication:** This is the highest-risk partial failure — STEMI may go undetected if IT fails. The report must display a prominent warning: "Ischemia analysis incomplete — manual review required." STAT fast-path cannot fire for ischemia in this scenario.

### 9.4 Single Phase 1 Agent Fails (RRC or MR)

**Response:** Continue with remaining two agents. CDS notes missing agent. STAT alerts from the two surviving agents still fire normally.

### 9.5 CDS Fails After Phase 1 Success

**Response:** Return Phase 1 outputs directly as the report. Mark `pipeline_status = "phase1_only"`. All STAT alerts from Phase 1 remain active. Add `clinical_confidence_note: "Cross-domain synthesis unavailable — findings from individual analyses only. Manual review of cross-domain conditions recommended."`.

**Clinical implication:** Phase 1-only reports are clinically useful — all three domain specialists have analyzed the ECG. The main loss is cross-domain interaction resolution (e.g., LBBB + STEMI Sgarbossa evaluation). The UI must surface this clearly.

## 10. Idempotency and State Management

The orchestrator is **stateless** — each pipeline run is independent. The `pipeline_run_id` (UUID, generated at Phase 0) uniquely identifies each run. The feature object is the sole input — no state from prior ECGs is read during the pipeline run.

**No retry loops inside the orchestrator.** Each agent is called once (with one retry on API error per Section 4.2). If retries fail, the orchestrator proceeds with degraded mode. Retry decision-making is the responsibility of the caller (the SDA-1→SDA-2 interface layer).

**Determinism at temperature=0.** Both DeepSeek R1 reasoner calls at temperature=0 and the same input should produce identical or near-identical outputs. This is the expected behavior but is not guaranteed by the API (floating-point non-determinism, load-balancing routing). For audit purposes, `processing_time_ms` per agent and `pipeline_run_id` are recorded in every report.

## 11. Downstream Requirements

### 11.1 Requirements on Node 2.6 (DeepSeek API Integration)

The orchestrator requires the following from the API integration layer:

1. **`async def call_agent(agent_id, system_prompt, user_content, timeout_s) -> dict`** — returns parsed JSON output or raises `AgentTimeoutError` / `AgentAPIError`
2. **Concurrent execution support** — three simultaneous API calls must be supported (within rate limits)
3. **Latency-aware error** — `AgentAPIError` must expose `response_latency_s: float` — the time elapsed from call start to error. The orchestrator uses this to distinguish fast failures (<2s) from slow failures (≥2s, treated as equivalent to timeout — see Section 4.2)
4. **No automatic retries on timeout** — timeout is orchestrator-managed; Node 2.6 must raise `AgentTimeoutError` at exactly the specified timeout, not retry
5. **Structured error types** — `AgentTimeoutError`, `AgentAPIError(status_code, message, response_latency_s)`, `AgentSchemaError(field, expected, got)` — all catchable separately by the orchestrator
6. **Native async HTTP client** — the `call_agent()` implementation must use a native async HTTP client (e.g., `httpx.AsyncClient`). Synchronous HTTP clients wrapped in `asyncio.to_thread()` are not acceptable — they block the event loop and delay STAT fast-path processing when one agent completes while another is still in flight

### 11.2 Requirements on SDA-1 (Feature Extraction)

1. **Feature schema version** — feature object must include `feature_schema_version: "1.0"` at top level
2. **Complete required fields** — all fields listed in Node 2.2 Section 1.1 must be present (null is acceptable for conditional fields; missing keys are not)
3. **`recording_duration_sec > 0`** — must be a positive float; zero values are rejected by Phase 0 validation
4. **At least one analyzable beat** — `beats[]` must contain at least one beat with `beat_class` in {0, 1, 2, 3}; all-noise recordings (`beat_class = 4` for all beats) are rejected by Phase 0 validation with `NO_ANALYZABLE_BEATS`

### 11.3 Requirements on SDA-3 (UI/UX)

1. **STAT alert array rendering** — UI must support an `alerts[]` array (multiple simultaneous STAT conditions); each alert can independently transition through `STAT_PROVISIONAL → STAT_CONFIRMED/CONTEXTUAL/DOWNGRADED/UNRESOLVED`
2. **`STAT_DOWNGRADED` acknowledgment** — UI must display the original STAT finding and the downgrade reason side-by-side; the clinician must explicitly acknowledge the downgrade event; the alert must not be removed without acknowledgment
3. **Partial pipeline display** — UI must display `pipeline_status` prominently when `partial` or `phase1_only`
4. **Finding priority ordering** — UI renders the pre-ordered finding list (`clinical_urgency_rank` ordered by orchestrator); does not reorder
5. **`clinical_confidence_note`** — must be displayed in a visually distinct area when present
6. **Normal ECG rendering** — when CDS output contains `no_significant_findings: true` (or equivalent), UI must render an explicit affirmative normal conclusion, not simply an empty finding list
