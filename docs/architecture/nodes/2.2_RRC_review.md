## 1. Feature Input Design Critique

### 1.1 Full-Feature Routing — Acknowledged Downsides Not Fully Addressed

The document justifies full-feature routing primarily by convenience: it avoids pre-classification and simplifies the orchestrator (lines 86–89). The rationale is defensible but the following real costs are not acknowledged:

**Token inflation at the tail.** The estimate of "1,800–2,600 tokens in JSON" for a 10-second ECG (~12 beats, 12 leads) is almost certainly understated. A single beat has 11 per-lead fields × 12 leads = 132 float values. At 12 beats, that is 1,584 leaf values in the `beats[]` array alone. At ~4 characters per float + JSON overhead (key names, brackets, indentation), the actual token count for the beats block is closer to 3,000–4,000 tokens. Add the aggregate and metadata blocks and the realistic estimate is 3,500–5,000 tokens — which violates the stated "1,500–2,500 token Phase 1 input budget from Node 2.1 Section 6.1." This is not a minor discrepancy. The document must either re-estimate carefully (building an actual JSON serialization of a representative 12-beat ECG and running it through a tokenizer) or acknowledge that the Phase 1 token budget must expand, and revise the latency and cost projections accordingly.

**Reasoning dilution.** The document asserts that "LLMs can follow scoped instructions reliably at temperature=0" (line 88), but this claim is unsupported by any empirical evidence in this design family. An RRC Agent receiving T-wave amplitude data for all 12 leads may or may not ignore it cleanly. At temperature=0 on DeepSeek R1, the model tends to comment on salient features it was told to ignore — it will note a coved ST morphology in V1 even when scoped to "rhythm/rate/conduction only." This is not a theoretical concern: it leads to findings leaking from one agent's scope into another's output schema in unpredictable ways. No measurement plan is proposed to test this assumption before implementation.

**No versioning/schema evolution path.** The feature schema is defined with no version field in Section 1.1 or 1.3. The feature object JSON in Section 1.3 has `ecg_id` at the top level but no `feature_schema_version`. As SDA-1 is developed and feature definitions change (e.g., the T-wave symmetry formula changes, or new leads are added), the agents will silently receive a different schema than they were designed for. This is a soft dependency that must be versioned.

### 1.2 Missing Critical Features

**QT interval dispersion is absent.** QT dispersion (max QT − min QT across leads) is an independent predictor of TdP risk and is used in risk stratification for Long QT syndrome. The MR Agent is responsible for QTc interpretation but has no per-lead QT interval access in the feature schema — only aggregate `qtc_bazett_ms` and `qtc_fridericia_ms`. To compute QT dispersion, the MR Agent would need `qt_intervals_ms[]` per lead, not just per beat averaged across leads. This is a material omission for a system explicitly handling Long QT / TdP risk.

**R-wave peak time (intrinsicoid deflection) is absent.** The interval from QRS onset to R-wave peak in V5-V6 (normal <45ms) and V1 (normal <35ms) is used in the LVH and RVH diagnostic criteria (Cornell voltage requires this indirectly through axis). More critically, it is used in BBB classification criteria. ECGdeli FPT has the R fiducial — this is computable and should be included.

**P-wave duration is absent from per-beat fields.** The document lists `p_wave_morphology_lead_ii[]` (amplitude only) at line 43. P-wave duration (Poff − Pon) is the primary criterion for Left Atrial Enlargement (≥120ms) and is also needed to distinguish MAT from sinus tachycardia. The RRC Agent's reasoning Step 2b references "notched/wide → LAE" (line 194) but the feature to measure this (P-wave duration) is not in the per-beat schema. The MR Agent's Step 6c (line 583) also refers to "notched P in II (duration ≥120ms)" without having access to the measured duration. This is an inconsistency between the reasoning chains and the declared feature schema.

**V4R is referenced but not defined.** Section 5.5 item 4 (line 775) references "raw feature `beats[*].leads.V4R`" for RV STEMI assessment. V4R is a right-sided lead not in the standard 12-lead schema. There is no discussion of how this lead is acquired, whether SDA-1 supports it, or what happens when it is absent. The document should either remove the V4R reference and substitute with V1 ST elevation criteria alone (acceptable for clinical use), or explicitly specify that V4R is an optional lead that CDS falls back from gracefully.

**`st_morphology` computation is underspecified in the feature schema.** The per-lead `st_morphology` enum includes `coved` (line 49). Computing whether an ST segment is "coved" vs "saddle" requires fitting the ST segment curve and classifying its shape — this is a non-trivial morphology classifier. Calling it a feature in Section 1.1 implies SDA-1 produces it automatically. Section 9.3 item 3 (line 1206) acknowledges "this requires a morphology classifier" but gives no algorithm, no performance requirement, and no fallback. Given that `brugada_morphology_v1` in the MR output (line 633) depends entirely on this field, a faulty morphology classifier breaks an entire diagnostic pathway. This is an unresolved dependency.

### 1.3 Feature Schema Completeness vs. Token Budget Conflict

The schema is comprehensive to a fault given the token constraints. No analysis is provided of which features could be omitted for ECGs that don't require them (e.g., V7-V9 leads are not in the 12-lead and are explicitly mentioned as "not in the standard 12-lead" at line 514 but there is no conditional inclusion mechanism).

## 2. Output Schema Critique

### 2.1 RRC Output Schema

**`rhythm.classification` enum is incomplete.** The enum at line 247 includes `sick_sinus` and `other` but omits `atrial_tachycardia` — an important rhythm that is neither sinus tachycardia (ectopic P-wave morphology), SVT in the traditional sense, nor MAT. Atrial tachycardia with AV block is a classic digoxin toxicity finding and should be in the vocabulary. The current enum forces the RRC Agent to classify it as `svt` or `other` — either is wrong or ambiguous.

**`p_waves.duration_ms` is in the output schema (line 261) but not in the input feature schema.** The RRC Agent is asked to output P-wave duration but Section 1.1 does not include a `p_wave_duration_ms[]` per-beat field. This forces the agent to infer or hallucinate the value. This inconsistency will produce unreliable outputs.

**`wide_complex_tachycardia.concordance_assessment` is in the RRC schema (line 289).** Concordance (whether QRS complexes in precordial leads all deflect in the same direction) is a morphology assessment that the document assigns to IT Agent (territorial concordance). Having it partially in the RRC output and not explaining who owns it creates a dual-ownership problem. The definition at line 289 says `"positive|negative|none|not_assessed"` but the Brugada algorithm's use of concordance is described as a CDS step using "IT concordance" (line 163). There are now two concordance assessments with no specification of which one CDS uses.

**`av_dissociation_type` field (line 269): `"complete_avb|vt|isorhythmic|null"** — isorhythmic AV dissociation is a benign condition (near-equal atrial and ventricular rates with no pathological block). Mixing it with VT and complete AVB in the same enum field without a separate severity indicator risks CDS treating isorhythmic dissociation with the same urgency as VT. The STAT flag logic in Step 7b should explicitly state that isorhythmic AV dissociation does NOT receive a STAT flag, but the schema does not enforce this.

### 2.2 IT Output Schema

**`j_point_measurements{}` field (lines 443–447) returns aggregate (single float per lead), not per-beat values.** During arrhythmias or rate-dependent ST changes, the J-point shifts beat to beat. A single per-lead value loses this information. Sgarbossa scoring (which CDS applies) should ideally use per-beat values to find the worst-case beat. The document provides per-beat per-lead ST values in the raw feature input but the IT output schema condenses this to a single value per lead, discarding the per-beat variation. This is a lossy aggregation with clinical consequence.

**`q_waves.type` enum (line 474): `"new|old|unknown"`.** There is no basis in a single 12-lead ECG to classify Q-waves as "new" vs "old" without comparison to a prior tracing. The IT Agent has no access to prior ECG data — the feature object contains only the current ECG. Outputting `new` will be either a hallucination (the agent guesses based on clinical context) or always `unknown`. This field should be removed from the Phase 1 IT schema and made a CDS field that is populated if/when a prior ECG is available.

**`avr_st_status.left_main_pattern: boolean` (line 451).** This boolean is computed by the IT Agent, but the document separately lists `LEFT_MAIN_PATTERN` as a cross_domain_hint (Section 6.2, line 982). CDS is supposed to respond to the hint and "integrate all three Phase 1 outputs" to make the diagnosis. If the IT Agent itself sets `left_main_pattern = true`, it has already made the diagnosis unilaterally, bypassing the CDS integration requirement. There is a conflict between the boolean in the IT schema and the cross_domain_hint architecture.

**No handling of ST changes during paced rhythm.** The IT schema has no `paced_rhythm` flag, and there is no mention of pacemaker artifact handling. The RRC conduction enum includes `paced` (line 273) but the IT agent has no corresponding LBBB-analog suppression rule for paced rhythms. Pacemaker artifact mimics LBBB morphology and makes ST analysis unreliable by the same mechanism as LBBB. A `PACED_RHYTHM_ST_UNRELIABLE` hint type is absent from the vocabulary.

### 2.3 MR Output Schema

**`brugada_morphology_v1: "type1_coved|type2_saddle|type3|normal"` (line 633).** The Brugada Type 3 classification was officially retired in the 2013 HRS/EHRA expert consensus statement on Brugada syndrome. The current international classification recognizes only Type 1 (diagnostic) and the "Brugada pattern" (formerly Type 2, the saddle-back — not diagnostic, requires provocation testing or Type 1 conversion). Including `type3` perpetuates an obsolete taxonomy and may cause confusion. This enum should be updated to `type1_coved|brugada_pattern|normal`.

**`t_wave_morphology.distribution_class` enum (line 620): `"territorial|diffuse|right_precordial|left_precordial|inferior|global"`.** The values `right_precordial`, `left_precordial`, `inferior`, and `global` are spatially specific but `diffuse` and `global` are near-synonyms without a defined distinction. The enum should be mutually exclusive and exhaustive with explicit definitions. As written, the MR Agent could plausibly output either `diffuse` or `global` for the same pattern, making test assertions fragile.

**`wellens_type` appears in both the MR and IT output schemas.** IT schema line 459 has `wellens_type: "A|B|null"` and MR schema line 623 also has `wellens_type: "A|B|null"`. There is no specification of which agent's Wellens classification CDS uses when they disagree. The document implies IT owns Wellens (Section 3.1: "ischemic T-wave changes in the context of territory" including Wellens) but MR Step 2d (line 558) also detects Wellens. This is a genuine dual-ownership conflict that will produce discordant outputs that CDS has no defined protocol to resolve.

### 2.4 CDS Output Schema

**`primary_findings[].condition: "string (condition name from controlled vocabulary)"** — the "controlled vocabulary" for condition names is never defined in the document. Without it, the CDS Agent will generate free-text condition names inconsistently: "STEMI anterior" vs "Anterior ST-elevation MI" vs "LAD territory STEMI" across different calls. This breaks downstream parsing, reporting, and regression test assertions. A controlled vocabulary for condition names must be defined in this document or referenced from another.

**`stat_class: "immediate|urgent|non_urgent|null"` (line 872).** The distinction between `immediate` and `urgent` is clinically meaningful (seconds vs minutes) but not defined. CDS will assign these values based on reasoning that has no documented basis, making the classification unreliable. A table mapping condition types to stat_class values is required.

## 3. cross_domain_hint Vocabulary Critique

### 3.1 Missing Hint Types

**Absence of `PACED_RHYTHM_DETECTED`.** As noted above, a paced rhythm is as disruptive to ST analysis as LBBB. The RRC Agent can detect paced rhythm from `beat_class[]` and QRS morphology, but there is no defined mechanism for this finding to suppress IT's ST analysis or trigger a modified assessment protocol for CDS. This is a structural gap.

**Absence of `ARVC_SUSPECT`.** The MR Agent at Step 5d (line 577) detects epsilon waves and at Step 2c (line 557) flags right precordial T-wave inversions (V1-V4) as possibly ARVC. But there is no `ARVC_SUSPECT` hint to coordinate a CDS response. The MR Agent can emit a finding, but without a structured hint, CDS has no trigger to load ARVC differential RAG or to check RRC for right-sided PVCs with LBBB morphology.

**Absence of `HYPERKALEMIA_SUSPECT`.** The `PEAKED_T_TERRITORIAL_CHECK` hint from MR is designed to resolve peaked T-waves as ischemic vs metabolic. But hyperkalemia has a clinical escalation path (sine-wave morphology → STAT) that extends beyond peaked T-waves: PR prolongation, QRS widening, absent P-waves. The `PEAKED_T_TERRITORIAL_CHECK` hint only carries T-wave data. CDS needs RRC's conduction data (QRS width, PR, P-wave absence) to stage hyperkalemia severity. A structured `HYPERKALEMIA_SUSPECT` hint that carries QRS duration and P-wave status from RRC — and is cross-referenced by CDS against MR's peaked T findings — would be more complete than the current design.

**Absence of `EARLY_REPOLARIZATION_PATTERN`.** Early repolarization (ER) — J-point elevation with notching or slurring in lateral/inferior leads — is a meaningful differential for STEMI (particularly in young males) and is also associated with ventricular fibrillation risk in its "malignant" form (inferior + lateral involvement with horizontal/descending ST). The MR Agent mentions early repolarization in its mission statement (line 526) but there is no corresponding hint type, no defined reasoning steps for ER in the MR reasoning chain, and no CDS resolution protocol. Early repolarization vs STEMI is a common and dangerous diagnostic confusion.

**Absence of `ELECTROLYTE_PATTERN_QTC`.** The document handles QTc prolongation with `CRITICAL_QTC_PROLONGATION` but has no mechanism for MR to flag suspected hypokalemia/hypomagnesemia (U-waves + prolonged QT) as a structured hint requiring clinical electrolyte correlation. The U-wave assessment (Step 7, lines 587–590) is an orphaned step — it produces findings but no hints and no defined CDS response.

### 3.2 Redundant/Ambiguous Hint Types

**`POSTERIOR_STEMI_SUSPECT` (from IT) and `POSTERIOR_STEMI_MORPHOLOGY` (from MR)** are described as complementary: CDS requires both to confirm posterior STEMI (Section 5.4, third row). This design is correct. However, the CDS response when only ONE of the two is present is not defined. If IT fires `POSTERIOR_STEMI_SUSPECT` but MR does not fire `POSTERIOR_STEMI_MORPHOLOGY` (because R/S ratio in V1 is not >1 due to concomitant anterior forces), does CDS still flag a possible posterior STEMI? Section 8.2 Scenario 3 assumes both hints fire, but real posterior STEMIs do not always show dominant R in V1 (it depends on timing in the infarction course). The single-hint scenario requires a defined response.

**`WCT_AV_DISSOCIATION` and `WCT_NO_AV_DISSOCIATION`** are mutually exclusive hint types from the same agent for the same condition. They represent the two branches of the RRC assessment. However, both have "required CDS actions" that ultimately both run the Brugada/Vereckei algorithm (lines 324–327). The distinction between them is only in confidence level — with AV dissociation, VT probability is already high. This could be simplified to a single `WIDE_COMPLEX_TACHYCARDIA` hint with a payload field `av_dissociation_confirmed: boolean`, reducing the hint vocabulary from 14 to 13.

### 3.3 Hint Priority Order Gap

The CDS Step 2 priority list (line 818) is: `WCT > LBBB > CRITICAL_QTC > LEFT_MAIN > INFERIOR_STEMI > POSTERIOR_STEMI > PEAKED_T > BRUGADA > others`. The hint type `COMPLETE_AVB_ISCHEMIC_CONTEXT` is not in this priority list. It appears in the vocabulary (Section 6.2) and has a defined CDS response (cross-reference IT for inferior/anterior STEMI) but its position in the processing queue is undefined. In a scenario with simultaneous inferior STEMI + complete AVB, both `INFERIOR_STEMI_EXTENSIONS` and `COMPLETE_AVB_ISCHEMIC_CONTEXT` will fire. Their relative processing order affects the output. The priority list is incomplete.

Similarly, `LVH_STRAIN_PRESENT` and `WPW_DETECTED` are also absent from the priority queue. The document lists 14 hint types but the CDS priority queue names only 8.

## 4. CDS Design Critique

### 4.1 Raw Feature Selection Protocol — Workability

**Feature path notation is undefined.** Section 5.3 introduces path strings like `"beats[*].leads.V1.st_j60_mv"` (line 728). The `[*]` wildcard notation is borrowed from JSONPath but its semantics here are ambiguous: does it mean "all beats" (returning an array), "the mean across beats," or "the worst-case beat"? For Sgarbossa, CDS needs the maximum concordant ST elevation across all beats. For QRS morphology, CDS needs a representative beat. The path notation needs formal semantics, not a suggestive example, or the orchestrator feature extractor has no unambiguous implementation specification.

**Budget priority (RRC > IT > MR) is clinically questionable in one scenario.** If a patient has an inferior STEMI with RV extension, the IT agent's flags (`beats[*].leads.V1.st_j60_mv`, `beats[*].leads.II.st_j60_mv`, etc.) are the most critical raw features for RV assessment. But under budget contention, RRC flags take precedence. In practice, an RRC flag for the same VT might consume the budget before IT's ST data is included — and CDS would apply the INFERIOR_STEMI_EXTENSIONS resolution without the raw ST measurements. This is a silent degradation that should at minimum trigger a `raw_feature_budget_exceeded` warning in the CDS output.

### 4.2 Dynamic RAG Trigger Logic

**The RAG triggers in Section 5.4 are all or nothing.** There is no mechanism to load partial RAG when the budget is exceeded. The document says "load in priority order until budget is exhausted" (line 750) but does not define the priority order for RAG chunks (different from the hint priority order). If Sgarbossa RAG (~600 tokens) and Brugada RAG (~800 tokens) are both triggered, they consume 1,400 of the 2,500-token RAG budget. Adding the next trigger (posterior STEMI, ~400 tokens) brings it to 1,800. A fourth trigger (inferior STEMI extensions, ~500 tokens) brings it to 2,300. The fifth (hyperkalemia, ~400 tokens) would push to 2,700 — over budget. In this scenario, the hyperkalemia RAG is dropped. But if the diagnosis is hyperkalemia masking STEMI (interaction 8 in Section 5.5), the dropped RAG is the one CDS needs. No fallback or warning mechanism is specified.

**RAG trigger for `afib + prolonged QTc` fires on `rhythm.classification = "afib"` (line 748).** But AFib is classified in the `rhythm` object of RRC output, and the orchestrator must compare it to MR's `qtc_class`. This is the only RAG trigger that requires a multi-agent logical AND evaluated by the orchestrator. All other triggers are single-hint-type triggers. The compound logic introduces a special case in orchestrator implementation that is not acknowledged as such.

### 4.3 Cross-Domain Interactions — Incomplete or Incorrect Resolutions

**Interaction 4 (Inferior STEMI → RV STEMI): V4R dependency is unresolved.** As noted in Section 1.2 of this review, V4R is referenced in the resolution but is not in the standard 12-lead feature schema and may not be available. The fallback when V4R is absent is not defined. Clinically, V1 ST elevation ≥1mm is the standard substitute for V4R in this context — this should be the stated primary criterion with V4R as an optional enhancement.

**Interaction 7 (Hyperkalemia mimics Brugada): The resolution logic is incomplete.** The resolution at lines 787–788 says "if diffuse peaked T + wide QRS → hyperkalemia more likely." However, true Brugada syndrome can coexist with fever-induced QRS widening and peaked T-waves. The resolution does not specify what confidence threshold the CDS should use for Brugada when there is a concurrent hyperkalemia signal. Recommending electrolytes is correct clinically, but the diagnostic pathway for Brugada (which may still require an EP study) cannot simply be suppressed on the basis of suspected hyperkalemia alone.

**Interaction 10 (VT prevents assessment): QTc is stated as "not measurable during VT" (line 808 context, and line 797).** This is correct — QT intervals cannot be measured reliably during VT. However, the document states this only in Section 5.5 interaction 10. The MR Agent's QTc measurement (Step 3, line 562) uses aggregate `qtc_bazett_ms` from SDA-1. SDA-1 computes this before the agents run. If VT is present, SDA-1 will have computed a QTc from VT beats that is meaningless. The MR Agent has no access to the RRC output (it runs in parallel), so it cannot know VT is present and cannot self-suppress the QTc. CDS's interaction 10 suppression addresses this, but the MR output will still contain a QTc value and possibly a STAT flag for QTc ≥500ms (from the wide, fast VT beats making the "QT" appear long). This creates a false STAT flag that CDS must actively downgrade — but the document does not include this as a known failure mode anywhere. It is a gap.

**Interaction 9 (Hyperkalemia mimics Complete AVB): The trigger condition requires `p_waves.present = false`.** In severe hyperkalemia, P-waves do not disappear suddenly — they gradually flatten. The `p_wave_present[]` per-beat boolean in SDA-1 may return `true` for flattened P-waves that are present but very small, missing the trigger. Conversely, for complete AVB, P-waves are present but dissociated from QRS. The trigger condition at line 794 should read: RRC `p_waves.present = false` OR (RRC `av_relationship.av_dissociation_present = true` AND MR `peaked_diffuse_present = true`). The current trigger will miss the hyperkalemia-mimics-CHB scenario when P-waves are small but technically detected.

**No interaction defined for atrial flutter with 2:1 block and LBBB.** This is a critical triple overlap: AFl produces regular rhythm at 2:1 (ventricular rate ~150 bpm), LBBB gives wide QRS, and the combination looks identical to VT at 150 bpm. The document has interactions for LBBB+STEMI and for VT+wide QRS but not for AFl+LBBB. The Brugada algorithm does not reliably distinguish VT from AFl+aberrancy, and the AV relationship in AFl at 2:1 may not show visible P-waves (flutter waves buried in QRS). This is the most common clinical diagnostic trap in wide complex tachycardia and is absent from the 12 interactions.

## 5. Reasoning Chain Critique

### 5.1 RRC Agent Reasoning Chain

**Step 2 (P-wave analysis) precedes Step 3 (rhythm classification) — correct in principle but one condition reverses this.** In AFib, there are no discrete P-waves to analyze. Running Step 2 on AFib will produce a `p_wave_present[] majority = false` and `distinct_morphology_count = 0` — which is correct. But for atrial flutter, Step 2 will detect multiple flutter wave "morphologies" and attempt to count them as ≥3 → MAT (Step 2d). Flutter waves at 250-350 bpm firing 3+ distinct shapes in different leads will confuse the P-wave morphology counting logic. The reasoning chain needs an explicit branch: "if Step 2d returns ≥3 morphologies AND rate 250-350 bpm AND regular — prioritize AFlutter (Step 3d) over MAT." The current chain does not have this branch.

**Step 4h (AV dissociation + tachycardia → VT) leads to `emit cross_domain_hint` but does not specify which hint.** Step 4h says "(emit cross_domain_hint)" in parentheses. The available hints are `WCT_AV_DISSOCIATION` and `WCT_NO_AV_DISSOCIATION`. Step 4h has already confirmed AV dissociation, so it should be `WCT_AV_DISSOCIATION`. But the reasoning chain does not say this — it defers to "Step 7c: emit cross_domain_hints per Section 2.5 rules," which means the emit actually happens in Step 7, not Step 4. This is a structural ambiguity: the agent may emit the hint mid-reasoning at Step 4h or wait until Step 7c. If the hint is emitted at Step 4h in the reasoning chain, it should still appear in the output JSON (produced at Step 7), so this is not a functional bug, but it creates confusion in the spec that could lead to duplicated hints in the output.

**Wide complex tachycardia concordance assessment appears in the RRC output (Step 6 and schema at line 289) but concordance requires examining precordial ST direction — an IT domain task.** The RRC Agent is told it is responsible for "QRS morphology classification" in WCT (lines 163, 229) but concordance in the Brugada algorithm is defined as "all precordial leads positive OR all negative deflection" — this is a QRS deflection measurement, not an ST measurement, so RRC Agent technically can assess it. However, the document never explicitly clarifies that concordance here refers to QRS direction (RRC domain) vs ST concordance (IT domain). This ambiguity will cause prompt interpretation problems. The concordance definition must be explicit in the RRC criteria reference section.

### 5.2 IT Agent Reasoning Chain

**Step 3 (ST depression survey) and Step 4 (de Winter pattern check) are in the wrong order.** De Winter pattern requires BOTH upsloping ST depression at the J-point AND peaked symmetric T-waves. Step 3 surveys ST depression. Step 4 checks T-wave morphology in the same leads. However, Step 3c says "upsloping ST depression V1-V6 at J-point → de Winter pattern if associated with peaked T-waves" — meaning Step 3 already partially diagnoses de Winter before Step 4 is run. Step 4 then re-checks the same condition. This redundancy is not wrong, but it creates conditions where the agent might flag de Winter at Step 3c without completing Step 4's verification of T-wave symmetry. The cleaner design is: Step 3 identifies upsloping STD as a candidate flag, and Step 4 completes the de Winter diagnosis. The Step 3c language should say "flag for de Winter assessment in Step 4" rather than "de Winter pattern if associated with peaked T-waves."

**Step 5 (Wellens pattern check): Step 5b requires "ABSENCE of ST elevation."** In Wellens Type A (biphasic T-wave), there is often a small amount of ST elevation before the T-wave inverts. The classic Wellens Type A presentation can have 1–2mm of ST elevation. Requiring complete absence of ST elevation will cause the IT Agent to miss Type A Wellens whenever there is any residual ST change. The criterion should be "no significant ST elevation (>2mm)" or "ST changes resolving from prior ischemia" rather than "ABSENCE of ST elevation."

**Step 7 (Q-wave assessment): The pathological Q-wave criterion at line 407 includes "depth ≥25% of R-wave amplitude."** If R-wave amplitude is very small (e.g., poor R-wave progression in an old anterior MI), a tiny Q-wave can meet the 25% criterion falsely. The standard clinical criterion also requires absolute depth ≥1mm in most leads. This relative-only criterion will over-diagnose pathological Q-waves in low-voltage situations. The criteria reference section for IT should include both absolute and relative thresholds.

### 5.3 MR Agent Reasoning Chain

**Step 5f (delta waves) duplicates RRC Step 6.** The MR Agent checks for delta waves at Step 5f (line 578) and the RRC Agent checks for WPW pattern (short PR + delta wave) with an explicit `delta_wave_present` field (line 279). Both agents emit findings about the same feature. There is no specification of which agent's delta wave finding takes precedence, and no hint type for MR to alert CDS that it independently detected delta waves that were not flagged by RRC. If RRC misses WPW (e.g., because PR is not short due to AV node disease) but MR detects the delta wave morphology, there is no structured cross-domain coordination path for this scenario.

**Step 2e (symmetric deep T-wave inversions in V3-V6 + tall precordial R-waves → apical HCM)** appears in the reasoning steps (line 559) but there is no corresponding output field for HCM in the MR schema. The schema has `qrs_morphology.r_wave_progression`, `t_wave_morphology.per_lead`, and a `findings[]` array. HCM would appear as a free-text `findings[]` entry with category `t_wave`. This is acceptable but the lack of a structured field means regression tests cannot assert HCM detection reliably — only that a finding with "HCM" in the description exists.

**Step 7 (U-waves) has no T-U fusion handling.** At elevated heart rates (>100 bpm), T-waves and U-waves fuse, making U-wave identification unreliable. The step does not instruct the agent to note this limitation when heart rate is elevated. A missed large U-wave fusion with T-wave (creating apparent T-wave prolongation) is a known cause of overestimation of QTc — the QTc measurement in Step 3 may be inflated by T-U fusion. The agent should be instructed: "if heart rate >85 bpm, U-wave assessment is unreliable and QTc may be overestimated due to T-U fusion."

### 5.4 CDS Agent Reasoning Chain

**Step 2 (process hints) precedes Step 3 (apply multi-criterion algorithms).** This is correct structurally. However, Step 3 depends on Step 2's output for its triggers (e.g., Sgarbossa is triggered by LBBB + ST findings). But Step 4 (resolve conflicts) follows Step 3. If Sgarbossa (Step 3) produces a finding that conflicts with IT's standalone assessment (which was already set in Step 2), the conflict resolution happens in Step 4 — after the algorithm that produced the conflict. This is the correct order (algorithm first, then integrate), but the chain does not explicitly say "update the accepted findings list after each multi-criterion algorithm before proceeding to conflict resolution." This creates ambiguity in whether Step 4 sees Step 3's results or only Step 2's results.

**Step 6 (differential diagnosis) has no guidance on what to include in the differential.** The instruction is "for each primary finding: list conditions with similar ECG presentation." This is an unbounded instruction. For AFib, the differential technically includes MAT, AFL with variable block, and multifocal PVC rhythm. CDS could generate an extremely long differential that is clinically unhelpful. A maximum differential depth (e.g., top 3 alternatives per primary finding) should be specified.

## 6. Prompt Engineering Critique

### 6.1 Temperature=0 Claims

The document asserts temperature=0 "prevents hallucination of criteria values" (Section 7.1) and that "LLMs can follow scoped instructions reliably at temperature=0" (line 88). Both claims are overstated.

Temperature=0 in DeepSeek R1 selects the greedy-decoded token at each step. It does not prevent the model from outputting incorrect criterion values — it makes outputs *deterministic*, not *correct*. A model that has a miscalibrated prior about Sgarbossa criterion thresholds will produce the same wrong threshold deterministically at temperature=0. The document conflates reproducibility with accuracy.

Furthermore, DeepSeek R1 is a reasoning model — it generates a chain-of-thought before the final output. At temperature=0, the CoT is deterministic, but the CoT itself may contain reasoning errors that propagate to the output. Embedding correct criteria values in the prompt is the actual mitigation (which the document does correctly, Section 7.1 point 2) — temperature=0 is about reproducibility, not error prevention.

### 6.2 System Prompt Length and Reasoning Quality

Section 7.1 point 5 states "short system prompts outperform long ones for reasoning quality" but this claim has no citation and contradicts the observation in many instruction-following benchmarks that longer, more specific prompts reduce out-of-distribution behavior for complex structured tasks. For a medical diagnostic task with embedded criteria, the prompt length concern is real (token budget) but the framing is misleading. The document should say: "minimize system prompt length within the constraint of embedding all criteria needed to prevent hallucination" rather than framing brevity as a quality principle.

### 6.3 Section 5 (Output Schema) "at the end" principle — DeepSeek R1 Specific

Section 7.1 point 3 says placing the output schema "at the END of the system prompt" produces better results with DeepSeek R1. This may be correct for instruction-following behavior, but for a reasoning model, the chain-of-thought runs before the output. The model must "know" the output schema before it reasons, so that it structures its reasoning toward the required output fields. Placing the schema at the end means the model sees the schema *after* its system-prompt instruction pass but *before* generating output — which is correct per transformer left-to-right attention. However, the claim that end-of-prompt placement outperforms mid-prompt placement is a model-specific empirical claim that should be validated (Task 1 should include a prompt placement ablation, not just latency measurement).

### 6.4 Prompt Language — Medical Safety Asymmetry Section

The recommended prompt language (Section 7.2, lines 1053–1054) is clinically appropriate in intent but contains a factual inaccuracy:

> "A false positive STEMI that leads to an unnecessary cath lab activation is costly but survivable."

Unnecessary cath lab activations carry real procedural risk (contrast nephropathy, access site complications, arrhythmia during catheterization). In high-risk patients (elderly, CKD, anticoagulated), unnecessary catheterization is not trivially "survivable." This prompt language will be presented to the model as clinical ground truth. The safer framing is: "A false positive STEMI results in unnecessary but generally reversible intervention. A missed STEMI results in preventable death or permanent myocardial damage." Remove the word "survivable" and the cost-minimizing framing.

### 6.5 JSON Enforcement — No `response_format` API Parameter

Section 7.3 discusses JSON output reliability strategies (example in prompt, "ONLY" instruction, orchestrator fallback). It does not mention whether the DeepSeek R1 API supports a `response_format: { type: "json_object" }` parameter (OpenAI-compatible structured output mode). If it does, this is the most reliable JSON enforcement mechanism and should be the primary strategy, with the in-prompt instructions as a secondary fallback. If it does not support structured outputs, this should be explicitly noted. The omission is significant because the entire reliability strategy currently depends on prompt engineering rather than API-level enforcement.

## 7. Validation Plan Critique

### 7.1 Unit Test Sufficiency

The 9 unit tests (3 per Phase 1 agent) are a minimal baseline. For a system being deployed in emergency medicine, the following test gaps are critical:

**No negative control tests.** All 9 unit tests are positive cases (known pathology). There is no test for a normal ECG that should produce `stat_flag = false` and `rhythm.classification = "sinus_normal"` with no findings. A regression failure where a normal ECG is classified as STEMI or VT is the most dangerous production failure mode and is not covered.

**No noise/artifact tests.** There are no unit tests for degraded signal quality (e.g., `signal_quality_per_lead < 0.5` for multiple leads). An agent receiving low-quality T-wave data should reduce its confidence, but this is untested.

**TC-IT-03 (pericarditis) expected output is incorrect.** The expected output says "no STAT flag" for pericarditis (line 1094). However, pericarditis with tamponade has hemodynamic instability and warrants urgent attention. More critically, the IT Agent's reasoning chain does not have a pericarditis-specific STAT flag rule — it does not know whether pericarditis is hemodynamically compromising from the ECG alone. The test assertion of "no STAT flag" is correct for uncomplicated pericarditis but may not be what the agent produces (the agent may produce a moderate-confidence STAT flag because it cannot exclude multi-vessel STEMI from a single ECG). The test should assert `stat_flag = false` with `differential_diagnoses` including pericarditis, and note that this is testing the agent's behavior, not clinical ground truth.

**TC-MR-01 (hyperkalemia) does not test the hint chain.** The test asserts `hint_type = "PEAKED_T_TERRITORIAL_CHECK"` but does not test the CDS response to that hint. Unit tests for Phase 1 agents are incomplete without a corresponding integration test that verifies CDS behaves correctly when the hint is received.

### 7.2 10-Scenario Pilot — Coverage Gaps

The pilot covers 10 scenarios, of which only 2 are described in detail (Scenarios 3 and 5). The 8 unspecified scenarios are referenced but not defined. Without knowing what all 10 scenarios are, the coverage cannot be assessed. The review assumes the scenarios include the 12 documented cross-domain interactions, but only Scenarios 3 and 5 can be verified from this document.

**Conspicuously absent scenarios:**
- Normal ECG (required negative control)
- Hyperkalemia progression (peaked T → sine wave)
- Long QT with PVCs (interaction 12 — TdP risk)
- Pericarditis vs multi-territory STEMI
- AFib with rapid ventricular response and LBBB (the AFl+LBBB VT mimic gap identified in Section 4)

### 7.3 Option D Parallel Pilot — Decision Rule Concerns

The decision rule at line 1128 states: "If Option D matches Option E's STAT detection rate AND per-finding accuracy within 5% on all 10 scenarios, revisit the architecture decision."

This is a dangerously low bar. 5% on 10 scenarios means a single scenario can differ. More critically, "STAT detection rate must be 100% both" is the stated primary criterion (line 1122) — but if both options achieve 100% STAT detection on 10 synthetic scenarios, the test does not distinguish them on real-world distribution. The pilot uses presumably idealized feature objects, not noisy real-world data. The comparison should occur on PTB-XL or a similar public dataset with known diagnoses, not just on the 10 synthetic scenarios.

### 7.4 Regression Test Protocol

The regression protocol (Section 8.5) triggers on "model update" only. It does not trigger on:
- Prompt changes (prompt engineering iterations will change behavior)
- Feature schema changes (SDA-1 pipeline updates)
- RAG content updates (adding or modifying diagnostic criteria chunks)

Any of these changes can cause regression without a model version change. The regression trigger list is incomplete.

## 8. SDA-1 Requirements Critique

### 8.1 T-Wave Symmetry Computation

Section 9.3 item 4 specifies: `symmetry = (Tpeak − Ton) / (Toff − Tpeak)` (line 1207).

This formula is problematic:

1. **The formula does not produce a 0-1 range.** For a perfectly symmetric T-wave, ascending duration = descending duration, so the ratio = 1.0. For a typical physiological T-wave (longer descending limb), the ratio would be <1.0 (e.g., 0.6–0.8). For a pathologically peaked narrow symmetric T-wave (hyperkalemia), the ratio approaches 1.0. For an upsloping T-wave (longer ascending), the ratio would be >1.0. The document claims "0=fully asymmetric, 1=perfectly symmetric" but the formula as written produces values in (0, ∞). To constrain to 0-1, the formula should be: `symmetry = min(Trise, Tfall) / max(Trise, Tfall)`, where `Trise = Tpeak − Ton` and `Tfall = Toff − Tpeak`. This gives 1.0 for perfect symmetry regardless of which limb is longer, and 0 approaches as the shorter limb → 0.

2. **Division by zero when Tpeak = Toff.** This can occur in inverted T-waves where ECGdeli may not reliably distinguish Tpeak from Toff, or in flat T-waves where the peak is ambiguous. No error handling for this degenerate case is specified.

3. **Asymmetric T-waves in LBBB are expected and normal.** The T-wave symmetry score in the presence of LBBB will consistently show near-1.0 symmetry (because LBBB T-waves are typically discordant with a specific symmetric shape). This will falsely elevate `t_symmetry` scores in LBBB beats, potentially triggering Wellens or hyperkalemia analysis inappropriately. The Feature Extraction node should set `t_symmetry = null` for beats with `beat_class = 1` (LBBB) and `beat_class = 2` (RBBB).

### 8.2 TWA Detection Threshold

Section 9.3 item 6 specifies: "threshold is >1.9μV beat-to-beat T-wave amplitude alternation" (line 1209).

The 1.9μV threshold originates from the Microvolt T-Wave Alternans (MTWA) literature, where it refers to spectral analysis of T-wave amplitude using sustained steady-state pacing or fixed-rate exercise. In the context of time-domain beat-to-beat comparison on a short 10-second ECG strip, the 1.9μV threshold is not validated. Standard 12-lead ECG systems (including GE MUSE and Philips TraceMaster) do not perform MTWA detection on short strips — MTWA requires at least 2–3 minutes of stable heart rate. On a 10-second strip with ~12 beats, the "spectral or time-domain TWA detection algorithm" (line 1209) would be computing beat-to-beat alternation in T-wave amplitude using only 6 odd-beat and 6 even-beat comparisons. The noise floor at this scale is likely to exceed 1.9μV in many normal ECGs with slight lead motion artifact.

The consequence is a high false-positive rate for `t_wave_alternans_present`, which triggers `CRITICAL_QTC_PROLONGATION` with TWA as the highest-urgency STAT flag. This could generate frequent false STAT alerts. Either the TWA detection should be explicitly flagged as unreliable on short strips (with a minimum recording duration check) or the threshold must be empirically validated on short 12-lead recordings before deployment. The document should acknowledge this limitation explicitly and set `t_wave_alternans_present = false` (or `null` with a quality flag) when `recording_duration_sec < 60`.

### 8.3 RS Interval Computation

Section 9.3 item 5 specifies: "`rs_interval_ms` (onset of R to peak of S) is required for Brugada Step 2 criterion." The Brugada algorithm Step 2 criterion is actually the RS interval defined as "any RS complex in precordial leads where the interval from the onset of R to the nadir of S exceeds 100ms" — this is the *duration* from R onset to S nadir. This is computed from QRSon to S fiducial (ECGdeli FPT column `S`), not from R peak to S peak. The specification says "onset of R to peak of S" which is neither the standard Brugada criterion definition nor a clean ECGdeli-computable value (QRSon is available, R peak is available, S nadir is available — but "onset of R" is ambiguous and may differ from QRSon in certain morphologies). The feature definition should state: `rs_interval_ms = S_fiducial − QRSon_fiducial`, matching the Brugada algorithm's actual criterion.

### 8.4 P-Wave Duration — Missing from SDA-1 Requirements

As noted in Section 1.2 of this review, `p_wave_duration_ms` per beat per lead is needed by the RRC reasoning chain (Step 2b: "notched/wide → LAE") and by the MR reasoning chain (Step 6c). Section 9.3 does not include P-wave duration computation as a requirement for SDA-1. ECGdeli FPT has Pon and Poff — this is a trivially computable field (`Poff − Pon`). Its absence from Section 9.3 means it will not be implemented in SDA-1 and the agents referencing it will either hallucinate values or fail to apply LAE criteria correctly.

## 9. Verdict and Rationale

**Verdict: GATE CONDITIONAL**

The document represents a substantially complete and clinically informed design. The core architecture decisions (full-feature routing, typed hint vocabulary, CDS raw feature selection) are defensible, and the reasoning chains for all four agents follow sound clinical logic. The cross-domain interaction catalog in Section 5.5 is comprehensive for the majority of life-threatening presentations.

However, the following issues are significant enough to require resolution before implementation begins:

**Critical issues (will cause clinical errors or system failures if not addressed):**
1. Token budget conflict: the full feature object likely exceeds the Phase 1 input budget by a factor of 1.5-2×. This must be measured empirically before implementation.
2. T-wave symmetry formula is mathematically incorrect as specified (produces values outside 0-1).
3. TWA detection on 10-second strips is methodologically unsound at the 1.9μV threshold — will generate high false STAT alert rate.
4. `p_wave_duration_ms` is referenced in reasoning chains but absent from both the feature schema and the SDA-1 requirements.
5. `q_waves.type = "new"` in the IT schema is uncomputable without a prior ECG — will be hallucinated.
6. Wellens dual-ownership conflict (both IT and MR classify Wellens) with no defined CDS resolution protocol.
7. Paced rhythm has no defined hint type and no suppression rule for ST analysis — a common real-world presentation with no pathway.

**Significant issues (will reduce reliability or maintainability):**
8. `V4R` referenced in CDS interaction 4 but not in the feature schema — no fallback defined.
9. CDS priority queue omits 6 of the 14 hint types (`COMPLETE_AVB_ISCHEMIC_CONTEXT`, `LVH_STRAIN_PRESENT`, `WPW_DETECTED`, and three others).
10. `LEFT_MAIN_PATTERN` boolean in IT schema conflicts with the cross_domain_hint architecture.
11. RS interval computation definition does not match the Brugada algorithm's actual criterion.
12. No version field in the feature object schema.
13. AFib + LBBB wide complex tachycardia is not a named cross-domain interaction and has no RAG trigger.
