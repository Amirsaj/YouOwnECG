## 1. Technology Stack

- **Framework:** Next.js 14 (App Router)
- **ECG rendering:** Canvas 2D API for signal traces (see Node 3.1 decision); D3.js for SVG annotations
- **State management:** Zustand (server-side data) + React Context (UI state)
- **Styling:** Tailwind CSS
- **Backend communication:** FastAPI via `fetch` with typed response schemas
- **Charts:** D3.js for trend charts; native Canvas for ECG trace

## 2. Next.js 14 App Router Structure

```
app/
  layout.tsx               ← Root layout (providers, global styles)
  page.tsx                 ← ECG upload / home
  ecg/
    [ecgId]/
      page.tsx             ← General dashboard (Server Component for initial data)
      layout.tsx           ← Shared layout for all ECG views
      disease/
        [condition]/
          page.tsx         ← Per-disease dashboard
      cardiologist/
        page.tsx           ← Cardiologist extension
  api/
    ecg/[ecgId]/route.ts   ← Proxies to FastAPI backend
```

**Server vs Client Components:**

| Component | Type | Reason |
|-----------|------|--------|
| Initial ECG data fetch | Server Component | No waterfalls; data arrives with HTML |
| ECG canvas renderer | Client Component | Canvas API requires browser |
| SVG annotation layer | Client Component | D3 requires DOM access |
| Patient summary panel | Server Component | Static data, no interactivity needed |
| Q&A interface | Client Component | Interactive, streaming response |
| STAT alert banner | Client Component | Animation, acknowledgment state |

## 3. D3.js + Canvas Integration

The ECG rendering engine is NOT pure D3. D3 handles scales (time → pixel, mV → pixel) and the annotation SVG layer only. The signal trace uses the Canvas 2D API directly.

```typescript
// Scale setup (D3)
const xScale = d3.scaleLinear()
    .domain([sampleStart, sampleEnd])
    .range([0, canvasWidth]);

const yScale = d3.scaleLinear()
    .domain([-mvRange, mvRange])
    .range([canvasHeight, 0]);

// Grid rendering (Canvas — faster than SVG for dense grids)
function renderGrid(ctx: CanvasRenderingContext2D) {
    const smallSquarePx = 6;
    ctx.strokeStyle = '#FFCCCC';
    ctx.lineWidth = 0.5;
    for (let x = 0; x < canvasWidth; x += smallSquarePx) {
        ctx.beginPath(); ctx.moveTo(x, 0); ctx.lineTo(x, canvasHeight); ctx.stroke();
    }
    // Large squares every 5 small squares
    ctx.strokeStyle = '#FF9999';
    ctx.lineWidth = 1.0;
    for (let x = 0; x < canvasWidth; x += smallSquarePx * 5) {
        ctx.beginPath(); ctx.moveTo(x, 0); ctx.lineTo(x, canvasHeight); ctx.stroke();
    }
    // Repeat for horizontal lines
}

// Signal trace (Canvas — min/max envelope per pixel column)
function renderTrace(ctx: CanvasRenderingContext2D, signal: Float32Array) {
    ctx.strokeStyle = '#000000';
    ctx.lineWidth = 1.5;
    // min/max envelope per pixel column (see Node 3.1 Section 2.2)
}
```

**D3 annotation SVG layer sits on top of canvas via CSS absolute positioning:**

## 4. State Management

**Zustand store — ECG data (server-fetched, immutable during session):**
```typescript
interface ECGStore {
    ecgId: string;
    featureObject: FeatureObject;      // from Node 1.5
    diagnosticResult: DiagnosticResult; // from Node 2.3
    visionResult: VisionVerificationResult; // from Node 1.6
    beatSummary: BeatSummary;          // from Node 2.5
}
```

**React Context — UI state (mutable, local):**
```typescript
interface ECGUIState {
    sampleStart: number;
    sampleEnd: number;
    mvRangePerStrip: number;
    visibleLayers: Set<AnnotationLayer>;
    selectedBeatIdx: number | null;
    selectedLead: string | null;
    activeConditionDashboard: string | null;
}
```

Separation rationale: ECG data is fetched once and never changes — Zustand with no re-render triggers. UI state changes frequently (every zoom/pan) — React Context with `useMemo` prevents unnecessary re-renders.

## 5. Performance Budget

| Operation | Target | Implementation |
|-----------|--------|---------------|
| Initial page load (FMP) | < 2 s | Server Components; lazy load cardiologist extension |
| 12-lead ECG render | < 200 ms | Canvas; min/max envelope; requestAnimationFrame |
| Zoom/pan re-render | < 100 ms | Only redraw visible canvas region; no DOM changes |
| Beat selection | < 50 ms | Pre-computed R-peak index array; O(log n) binary search |
| Q&A response (first token) | < 3 s | DeepSeek streaming; display tokens as they arrive |
| Per-disease dashboard | < 300 ms | Route-level code splitting; lazy loaded |

**ECG data transfer size:** 500 Hz × 10 s × 12 leads × 4 bytes (Float32) = 240 KB raw. After zlib compression (ECG signals compress ~5×): ~48 KB. Acceptable.

## 6. Responsive Design

**Primary breakpoint: desktop (≥ 1440 px)** — ER workstations
- Full 3×4+rhythm ECG grid visible without scrolling
- All panels in view

**Secondary breakpoint: tablet (768–1439 px)** — bedside devices
- ECG grid: 2×6 layout (2 columns instead of 4)
- Patient summary and findings panels collapse to accordion

**Mobile (< 768 px):** not a primary target but must remain functional
- Single-lead view only
- Rhythm strip + summary
- No per-disease dashboards (too information-dense)
