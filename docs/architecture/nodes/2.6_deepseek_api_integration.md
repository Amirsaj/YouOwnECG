## Executive Summary

Node 2.6 defines the Python layer that connects the orchestrator (Node 2.3) to the DeepSeek API. It specifies the exact API call structure, async execution model, error handling contract, prompt assembly format, safety guardrails, and reproducibility strategy.

Three design constraints from upstream nodes drive all decisions in this document:

1. **Node 2.3 interface contract**: the API layer must expose `async call_agent()` with `AgentTimeoutError` / `AgentAPIError(response_latency_s)` / `AgentSchemaError` error types, native async HTTP (no `asyncio.to_thread()`), and no automatic timeout retries.

2. **DeepSeek R1 reasoner mode**: the model produces a reasoning trace (thinking tokens) followed by the output. At `temperature=0` the output is deterministic. Reasoning tokens are not part of the output string but are billed separately. This matters for cost projection (VT-2.2-01, VT-2.1-03).

3. **Local clinical system, no cloud PHI**: patient data sent to the DeepSeek API must be limited to de-identified ECG signal features (UUID `ecg_id`, age integer, sex M/F/unknown). No name, MRN, DOB, or free-text clinical notes may ever be included in an API call. This is enforced at this layer, not by the caller.

What this document does NOT specify: the content of system prompts (those are implementation-phase work, guided by Node 2.2 Section 2.2/3.2/4.2/5.2 prompt architectures), or the feature extraction logic (SDA-1). Section 9 lists requirements placed on the implementation phase.

## 1. DeepSeek API: Reasoner Mode Specification

### 1.1 Model and Endpoint

```
Model:    deepseek-reasoner
Endpoint: https://api.deepseek.com/chat/completions
Method:   POST
Auth:     Bearer token — DEEPSEEK_API_KEY environment variable
```

The `deepseek-reasoner` model corresponds to DeepSeek-R1. It uses chain-of-thought reasoning: the model first produces a reasoning trace (the "thinking" block), then produces the final output. The API returns both as separate fields in the response.

### 1.2 Request Structure

```json
{
  "model": "deepseek-reasoner",
  "messages": [
    {
      "role": "system",
      "content": "<agent system prompt — see Section 3>"
    },
    {
      "role": "user",
      "content": "<feature object JSON + any RAG context — see Section 3>"
    }
  ],
  "temperature": 0,
  "max_tokens": 4096,
  "stream": false
}
```

**`temperature: 0`**: Required for reproducibility (Section 6). DeepSeek R1 at temperature=0 produces deterministic or near-deterministic outputs for the same input. This is the single most important safety measure for a clinical system — different runs on the same ECG should not produce different diagnoses.

**`max_tokens: 4096`**: This cap applies to **output tokens only** (the content after reasoning). Reasoning tokens are not counted against this limit. 4,096 output tokens is sufficient for all four agent schemas (Phase 1 outputs are ~1,000–3,000 tokens; CDS output is ~2,000–3,000 tokens). Setting this conservatively at 4,096 prevents runaway generation.

**`stream: false`**: The orchestrator needs the complete output before schema validation and STAT fast-path logic. Streaming would require partial-JSON parsing and is not necessary given the 15-second timeout headroom.

### 1.3 Response Structure

The DeepSeek API response for reasoner mode contains:

```json
{
  "id": "string",
  "model": "deepseek-reasoner",
  "choices": [
    {
      "message": {
        "role": "assistant",
        "content": "<final output — the JSON the orchestrator parses>",
        "reasoning_content": "<thinking trace — logged but not parsed>"
      },
      "finish_reason": "stop|length|content_filter"
    }
  ],
  "usage": {
    "prompt_tokens": "integer",
    "completion_tokens": "integer",
    "prompt_cache_hit_tokens": "integer",
    "prompt_cache_miss_tokens": "integer"
  }
}
```

**`content`**: The final agent output JSON. This is what the orchestrator parses and validates.

**`reasoning_content`**: The thinking trace. This is logged for audit purposes but not passed to the orchestrator or included in the clinical report. It is valuable for debugging reasoning failures.

**`finish_reason`**:
- `"stop"`: Normal completion — agent produced complete output.
- `"length"`: Output was truncated at `max_tokens`. This is a **critical failure** for a clinical agent — partial JSON is unparseable and the finding is incomplete. The API integration layer must detect `finish_reason = "length"` and raise `AgentSchemaError("finish_reason", "stop", "length")`.
- `"content_filter"`: DeepSeek flagged the content. For ECG feature data this should be impossible — log and raise `AgentAPIError`.

**`usage.completion_tokens`**: Output token count. Used for cost tracking.

**Reasoning tokens**: Not directly exposed in `usage` for all API versions. If the API version exposes `reasoning_tokens` separately, log it for VT-2.1-03 measurement. Otherwise, estimate as `total_tokens - prompt_tokens - completion_tokens`.

## 2. Async Client Architecture

### 2.1 Client Implementation

```python
import httpx
import asyncio
import time
import json
import os
from typing import Any

DEEPSEEK_API_URL = "https://api.deepseek.com/chat/completions"
DEEPSEEK_API_KEY = os.environ["DEEPSEEK_API_KEY"]  # never hardcoded, never logged

class AgentTimeoutError(Exception):
    pass

class AgentAPIError(Exception):
    def __init__(self, status_code: int, message: str, response_latency_s: float):
        self.status_code = status_code
        self.message = message
        self.response_latency_s = response_latency_s
        super().__init__(f"API {status_code} after {response_latency_s:.2f}s: {message}")

class AgentSchemaError(Exception):
    def __init__(self, field: str, expected: Any, got: Any):
        self.field = field
        self.expected = expected
        self.got = got
        super().__init__(f"Schema error: {field} expected {expected}, got {got}")
```

### 2.2 Core `call_agent()` Function

```python
async def call_agent(
    agent_id: str,
    system_prompt: str,
    user_content: str,
    timeout_s: float,
    client: httpx.AsyncClient,
) -> dict:
    """
    Makes a single DeepSeek R1 reasoner API call for one agent.

    Returns the parsed JSON dict from response.choices[0].message.content.
    Raises AgentTimeoutError, AgentAPIError, or AgentSchemaError on failure.
    Never retries — retry logic is the caller's (orchestrator's) responsibility.

    PRECONDITION: `client` must be constructed with timeout=httpx.Timeout(None).
    Any internal httpx timeout will race with asyncio.wait_for() and produce
    AgentAPIError instead of AgentTimeoutError — breaking the orchestrator's
    retry branching logic. This is enforced by assertion at entry.
    """
    assert client.timeout == httpx.Timeout(None), (
        f"call_agent() requires client.timeout=httpx.Timeout(None); "
        f"got {client.timeout}. httpx internal timeouts race with asyncio.wait_for()."
    )

    payload = {
        "model": "deepseek-reasoner",
        "messages": [
            {"role": "system", "content": system_prompt},
            {"role": "user", "content": user_content},
        ],
        "temperature": 0,
        "max_tokens": 4096,
        "stream": False,
    }

    headers = {
        "Authorization": f"Bearer {DEEPSEEK_API_KEY}",
        "Content-Type": "application/json",
    }

    call_start = time.monotonic()

    try:
        response = await asyncio.wait_for(
            client.post(DEEPSEEK_API_URL, json=payload, headers=headers),
            timeout=timeout_s,
        )
    except asyncio.TimeoutError:
        # asyncio.wait_for() cancels the underlying coroutine on timeout.
        # httpx.AsyncClient handles cancellation gracefully — the connection is
        # returned to the pool in a closed state. Subsequent calls on the same
        # client are not affected. (Verified: httpx >= 0.23 cleans up cancelled requests.)
        raise AgentTimeoutError(f"Agent {agent_id} timed out after {timeout_s}s")
    except httpx.RequestError as e:
        latency = time.monotonic() - call_start
        raise AgentAPIError(0, f"Network error: {e}", latency)

    latency = time.monotonic() - call_start

    if response.status_code != 200:
        raise AgentAPIError(response.status_code, response.text[:500], latency)

    body = response.json()
    choice = body["choices"][0]

    if choice["finish_reason"] == "length":
        raise AgentSchemaError("finish_reason", "stop", "length")

    if choice["finish_reason"] == "content_filter":
        raise AgentAPIError(200, "Content filter triggered", latency)

    content = choice["message"]["content"]
    reasoning = choice["message"].get("reasoning_content", "")

    # Log reasoning trace for audit (do not include in return value)
    _log_reasoning_trace(agent_id, reasoning, body.get("usage", {}))

    # Extract JSON from content — the agent may wrap it in markdown code fences
    parsed = _extract_json(content, agent_id)

    return parsed
```

### 2.3 JSON Extraction

Agent outputs may include markdown code fences (` ```json ... ``` `) or inline prose before the JSON. The extraction function handles this:

```python
import re

def _extract_json(content: str, agent_id: str) -> dict:
    """
    Extracts and parses JSON from agent output content.

    Handles three known DeepSeek R1 output patterns:
    1. Clean JSON (most common at temperature=0)
    2. Markdown-fenced JSON (```json ... ```)
    3. Reasoning bleed: prose before the JSON object (empirically observed with R1)

    Strategy 3 uses json.JSONDecoder().raw_decode() which parses exactly one
    JSON value at a given offset and returns (obj, end_pos). This is robust to:
    - Trailing text after the closing brace
    - Prose containing { characters before the JSON
    It does NOT use greedy regex, which is fragile for nested JSON.
    """
    decoder = json.JSONDecoder()

    # Strategy 1: direct parse
    try:
        return decoder.decode(content.strip())
    except json.JSONDecodeError:
        pass

    # Strategy 2: markdown code fences
    match = re.search(r"```(?:json)?\s*([\s\S]+?)\s*```", content)
    if match:
        try:
            return decoder.decode(match.group(1))
        except json.JSONDecodeError:
            pass

    # Strategy 3: find the first valid JSON object by scanning for '{' positions
    # raw_decode(s, idx) parses one JSON value starting at idx; raises on failure
    # This handles reasoning bleed: prose may contain '{' characters, but only
    # the correct offset will decode a complete valid JSON object
    for i, char in enumerate(content):
        if char == "{":
            try:
                obj, _ = decoder.raw_decode(content, i)
                if isinstance(obj, dict):
                    return obj
            except json.JSONDecodeError:
                continue

    raise AgentSchemaError("content", "valid JSON object", f"unparseable: {content[:200]}")
```

### 2.4 Phase 1 Dispatch with STAT Fast-Path

Phase 1 dispatch uses `asyncio.as_completed()` semantics so that each agent's output is processed as it arrives — not in a fixed order. This is the STAT fast-path requirement: if `it` finishes first with a STEMI flag, the orchestrator sees it before `rrc` completes.

The canonical dispatch pattern is an **async generator** that yields `(agent_id, result_or_exception)` tuples in completion order:

```python
from typing import AsyncGenerator

async def iter_phase1_results(
    feature_content: str,
    prompts: dict,  # {"rrc": str, "it": str, "mr": str}
    client: httpx.AsyncClient,  # persistent client from application context
    timeout_s: float = 15.0,
) -> AsyncGenerator[tuple[str, dict | Exception], None]:
    """
    Dispatches RRC, IT, MR agents concurrently.
    Yields (agent_id, output_or_exception) tuples in COMPLETION ORDER — not insertion order.
    The orchestrator calls check_stat_fast_path() on each yielded output immediately.

    Precondition: client must have timeout=httpx.Timeout(None) (enforced by call_agent).
    """
    agent_ids = ("rrc", "it", "mr")
    task_to_agent = {
        asyncio.create_task(
            call_agent(agent_id, prompts[agent_id], feature_content, timeout_s, client)
        ): agent_id
        for agent_id in agent_ids
    }

    for completed_task in asyncio.as_completed(task_to_agent.keys()):
        agent_id = task_to_agent[completed_task]
        try:
            result = await completed_task
            yield agent_id, result
        except (AgentTimeoutError, AgentAPIError, AgentSchemaError) as e:
            yield agent_id, e
```

**Orchestrator usage pattern:**

```python
results = {}
async for agent_id, result in iter_phase1_results(feature_content, prompts, client):
    results[agent_id] = result
    if not isinstance(result, Exception):
        if check_stat_fast_path(result):
            fire_stat_alert(agent_id, result)  # fires immediately on first STAT output
```

**Why a generator, not a helper that returns a dict:** A function returning a complete dict cannot emit STAT alerts mid-execution. The generator pattern gives the orchestrator control at each completion event without the API layer making clinical decisions.

**Client lifecycle:** The `client` parameter is an application-scoped `httpx.AsyncClient` created at startup (see Section 9), NOT created per-call. Per-call client creation adds 100–300ms TLS overhead per ECG and risks burst connection patterns that trigger rate limiting.

## 3. Prompt Assembly

### 3.1 Message Structure Per Agent

Each API call uses a two-message structure:

**System message** = the agent's static role definition (Node 2.2 Sections 2.2, 3.2, 4.2, 5.2):
- Defines the agent's mission, reasoning protocol, criteria reference, hint emission rules, output schema
- Static — does not change between ECG calls
- Loaded from file at startup, not assembled per-call
- **Never includes patient data**

**User message** = the dynamic per-ECG content:
- The feature object JSON (Section 3.2)
- Optional RAG context for CDS (Section 3.3)
- No free-text clinical narrative
- No patient identifiers beyond `ecg_id` (UUID) and `patient_sex`/`patient_age_years` (de-identified)

### 3.2 Feature Object Serialization

The feature object is serialized to compact JSON (no indentation) to minimize token count:

```python
import json

class _RoundedFloatEncoder(json.JSONEncoder):
    """
    Custom JSON encoder that rounds floats to 4 decimal places.

    json.dumps() calls default() only for non-natively-serializable types.
    Python float IS natively serializable, so a default= handler is never
    invoked for floats. A JSONEncoder subclass that overrides iterencode
    is the correct approach.

    4 decimal places preserves all clinically significant precision
    (ECG amplitudes to 0.1mV, intervals to 1ms) and reduces token count
    by ~15-20% for float-dense beat arrays.
    """
    def iterencode(self, obj, _one_shot=False):
        if isinstance(obj, float):
            yield f"{obj:.4f}"
        elif isinstance(obj, dict):
            yield "{"
            first = True
            for key, value in obj.items():
                if not first:
                    yield ","
                yield from self.iterencode(key)
                yield ":"
                yield from self.iterencode(value)
                first = False
            yield "}"
        elif isinstance(obj, (list, tuple)):
            yield "["
            first = True
            for item in obj:
                if not first:
                    yield ","
                yield from self.iterencode(item)
                first = False
            yield "]"
        elif obj is None:
            yield "null"
        elif isinstance(obj, bool):
            yield "true" if obj else "false"
        elif isinstance(obj, int):
            yield str(obj)
        elif isinstance(obj, str):
            yield json.dumps(obj)
        else:
            raise TypeError(f"Not JSON serializable: {type(obj)}")


def serialize_feature_object(feature_obj: dict) -> str:
    """
    Serializes the feature object to compact JSON for API transmission.
    Rounds floats to 4 decimal places to reduce token count (~15-20% savings
    on float-dense beat arrays).
    """
    return "".join(_RoundedFloatEncoder(separators=(",", ":")).iterencode(feature_obj))
```

**Float precision**: 4 decimal places is sufficient for all clinical measurements (ECG amplitudes to 0.1mV, intervals to 1ms — both within 4-decimal float precision). Rounding reduces token count by ~15–20% for float-heavy arrays.

### 3.3 RAG Context Injection (CDS Only)

CDS receives RAG-retrieved textbook context per Node 2.2 Section 5.4. The RAG content is injected as a separate section in the user message, clearly delimited:

```
<features>
{feature_object_json}
</features>

<phase1_outputs>
{phase1_outputs_json}
</phase1_outputs>

<cross_domain_hints>
{hints_json}
</cross_domain_hints>

<rag_context>
{rag_chunks}
</rag_context>
```

**Delimiter rationale**: XML-style tags are clearer than prose headers for DeepSeek R1's reasoning parser and reduce ambiguity about which section a given piece of text belongs to.

**RAG budget enforcement**: The RAG context is pre-trimmed to ≤2,500 tokens before injection (Node 2.2 Section 5.4). The API layer receives the pre-trimmed string — it does not perform RAG retrieval.

**Phase 1 agents**: do NOT receive RAG context in the user message. Criteria are embedded in the system prompt (Node 2.2 Section 2.2 criteria reference sections). This reduces Phase 1 token cost and latency.

## 4. Safety Guardrails

### 4.1 PHI Stripping (Pre-Call Validation)

Before any API call, the feature object is validated for PHI:

```python
# Allowlist: the only patient-linked fields permitted in API calls.
# All other patient_*, subject_*, visit_*, encounter_*, mrn*, dob*, ssn* fields are blocked.
ALLOWED_PATIENT_FIELDS = {"patient_sex", "patient_age_years"}

# Prefixes that identify quasi-identifier fields — any field matching these prefixes
# and NOT in ALLOWED_PATIENT_FIELDS is blocked.
_PHI_PREFIXES = ("patient_", "subject_", "visit_", "encounter_", "mrn", "dob", "ssn", "name")

def validate_no_phi(feature_obj: dict) -> None:
    """
    Allowlist-based PHI check. Rejects any field that matches a PHI-prefix pattern
    and is NOT in ALLOWED_PATIENT_FIELDS. A denylist of known bad field names is
    insufficient — any new PHI field added by SDA-1 would pass silently.

    This is a defense-in-depth check at the API boundary.
    """
    blocked = []
    for key in feature_obj.keys():
        key_lower = key.lower()
        if any(key_lower.startswith(prefix) for prefix in _PHI_PREFIXES):
            if key not in ALLOWED_PATIENT_FIELDS:
                blocked.append(key)
    if blocked:
        raise ValueError(
            f"Unrecognized patient-linked fields detected before API call: {blocked}. "
            f"Only {ALLOWED_PATIENT_FIELDS} are permitted."
        )
```

**Key guarantee**: `ecg_id` is a UUID — not a patient identifier. `patient_sex` and `patient_age_years` are de-identified per HIPAA Safe Harbor (age <90, no other quasi-identifiers). These are the only patient-linked fields permitted in API calls.

**Logging rule**: API call logs must NEVER log the `user_content` payload in full — only `agent_id`, `ecg_id`, token counts, and latency. Logging the full payload would expose de-identified ECG features to log aggregators. The reasoning trace is logged locally only (not to any external log aggregator).

### 4.2 Output Grounding Check

After parsing the agent output, a lightweight grounding check validates that the agent's findings are internally consistent with the feature object it received. This is not a clinical validation — it is a hallucination detection heuristic:

**Check 1 — STAT rate plausibility**: If a Phase 1 agent flags `stat_flag = true` with `stat_reason` referencing a heart rate (e.g., "VT at 180 bpm"), the `heart_rate_ventricular_bpm` in the feature object should be in a plausible range (>100 for tachycardia). If the feature object shows 65 bpm and the agent flags VT, this is a hallucination indicator.

**Check 2 — No non-existent leads**: If a finding references lead "V8" or "aVX" — leads not in the standard 12-lead schema — this is a hallucination. Log and flag the finding with `hallucination_suspect = true`.

**Check 3 — Confidence-finding consistency**: If `confidence_overall < 0.3` but `stat_flag = true` appears in a finding, flag for review — high uncertainty with STAT flag is unusual and may indicate unreliable output.

Grounding check failures do NOT suppress the finding — they add a `hallucination_suspect: boolean` field to the affected finding. The orchestrator logs these for monitoring. CDS is notified via the `phase1_agent_status` metadata.

### 4.3 API Key Management

```python
import os

def get_api_key() -> str:
```

The key is loaded once at startup and stored in memory. It is never written to logs, never included in error messages, and never passed through the feature object.

### 4.4 Rate Limiting and Backpressure

DeepSeek API rate limits (as of 2025): not publicly documented with precision, but empirically 50–200 RPM (requests per minute) at standard tier. For a 3-agent parallel call per ECG, each ECG analysis consumes 3 API calls (+ 1 for CDS = 4 total). At 50 RPM, this supports ~12 concurrent ECG analyses per minute — sufficient for a single-site ER.

**Rate limit handling**: If the API returns HTTP 429 (Too Many Requests):
- The orchestrator treats this as an `AgentAPIError` with `status_code = 429`
- The caller (above the orchestrator) is responsible for backpressure — the orchestrator does NOT retry 429s internally
- A 429 on a Phase 1 agent triggers degraded mode (same as any other API failure)

**Exponential backoff at the caller level** is recommended for pipeline retries: first retry after 2s, second after 4s, third after 8s, then fail.

## 5. Token Counting and Cost Estimation

### 5.1 Pre-Call Token Estimation

The API layer provides a token estimation function for pre-flight budget checks (used by the orchestrator for raw feature budget enforcement):

```python
def estimate_tokens(text: str) -> int:
    """
    Fast approximation: 1 token ≈ 4 characters for English/JSON text.
    DeepSeek uses a BPE tokenizer similar to GPT-4. For JSON with numeric values,
    this approximation is accurate to ±15%.
    For precise counts, use tiktoken with cl100k_base encoding as a proxy.
    """
    return max(1, len(text) // 4)
```

For VT-2.2-01 and VT-2.3-01 (empirical token measurement), use the actual DeepSeek tokenizer if available, or tiktoken `cl100k_base` as the best available proxy:

```python
import tiktoken

def count_tokens_precise(text: str) -> int:
    enc = tiktoken.get_encoding("cl100k_base")
    return len(enc.encode(text))
```

### 5.2 Cost Model

| Token type | DeepSeek R1 pricing (as of 2025-03) | Notes |
|-----------|--------------------------------------|-------|
| Input (prompt) tokens | $0.55 / 1M | Cached tokens: $0.14 / 1M |
| Output (completion) tokens | $2.19 / 1M | |
| Reasoning tokens | $0.55 / 1M | Same as input; billed per token |

**Per-ECG cost estimate — TYPICAL case (simple/moderate ECG: normal sinus, isolated rate abnormality, stable rhythm):**

| Call | Prompt tokens | Reasoning tokens | Output tokens | Cost |
|------|--------------|-----------------|---------------|------|
| RRC Agent | ~7,000 | ~3,000 | ~1,500 | ~$0.009 |
| IT Agent | ~7,500 | ~4,000 | ~2,000 | ~$0.011 |
| MR Agent | ~7,000 | ~3,000 | ~1,500 | ~$0.009 |
| CDS Agent | ~12,000 | ~5,000 | ~2,500 | ~$0.017 |
| **Total (typical)** | **~33,500** | **~15,000** | **~7,500** | **~$0.046** |

**Per-ECG cost estimate — COMPLEX case (STEMI with atypical presentation, polymorphic VT, Brugada pattern, LBBB masking ischemia):**

DeepSeek R1 produces reasoning tokens proportional to diagnostic complexity. For complex cases, empirical R1 behavior produces 10,000–40,000 reasoning tokens per agent call.

| Call | Prompt tokens | Reasoning tokens | Output tokens | Cost |
|------|--------------|-----------------|---------------|------|
| RRC Agent | ~7,000 | ~15,000 | ~2,500 | ~$0.017 |
| IT Agent | ~7,500 | ~20,000 | ~3,000 | ~$0.022 |
| MR Agent | ~7,000 | ~15,000 | ~2,500 | ~$0.017 |
| CDS Agent | ~12,000 | ~25,000 | ~3,000 | ~$0.030 |
| **Total (complex)** | **~33,500** | **~75,000** | **~11,000** | **~$0.086** |

**Upper bound estimate (extreme case — 40,000 reasoning tokens per agent):** ~$0.15/ECG.

**Operational cost basis:** Use the range $0.046–$0.15 per ECG. The $0.046 figure is not a safe planning number for capacity or budget — complex STAT cases (the most important ones) routinely cost 2–3× the typical case. VT-2.1-03 must measure against complex ECGs, not only simple ones.

**Note:** Reasoning token billing (VT-2.1-03) must be verified empirically. If reasoning tokens are billed as output tokens (at $2.19/1M rather than $0.55/1M), the complex-case cost rises to ~$0.25/ECG. If reasoning tokens are not billed at all, the typical case drops to ~$0.027.

**Cache benefit**: The system prompt is static per agent. DeepSeek API supports prompt caching for repeated prefixes. If the same system prompt is used across all calls (which it is — static per agent), the prompt prefix after the first call will be cached. At scale, this reduces per-ECG cost by ~30-40%.

### 5.3 Cost Monitoring

The API layer logs `usage` from every response:

```python
# Circuit breaker: alert if any single ECG run exceeds this threshold.
# Complex STAT cases legitimately reach ~$0.10–$0.15; above $0.20 suggests
# reasoning runaway or prompt inflation and warrants investigation.
MAX_COST_PER_ECG_USD = 0.20

# Pricing (as of 2025-03) — update if DeepSeek pricing changes
_PRICE_INPUT_PER_TOKEN = 0.55 / 1_000_000
_PRICE_OUTPUT_PER_TOKEN = 2.19 / 1_000_000
_PRICE_REASONING_PER_TOKEN = 0.55 / 1_000_000  # pending VT-2.1-03 verification
_PRICE_CACHE_HIT_PER_TOKEN = 0.14 / 1_000_000


def _estimate_call_cost(usage: dict) -> float:
    """Estimates USD cost for a single agent call from the API usage dict."""
    prompt = usage.get("prompt_tokens", 0)
    completion = usage.get("completion_tokens", 0)
    cache_hit = usage.get("prompt_cache_hit_tokens", 0)
    cache_miss = usage.get("prompt_cache_miss_tokens", prompt)
    reasoning = usage.get("reasoning_tokens", 0)

    # If cache breakdown is available, use it; otherwise bill all prompt tokens at full rate
    if cache_hit + cache_miss > 0:
        input_cost = (cache_miss * _PRICE_INPUT_PER_TOKEN) + (cache_hit * _PRICE_CACHE_HIT_PER_TOKEN)
    else:
        input_cost = prompt * _PRICE_INPUT_PER_TOKEN

    output_cost = completion * _PRICE_OUTPUT_PER_TOKEN
    reasoning_cost = reasoning * _PRICE_REASONING_PER_TOKEN
    return input_cost + output_cost + reasoning_cost
```

## 6. Reproducibility Strategy

### 6.1 Determinism at temperature=0

DeepSeek R1 at `temperature=0` is documented to produce deterministic outputs for identical inputs. In practice, floating-point non-determinism in distributed inference may cause rare output variations (<1% of calls). For a clinical system, this is acceptable — the system is not a medical device and the non-determinism risk is acknowledged in the clinical governance documentation.

**What temperature=0 guarantees:** Given the same system prompt + user content + model version, the output is deterministically the same token sequence in the vast majority of cases. Two analysts running the same ECG will get the same report.

**What it does not guarantee:** Different model versions or API-side load balancing to different hardware may produce different outputs even at temperature=0. Model version pinning (Section 6.2) addresses this.

### 6.2 Model Version Pinning

The `model` field in API requests should specify the exact model version when DeepSeek exposes version suffixes (e.g., `deepseek-reasoner-20250101`). If only `deepseek-reasoner` is available, the model version is logged from the response `model` field and alerted if it changes.

**Persistence requirement:** The expected model version must be stored in the local database and loaded at application startup. An in-memory module global (`_current_model_version: str | None = None`) resets on every application restart — the exact scenario this check is designed to catch (a model update between two shifts) will always be missed.

```python
def load_model_version_baseline(db) -> str | None:
    """
    Loads the expected model version from the database on session start.
    Returns None if no version has been recorded yet (first ever run).

    Table: api_config
    Columns: key TEXT PRIMARY KEY, value TEXT, updated_at TIMESTAMP
    Row: key = 'deepseek_model_version_baseline'
    """
    pass  # implementation-phase detail


def _check_and_persist_model_version(response_model: str, db) -> None:
    """
    Called after each API response. Compares the returned model string
    against the persisted baseline.

    - If no baseline exists: stores response_model as the baseline and logs
      "model version baseline established: {response_model}".
    - If baseline matches: no action.
    - If baseline differs: logs an alert "model version changed:
      expected {baseline}, got {response_model}" and updates the stored
      baseline so subsequent calls in this session do not re-alert.

    The in-memory module global is NOT used as the primary tracking mechanism.
    The database is authoritative. This correctly detects changes between
    application restarts (e.g., overnight API updates between ER shifts).

    Async safety: database writes in CPython with an async-safe DB driver
    (aiosqlite) are safe under concurrent Phase 1 calls — the first write
    wins and subsequent reads in the same session hit the updated value.
    No mutex is required because the only cross-session concern is the
    baseline load at startup (single-threaded initialization path).
    """
    pass  # implementation-phase detail
```

**Startup sequence:**
1. Call `load_model_version_baseline(db)` once at application startup (before accepting requests).
2. Store the result in an application-scoped variable (not a module global).
3. Pass the loaded baseline to `_check_and_persist_model_version()` on each response.

**Why not a module global at all:** Module globals in async Python are not inherently unsafe (CPython GIL protects simple assignment), but a global that is `None` at startup is not a baseline — it is the absence of one. Storing `None` at startup and then setting it on the first response is indistinguishable from the "no stored version" state, making cross-restart detection impossible.

### 6.4 Audit Record Persistence Specification

**Database and table:**

```sql
-- Local SQLite database: youownecg.db (application data directory)
-- Table: agent_audit_log
CREATE TABLE agent_audit_log (
    audit_id          TEXT PRIMARY KEY,          -- UUID v4
    ecg_id            TEXT NOT NULL,             -- UUID — links to ecg_records table
    pipeline_run_id   TEXT NOT NULL,             -- UUID — groups all 4 agent calls per ECG
    agent_id          TEXT NOT NULL,             -- 'rrc' | 'it' | 'mr' | 'cds'
    model_version     TEXT NOT NULL,             -- from response body "model" field
    prompt_tokens     INTEGER NOT NULL,
    completion_tokens INTEGER NOT NULL,
    reasoning_tokens  INTEGER,                   -- NULL if not exposed by API version
    cache_hit_tokens  INTEGER,
    estimated_cost_usd REAL,                     -- from _estimate_call_cost()
    latency_s         REAL NOT NULL,
    finish_reason     TEXT NOT NULL,             -- 'stop' | 'length' | 'content_filter'
    hallucination_flags TEXT,                    -- JSON array of flag strings; NULL if none
    timestamp_utc     TEXT NOT NULL              -- ISO 8601, UTC
);
CREATE INDEX idx_agent_audit_ecg ON agent_audit_log (ecg_id);
CREATE INDEX idx_agent_audit_time ON agent_audit_log (timestamp_utc);
```

**Write failure behavior: fail-open.**

If the audit write fails (database locked, disk full, schema mismatch), the pipeline continues and the clinical finding is surfaced to the clinician. The write failure is logged to the application error log. Rationale: suppressing a STEMI finding to preserve audit trail integrity is not an acceptable trade-off. Audit completeness is a quality requirement; patient safety is inviolable. Persistent write failures (>3 consecutive failures) trigger an operational alert: "Audit writes failing — investigate database."

**Retention period:**

Audit records must be retained for a minimum of 7 years from the ECG analysis date. This is consistent with US HIPAA minimum retention (6 years from creation) and typical hospital medical records retention (7–10 years). The application must not purge `agent_audit_log` records except under an explicit data retention policy approved by the deploying institution.

**Access control:**

`agent_audit_log` rows contain `ecg_id` which links to patient records — this is effectively part of the medical record. Read access is restricted to: (a) the application service account (write only during pipeline), (b) authorized clinical staff via the application's audit query interface, (c) system administrators with documented access logs. No direct database access is granted to front-end application users.

**Write mode: synchronous, within pipeline execution.**

The audit write is a synchronous `INSERT` that completes before `call_agent()` returns. An async-deferred write could silently fail without blocking the pipeline and create a false impression of an audit trail. The added latency for a local SQLite insert is <2ms and is within the pipeline's latency budget.

## 7. Failure Mode Catalog

### 7.1 API Unavailable (Connection Error / DNS Failure)

**Trigger:** `httpx.RequestError` — network is unreachable or DNS resolution fails.

**Response:** Raise `AgentAPIError(0, "Network error: ...", latency)`. The orchestrator treats this as any other API failure. The error code 0 indicates a network-level failure (no HTTP status received).

**Recovery:** The system should queue the ECG for retry with exponential backoff at the caller level. If the API is unavailable for >5 minutes, alert operations staff — this is an infrastructure problem, not a per-ECG error.

### 7.2 Rate Limit (429)

**Response:** Raise `AgentAPIError(429, response_body, latency)`. The orchestrator treats this as a fast API error (<2s latency typical for 429) — eligible for one immediate retry per Section 4.2. If the retry also returns 429, the caller must implement backoff.

### 7.3 Output Truncation (`finish_reason = "length"`)

**Trigger:** Agent output exceeds `max_tokens = 4096`.

**Response:** Raise `AgentSchemaError("finish_reason", "stop", "length")`. The orchestrator treats this as `SCHEMA_INVALID`.

**Prevention:** If this error occurs in testing, increase `max_tokens` or add output compression instructions to the agent system prompt (e.g., "Omit evidence details for findings with confidence < 0.3").

### 7.4 Content Filter (200 + `finish_reason = "content_filter"`)

**Trigger:** DeepSeek flags the ECG feature content as violating usage policy.

**Expected frequency:** Extremely rare — ECG numerical data should never trigger content filters.

**Response:** Raise `AgentAPIError(200, "Content filter", latency)`. Log the full request payload (minus PHI) for investigation. This is likely a DeepSeek API bug or policy change.

### 7.5 Invalid JSON in Output

**Trigger:** Agent produces non-JSON output (prose, partial JSON, or corrupted JSON).

**Response:** `_extract_json()` raises `AgentSchemaError("content", "valid JSON object", ...)`. The orchestrator treats this as `SCHEMA_INVALID`.

**Root cause:** Almost always a prompt engineering issue — the agent was not given a strong enough output format instruction, or the schema example in the prompt had a syntax error. The `reasoning_content` log is valuable here — the reasoning trace will show whether the agent understood the task and produced valid JSON before it was corrupted.

### 9.4 Persistent Client Lifecycle (Connection Pooling)

`httpx.AsyncClient` must be created once at application startup and injected into all `call_agent()` calls via `iter_phase1_results()` and the CDS call site. It must NOT be created per-ECG.

Per-ECG client creation causes a TLS handshake to `api.deepseek.com` on every ECG analysis — approximately 100–300ms additional latency per ECG. For concurrent ECG analyses during a busy ER shift, this creates a burst of simultaneous TLS connections that may trigger rate-limiting at the connection level before the per-request 429 threshold.

**Startup:**
```python
# Application startup (FastAPI lifespan or equivalent)
async def startup():
    app.state.deepseek_client = httpx.AsyncClient(
        timeout=httpx.Timeout(None),
        limits=httpx.Limits(max_connections=20, max_keepalive_connections=10),
    )
```

**Shutdown (graceful):**
```python
async def shutdown():
    await app.state.deepseek_client.aclose()
```

The `timeout=httpx.Timeout(None)` is set here in one place — enforcing the precondition for all `call_agent()` calls without relying on the assertion at each call site.

### 9.5 Local Replay Proxy (Integration Testing)

The project prohibits mocking external data sources. For integration testing of the `call_agent()` code path without incurring real API costs, a local replay proxy captures and replays real API responses.

**Interface:** The replay proxy exposes the same endpoint as the DeepSeek API (`POST /chat/completions`) and is activated by setting `DEEPSEEK_API_URL=http://localhost:8765`. The proxy is indistinguishable from the real API from `call_agent()`'s perspective — no mock logic in production code.

**Response fixture format:** Each fixture is a real DeepSeek API response captured during a VT run, stored as a JSON file:

```
tests/fixtures/replay/
    <ecg_id>_<agent_id>.json    # full API response body
```

Example: `tests/fixtures/replay/ptbxl_00001_rrc.json` — the exact response returned by the DeepSeek API for PTB-XL record 00001, RRC agent.

**Fixture capture:** Run the real API call with `CAPTURE_FIXTURES=true` set. The API layer writes the full response body to the fixture path. Fixtures are committed to the test suite. They represent real model outputs on real ECG data — not fabricated responses.

**Lookup logic:** The replay server matches requests by hashing the `messages[1].content` field (the user content, which contains the ECG feature object). If a matching fixture exists, it returns it. If not, the server returns HTTP 404 — the test fails with a clear error rather than silently substituting a wrong fixture.

**Initial fixture population:** VT-2.6-01 and VT-2.6-02 are the first fixture capture events. After those verification tasks complete, fixtures for the 10 PTB-XL regression ECGs (Section 9.2) must also be captured and committed.
