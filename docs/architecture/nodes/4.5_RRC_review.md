## Issues Found

### REQUIRED-1: check_claim_supported() 0.25 keyword overlap threshold is too low for clinical false-positive suppression

**Location:** Node 4.5, Section 2.1

The 25% keyword overlap check is designed as a lightweight proxy for semantic entailment. However, clinical terminology has high specificity — the word "elevation" appears in both ST elevation (STEMI) and T-wave elevation (hyperacute T), which are related but distinct concepts. A claim about "hyperacute T-wave elevation" citing a chunk about "ST elevation criteria" could pass the 25% overlap check incorrectly because both share "elevation" and lead names.

The result: the safety layer gives a false "supported" verdict for a claim citing the wrong evidence. The problem is that the check is too coarse to catch this failure mode.

**Required fix:** The threshold alone is insufficient. Add a finding-type exclusivity check: if the claim's `supporting_chunk_ids` are from a chunk tagged with a different `finding_type` than the claim's finding, flag it regardless of keyword overlap:

```python
def check_claim_supported(claim: str, claim_finding_type: str, chunks: list[dict]) -> bool:
    if not chunks:
        return False
    # Check for finding-type mismatch
    for chunk in chunks:
        chunk_finding_hint = chunk.get("section", "").lower()
        if claim_finding_type not in chunk_finding_hint and claim_finding_type != "general":
            # Log a soft warning but don't hard-fail (sections don't always match)
            pass
    # keyword overlap (existing logic)
    ...
```

Additionally, raise the keyword overlap threshold from 0.25 to 0.35 to reduce false positives.

### REQUIRED-2: OUT_OF_SCOPE_PATTERNS truncates at the flagged sentence — but truncation point is not specified

**Location:** Node 4.5, Section 3

The spec states out-of-scope outputs are "truncated at the flagged sentence." However, sentence boundary detection in clinical text is non-trivial. "Dr. Smith's findings (see Fig. 3.2) suggest a dosage of 80 mg" — the regex `dosage` fires, but the sentence boundary is ambiguous due to "Dr." and "Fig." abbreviations.

A naive truncation at `. ` would truncate at "Dr.", producing a broken output. A proper sentence boundary detector is needed.

**Required fix:** Use a simple heuristic for truncation: instead of truncating mid-sentence, replace the ENTIRE sentence containing the flagged pattern with the out-of-scope notice. "Sentence" is defined as the substring between the previous `\n` or `. ` (after a capital letter) and the next `\n` or `. `.

For MVP, the simpler approach: flag with a visible warning but do not truncate. The UI displays: "[WARNING: This response may contain content outside ECG interpretation scope. Clinical review required.]" appended to the output. Truncation is deferred to a post-MVP release with proper sentence segmentation.
