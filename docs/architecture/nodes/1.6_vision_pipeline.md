## Executive Summary

Node 1.6 adds a secondary visual interpretation layer on top of the signal-based pipeline. The DeepSeek-VL2 model receives a rendered ECG image and returns a structured JSON of visual findings that supplements — but never replaces — the signal-computed features from Node 1.5.

Two design constraints govern everything in this node:

1. **Vision is advisory, not authoritative.** When VL2 and the signal pipeline disagree, the disagreement is flagged for the downstream agents to resolve. The signal pipeline has ground-truth millisecond precision; VL2 has human-like gestalt pattern recognition. Both are wrong sometimes in different ways.

2. **VL2 sees a rendered image, not the raw signal.** The pipeline produces zero ECG images at ingestion. Images are synthesized on-demand in Node 1.6 from `morphology_signal` (raw amplitude, 500 Hz grid) with explicit rendering parameters that are part of this specification.

## 1. VLM Landscape for ECG Analysis (2024–2026)

### 1.1 State of the Field

Vision-language models applied to ECG interpretation have advanced rapidly 2023–2026. Key findings:

**What VLMs can do well on ECGs:**
- Pattern recognition for morphologically distinct conditions: LBBB, RBBB, ST elevation, AF, ventricular tachycardia (large visual signatures)
- Global rhythm classification (regular/irregular, narrow/wide QRS)
- Identifying conditions they were trained to recognize in natural ECG images (standard 12-lead grid layout)
- Visual confirmation of "obvious" findings (e.g., 2:1 flutter waves, complete AV dissociation)
- Detecting findings a cardiologist would see gestaltically — Wellens pattern, de Winter pattern, epsilon waves — when these have distinct visual signatures in the rendered image

**What VLMs cannot reliably do:**
- Precise measurement of intervals to within ±10 ms (they estimate, not measure)
- Detect subtle ST changes < 0.5 mm that require comparison to a baseline
- Correctly orient when lead labels are absent or ambiguous
- Reliably distinguish artifact from pathology in noisy signals
- Count exact beat numbers in long rhythm strips

**DeepSeek-VL2 specifically (2024 release):**
- Architecture: mixture-of-experts VLM with dynamic-resolution visual tokenizer
- Supports arbitrary image sizes up to 4096 tokens per image tile
- Strong performance on visual reasoning tasks; competitive with GPT-4V on medical image benchmarks
- Reasoning mode (deepseek-reasoner) available and beneficial for ECG analysis — the model performs better when asked to "think through" what it sees before answering
- No native ECG fine-tuning documented; performance depends entirely on zero-shot/few-shot prompting quality

**Competitive landscape (for awareness only — VL2 is the chosen model):**
- GPT-4V/GPT-4o: strong ECG performance, PHI concerns (cloud)
- Gemini 1.5 Pro: long context, supports multi-image; PHI concerns
- Med-PaLM 2: medically fine-tuned, not publicly available
- LLaVA-Med: fine-tuned on medical images; open-source; lower accuracy than VL2

**Conclusion:** DeepSeek-VL2 in reasoning mode is the appropriate choice. The dynamic-resolution tokenizer allows rendering at clinically meaningful resolution without arbitrary downsampling. The reasoning mode reduces hallucination on subtle findings.

## 2. ECG Rendering for Vision Input

### 2.1 Layout Standard

Standard clinical 12-lead ECG layout:

```
Row 1 (3 sec each):  I   —  aVR  —  V1  —  V4
Row 2 (3 sec each):  II  —  aVL  —  V2  —  V5
Row 3 (3 sec each):  III —  aVF  —  V3  —  V6
Row 4 (10 sec):      II  (continuous rhythm strip)
```

This is the universal clinical format. VL2 was trained on internet data containing standard clinical ECGs in this layout — using a non-standard layout degrades performance.

**Paper speed:** 25 mm/s (standard). One small square = 40 ms = 1 mm.
**Amplitude scale:** 10 mm/mV standard. One small square = 0.1 mV = 1 mm.
**Small square size:** 1 mm; large square = 5 small squares = 5 mm.

### 2.2 Image Rendering Specification

```python
def render_ecg_image(
    morphology_signal: np.ndarray,       # (n_samples_500hz, 12) float64 mV
    lead_names: list[str],               # canonical order, 12 leads
    sampling_rate_hz: int = 500,
    paper_speed_mm_per_s: float = 25.0,
    amplitude_mm_per_mv: float = 10.0,
    small_square_px: int = 6,            # pixels per 1 mm small square
    output_path: str | None = None,
) -> np.ndarray:
    """
    Renders a standard 12-lead ECG image for VL2 input.

    Returns: uint8 RGB array (H, W, 3).

    Design constants at 500 Hz, 25 mm/s, 6 px/mm:
      - Samples per small square: 500 / 25 = 20 samples = 40 ms
      - Pixels per small square: 6 px
      - Pixels per sample: 6/20 = 0.3 px/sample
      - 3-second strip = 1500 samples = 450 px wide
      - Each row = 4 × 450 px = 1800 px wide
      - Amplitude: 10 mm/mV × 6 px/mm = 60 px/mV
      - Typical ECG trace (±2 mV range) = ±120 px = 240 px strip height
      - Full image: ~1800 × 1100 px
    """
```

**Rendering parameters (fixed for VL2 input):**
- `small_square_px = 6` → image is ~1800 × 1100 pixels
- Background: white (#FFFFFF)
- Small grid lines: light red (#FFAAAA), 0.5px
- Large grid lines: red (#FF6666), 1.0px
- ECG trace: black (#000000), 1.5px
- Calibration pulse: 1 mV square wave drawn at the start of each lead strip
- Lead labels: printed in upper-left of each strip (Arial, 10pt, black)
- Speed/gain annotation: printed bottom-left ("25 mm/s  10 mm/mV")
- No fiducial markers, no computed annotations — VL2 sees the raw ECG only

**Why no annotations on the image:**
Providing computed fiducial markers (e.g., marked R peaks) on the image biases VL2 to agree with the signal pipeline rather than providing an independent check. Node 1.6 exists precisely to catch signal-pipeline errors — those errors will not be caught if VL2 is anchored to the same data.

**Critical rendering concern — signal reference for display:**
Use `morphology_signal` (raw amplitude, pre-filtering). This is the signal a cardiologist would see on actual ECG paper. Using `preprocessed_signal` (filtered) would make baseline-wander artifacts disappear before VL2 sees them — hiding one class of findings the vision layer should catch.

**Image token budget:**
At ~1800 × 1100 pixels, with VL2's dynamic tiling (typically 336×336 tiles), the image produces approximately 30–40 tiles = 30–40 × 256 = ~8,000–10,000 image tokens. At DeepSeek-VL2 pricing this is negligible relative to reasoning tokens. Acceptable.

## 3. Vision Prompting Strategy

### 3.1 Prompting Architecture

VL2 is called once per ECG (not per beat, not per lead). It receives:
- One rendered 12-lead ECG image (per Section 2)
- One system prompt establishing the clinical context
- One user prompt with a structured checklist

**System prompt:**
```
You are an experienced cardiologist reviewing a 12-lead ECG image.
You will be given an ECG image and a structured checklist.
For each item in the checklist, provide your assessment and a confidence
level (HIGH / MEDIUM / LOW). If you cannot determine something from the
image, say CANNOT_DETERMINE rather than guessing.
Your findings will be cross-validated against a computational ECG analysis
pipeline. Disagreements will be reviewed by a clinician. It is better to
say CANNOT_DETERMINE than to hallucinate a finding.
```

**User prompt template:**

### 3.2 Why Structured Output (Not Free-Form)

Free-form narrative ECG reports from VLMs are difficult to parse reliably and tend to produce false-positive findings (hallucinated findings stated with confidence). Structured JSON output with explicit CANNOT_DETERMINE options reduces hallucination by giving the model a safe exit. The `image_quality.readable = false` option allows VL2 to reject a rendering failure rather than report findings from a corrupted image.

### 3.3 Reasoning Mode Usage

VL2 is called with reasoning mode enabled (`"thinking": {"type": "enabled", "budget_tokens": 8000}`). For ECG analysis, intermediate thinking significantly improves accuracy on:
- Multi-territory ST-T pattern recognition (e.g., recognizing diffuse concave elevation as pericarditis vs localized convex elevation as STEMI)
- Distinguishing LBBB morphology from pre-excitation
- Interpreting borderline PR or QRS widths

Reasoning token budget = 8,000. This is lower than the diagnostic agent budget (up to 32,000) because VL2's task is pattern recognition, not differential synthesis.

### 3.4 VL2 Invocation Logic

VL2 is invoked on every ECG that passes quality gating, unconditionally. Rationale:

- **Cost:** ~$0.003–0.008/ECG for rendering + VL2 call (negligible relative to diagnostic agents)
- **Value:** Visual cross-check is most valuable precisely when signal-computed features are borderline — we cannot know which ECGs are borderline before running the signal pipeline AND VL2
- **Latency:** VL2 call runs in parallel with the Phase 1 diagnostic agents (Node 2.3 orchestrator is async). VL2 adds ~2–4 seconds but does not extend total pipeline latency if dispatched concurrently

VL2 is NOT invoked when:
- `QualityReport.gate_decision == "REJECT"` — no point analyzing an unreadable ECG visually either
- `morphology_signal` is unavailable (should never happen; Node 1.2 guarantees this)

## 5. Signal-Vision Cross-Validation Protocol

### 5.1 Comparison Logic

After VL2 returns and the `VisionVerificationResult` is populated, a deterministic comparison function checks for conflicts against the Node 1.5 feature object:

```python
def check_signal_vision_agreement(
    features: FeatureObject,
    vision: VisionVerificationResult,
) -> list[dict]:
    """
    Returns a list of conflicts. Empty list = full agreement.
    Each conflict has: feature, signal_value, vision_flags,
    conflict_type ("SIGNAL_ONLY" | "VISION_ONLY" | "BOTH_OPPOSITE"), severity.
    """
    conflicts = []

    # ST elevation conflicts
    for lead in ALL_LEADS:
        sig_elevated = features.aggregate.st_elevation_max_mv.get(lead, 0) >= \
                       get_st_elevation_threshold_mv(lead, features.patient_sex,
                                                     features.patient_age_years)
        vis_elevated = lead in vision.st_elevation_leads
        if sig_elevated and not vis_elevated and \
           vision.confidence_by_section.get("st_t_changes") in ("HIGH", "MEDIUM"):
            conflicts.append({
                "feature": f"st_elevation_{lead}",
                "signal_value": features.aggregate.st_elevation_max_mv.get(lead),
                "signal_flags": sig_elevated,
                "vision_flags": vis_elevated,
                "conflict_type": "SIGNAL_ONLY",
                "severity": "HIGH",
            })
        elif vis_elevated and not sig_elevated:
            conflicts.append({
                "feature": f"st_elevation_{lead}",
                "signal_value": features.aggregate.st_elevation_max_mv.get(lead),
                "signal_flags": sig_elevated,
                "vision_flags": vis_elevated,
                "conflict_type": "VISION_ONLY",
                "severity": "MEDIUM",
            })

    # QRS width conflict
    sig_wide = features.aggregate.qrs_duration_global_mean_ms >= 120
    if vision.qrs_wide is not None and sig_wide != vision.qrs_wide:
        conflicts.append({
            "feature": "qrs_wide",
            "signal_value": features.aggregate.qrs_duration_global_mean_ms,
            "signal_flags": sig_wide,
            "vision_flags": vision.qrs_wide,
            "conflict_type": "BOTH_OPPOSITE",
            "severity": "HIGH",
        })

    # Rhythm regularity conflict
    sig_irregular = features.aggregate.rhythm_regularity_coefficient < 0.75  # RR_cv > 0.25
    if vision.rhythm_regular is not None and sig_irregular == vision.rhythm_regular:
        conflicts.append({
            "feature": "rhythm_regularity",
            "signal_value": features.aggregate.rhythm_regularity_coefficient,
            "signal_flags": not sig_irregular,
            "vision_flags": vision.rhythm_regular,
            "conflict_type": "BOTH_OPPOSITE",
            "severity": "MEDIUM",
        })

    # P wave visibility vs afib
    if features.aggregate.afib_suspected and vision.p_waves_visible is True and \
       vision.confidence_by_section.get("rhythm") in ("HIGH", "MEDIUM"):
        conflicts.append({
            "feature": "p_waves_in_afib",
            "signal_value": "afib_suspected=True",
            "signal_flags": True,
            "vision_flags": vision.p_waves_visible,
            "conflict_type": "BOTH_OPPOSITE",
            "severity": "HIGH",
        })

    return conflicts
```

### 5.2 Conflict Resolution Rules

| Conflict type | Rule |
|---------------|------|
| SIGNAL_ONLY (signal flags, VL2 doesn't) — VL2 HIGH confidence | Soft flag: "VL2 did not confirm. Review recommended." Signal value stands. |
| SIGNAL_ONLY (signal flags, VL2 doesn't) — VL2 LOW/CANNOT_DETERMINE | No flag. VL2 considered unreliable on this finding. Signal value stands. |
| VISION_ONLY (VL2 flags, signal doesn't) | Soft flag: "VL2 suggests finding not in signal features. Review recommended." Signal value stands. |
| BOTH_OPPOSITE — QRS width | HIGH severity flag. Both shown to CDS agent. Neither is suppressed. |
| BOTH_OPPOSITE — STEMI leads | CRITICAL flag. Sent to IT agent with explicit note: "Signal and vision disagree on ST elevation in [lead]." |

**Core rule: the signal pipeline value always goes to the agents.** Vision findings never overwrite signal-computed features. The `signal_vision_conflicts` list is passed to agents as additional context — they may weight it as evidence but cannot be required to accept VL2 over signal measurements.

### 5.3 Vision-Only Findings

Some findings VL2 may report that the signal pipeline does not have a direct equivalent feature for:
- `osborn_wave_hypothermia` — not computed in Node 1.5 (a gap; Node 1.5 can be extended)
- `st_morphology_notable = "de Winter"` — useful confirmation of Node 1.5's `de_winter_features`
- `overall_impression` — free-text clinical summary; passed verbatim to CDS agent

These vision-only findings are passed directly to the CDS agent as supplementary context.

## 6. Confidence Calibration

### 6.1 When to Trust VL2

VL2 findings are most reliable when:
- `image_readable = True` and `leads_obscured = []`
- VL2 confidence for the relevant section is "HIGH"
- The finding is visually large/obvious (LBBB, QRS widening, massive ST elevation)
- VL2 agrees with the signal pipeline (confirmation increases confidence)

VL2 findings are least reliable when:
- VL2 reports LOW confidence or CANNOT_DETERMINE
- The finding requires precision measurement (e.g., QTc: VL2 can say "long QT impression" but cannot say "QTc = 510 ms")
- Artifacts are noted in the relevant region
- The condition requires subtle signal features (< 1 mm ST changes, Brugada type 2 saddle-back)

### 6.2 Hallucination Failure Mode Catalog

| Failure mode | Mechanism | Mitigation |
|-------------|-----------|------------|
| False ST elevation | VL2 sees tall T wave or QRS slur as ST elevation | ST morphology classification cross-check; LOW signal elevation value |
| False AFib | VL2 sees irregular baseline as absent P waves | Check `afib_suspected` from signal; only flag conflict if VL2 is HIGH confidence |
| False LBBB | VL2 sees any wide QRS (including paced rhythm) as LBBB | Signal `lbbb_raw_features` cross-check |
| Territorial mis-assignment | VL2 names wrong lead for elevation | Check against signal per-lead values |
| Image quality attribution | VL2 reports "artifact" when there is none | `image_readable` must be True; low-confidence artifacts ignored |

### 6.3 VL2 Call Failure Handling

If the VL2 API call fails or returns unparseable JSON:
- `vl2_call_succeeded = False`
- All VL2-derived fields set to None
- `signal_vision_conflicts = []`
- Pipeline continues without vision layer — this is non-fatal
- CDS agent receives a flag: `vision_unavailable = True`

The vision pipeline is never on the critical path. If it fails, signal-based diagnosis proceeds normally.

## 7. Rendering Quality Assurance

Before sending the image to VL2:
1. Verify rendered image dimensions are within expected range (1600–2000 × 900–1300 px)
2. Check that at least 8 of 12 leads were rendered with non-zero signal
3. Verify calibration pulse is visible in at least one lead
4. Verify grid lines are present (check pixel color variance)

If any check fails: log warning, attempt re-render once. If re-render fails: set `vl2_call_succeeded = False` and skip VL2 call.

## 9. Pipeline Position and Timing

```
Node 1.5 Feature Extraction
        ↓
Node 1.6 Image Rendering (sync, ~200 ms)
        ↓
    [Dispatch: asyncio.as_completed() pool — 4 awaitables]
        ├── Phase 1 RRC agent
        ├── Phase 1 IT agent
        ├── Phase 1 MR agent
        └── VL2 API call (Node 1.6)          ← 4th awaitable in the pool
                ↓
    [All 4 complete OR MAX_VL2_WAIT_SEC exceeded]
    [Conflict check runs on VL2 result (if available)]
                ↓
    Node 2.3 CDS agent — receives signal features + VL2 result + any conflicts
```

**Node 2.3 Amendment Required:** The Node 2.3 orchestrator must be amended to include the VL2 call as a 4th awaitable in the `iter_phase1_results()` / `asyncio.as_completed()` pool. Without this explicit amendment, the CDS will be dispatched after the 3 Phase 1 agents complete, before VL2 returns, and `vision = None` on every ECG.

**VL2 timeout:** `MAX_VL2_WAIT_SEC = 15` seconds after the rendering completes. If VL2 has not returned within this window, set `vl2_call_succeeded = False` with `reason = "timeout"` and dispatch the CDS without vision results. This prevents VL2 latency from blocking the diagnostic pipeline.

Image rendering is synchronous (200 ms). The VL2 API call is asynchronous and runs in parallel with the three Phase 1 diagnostic agents. Total additional latency is zero if VL2 finishes within the Phase 1 window (~6–10 s).

## 10. Dynamic Research Node: 1.4.2.2 — VL2 Prompting for Fiducial Verification

This dynamic research node was identified at 1.4.2.2. After research, the conclusion is:

**VL2 should NOT be used for per-beat fiducial verification.**

Reasons:
1. Precise fiducial localization requires sub-sample (< 2 ms) accuracy. VL2 cannot localize to pixel precision at ECG image scale — one small square = 40 ms = 6 pixels. Asking VL2 "where exactly does the T wave end?" will produce answers with ±20–40 ms precision at best.
2. A per-beat VL2 call (n = 5–15 beats per ECG) would cost 5–15× the image tokens and add latency on the critical path.
3. ECGdeli's consensus fiducial positions are already validated against PTB-XL ground truth (Node 1.4.3). VL2 cannot improve on a validated algorithmic detector for fiducial positions.

**What VL2 CAN contribute for fiducial-related quality:**
- Visual confirmation that the P wave is globally absent (AFib check) — `p_waves_visible`
- Flagging that QRS morphology is "WIDE" (cross-checks ECGdeli's fiducial-based QRS duration)
- Detecting that rhythm strip lead is obscured by artifact

The fiducial cross-validation role (1.4.2.3) is fulfilled by the `qrs_wide` and `rhythm_regular` fields in `VisionVerificationResult` plus the conflict checking in Section 5.

Dynamic node 1.4.2.2 is resolved as: "VL2 does not verify individual fiducial positions; it provides global morphological verification at the rhythm and pattern level."
