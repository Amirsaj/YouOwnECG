# RRC Adversarial Review — Node 1.6 Vision Pipeline (FD-FPRE)

**Reviewed by:** Research Review Committee (RRC)
**Document under review:** `docs/architecture/nodes/1.6_vision_pipeline.md`
**Review date:** 2026-03-29
**Verdict:** GATE CONDITIONAL

---

## Preamble

Node 1.6 is the most architecturally uncertain component of SDA-1. VL2 is called with reasoning mode on every ECG, its output is cross-validated against signal features, and conflicts are passed to the CDS agent. The document is well-structured. Two issues are raised: one REQUIRED (a race condition in the pipeline timing design that could deliver incomplete vision results to the CDS agent), one REQUIRED (VL2 receives no patient context in the rendered image, but the prompt provides sex and age — the rendered image lacks the speed/gain annotation that anchors VL2's scale perception, and this is internally inconsistent with the spec).

---

## Issue 1 — CDS Agent Is Not Guaranteed to Receive VL2 Results Before Executing

**Severity: REQUIRED**

**Location:** Section 9, pipeline timing:

```
[Dispatch to async parallel execution]
    ├── Phase 1 agents (RRC, IT, MR) — text-only
    └── VL2 API call (async, ~2–4 s)
            ↓
[VL2 result arrives; conflict check runs]
            ↓
CDS agent — receives signal features + VL2 result + any conflicts
```

**The flaw:**

The diagram states VL2 runs in parallel with Phase 1 agents and the CDS waits for "all Phase 1 results including VL2." But the Node 2.3 orchestrator design (Gate Passed 2026-03-28) specifies `iter_phase1_results()` with `asyncio.as_completed()` — results are processed as they arrive, and the CDS is dispatched after all three Phase 1 agents complete. Node 1.6 is a fourth asynchronous task that must be added to the `asyncio.as_completed()` pool.

If the Node 2.3 orchestrator is implemented without explicit knowledge that VL2 is a fourth awaitable in the Phase 1 pool, the CDS will be dispatched after RRC + IT + MR complete, but before VL2 returns. The CDS will receive `vision = None` on every ECG, silently.

The document says "CDS agent waits for all Phase 1 results including VL2" but does not specify HOW this is enforced at the orchestrator level — specifically, it does not say VL2 must be added to the `iter_phase1_results()` pool in Node 2.3, or that Node 2.3 must be amended to include VL2.

**Required action:** (1) Document that Node 2.3 orchestrator must be amended: the VL2 call is a fourth awaitable dispatched alongside RRC/IT/MR in `asyncio.as_completed()`. (2) Specify explicit timeout for VL2: if VL2 has not returned within `MAX_VL2_WAIT_SEC = 15` seconds after Phase 1 agents complete, dispatch the CDS without VL2 results (set `vl2_call_succeeded = False` with reason `"timeout"`). This prevents VL2 latency from blocking the diagnostic pipeline. (3) Add amendment notice: "Node 2.3 orchestrator requires amendment to include VL2 as a Phase 1 parallel task."

## Issue 2 — Rendering Spec Specifies Speed/Gain Annotation in Section 2.2 but Section 2.3 Omits It from the Implementation

**Severity: REQUIRED**

**Location:** Section 2.2:
> Speed/gain annotation: printed bottom-left ("25 mm/s  10 mm/mV")

Section 2.3:
```python
LEADS_LAYOUT = [
    ["I", "aVR", "V1", "V4"],
    ...
]
RHYTHM_STRIP_LEAD = "II"
SAMPLES_PER_STRIP = 1500
```

**The flaw:**

The implementation sketch in Section 2.3 lists only layout constants. It contains no mention of the calibration pulse (Section 2.2), the speed/gain annotation (Section 2.2), or the lead labels. These are not implementation details — they are critical for VL2's ability to interpret the image correctly:

- Without the calibration pulse, VL2 cannot anchor its amplitude scale perception. It will not know whether a deflection is 0.5 mV or 2.0 mV.
- Without the speed annotation, VL2 cannot verify its interval estimation (it may assume 50 mm/s or 12.5 mm/s if not told).
- Without lead labels on the image, VL2 must infer lead identity from position — possible only if the standard 12-lead layout is perfectly rendered and VL2 was trained on that layout.

The document is internally inconsistent: Section 2.2 specifies these elements are required, but Section 2.3 (the implementation spec) does not include them in the constants or function signature.

**Required action:** Add explicit constants to Section 2.3:
```python
CALIBRATION_PULSE_MV = 1.0          # 1 mV square wave at start of each row
CALIBRATION_PULSE_DURATION_MS = 200  # 200 ms wide
SHOW_SPEED_GAIN_ANNOTATION = True    # "25 mm/s  10 mm/mV" bottom-left
SHOW_LEAD_LABELS = True              # lead name in upper-left of each strip
```
And specify that `render_ecg_image()` must draw the calibration pulse and annotations as part of the function contract — not as optional post-processing.

## Verdict

**GATE CONDITIONAL**

Node 1.6 is architecturally sound. The advisory-not-authoritative principle is correctly specified, the conflict resolution rules are well-reasoned, and the dynamic node 1.4.2.2 resolution (no per-beat fiducial VL2 verification) is the right call. Both issues are small fixes. The gate opens when they are addressed.
