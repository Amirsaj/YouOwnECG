## Issues Found

### REQUIRED-1: assemble_qa_context() passes full feature object to LLM

**Location:** Node 3.1, Section 5 (Q&A context assembly)

The `assemble_qa_context()` function assembles a context dict and sends it to the LLM for Q&A. The spec does not limit which fields are included. If the full FeatureObject is forwarded, the LLM context window is consumed by ~8,000 tokens of raw signal data that is irrelevant to the user's question. This leaves insufficient room for retrieval context and the reasoning chain.

**Required fix:** The context assembly must pass only the BeatSummary (Node 2.5, ~600 tokens) + the relevant per-beat detail if the selected beat is a PVC or PAC + the DiagnosticResult (finding names + confidence + key feature values, not the full object). The full FeatureObject is NOT forwarded to the LLM.

### REQUIRED-2: Beat selection via R-peak proximity has no fallback for missed R-peaks

**Location:** Node 3.1, Section 4 (beat selection)

The beat selection algorithm uses `argmin(|click_x - R_peak_positions|)`. If the quality assessment (Node 1.3) marked a lead as POOR and the R-peak detector produced no R-peaks for that lead, the R-peak array is empty and `argmin` returns an undefined index. The UI would crash or return beat index 0 silently.

**Required fix:** Before beat selection, check that `r_peak_positions` is non-empty for the selected lead. If empty, show a tooltip: "Beat selection unavailable for this lead — signal quality insufficient." Disable the Q&A context assembly for that click.
