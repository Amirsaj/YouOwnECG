## Executive Summary

Node 2.2 defines the internal design of each of the four agents in the confirmed Option E architecture. Where Node 2.1 answered "which architecture" (3 parallel Phase 1 specialists + 1 sequential CDS), Node 2.2 answers "how exactly is each agent built" — its mission boundary, reasoning chain, structured input/output schema, cross-domain communication protocol, prompt architecture, and failure modes.

The central design challenge this document resolves is the **feature routing question**: all three Phase 1 agents receive an identical full-feature object. This is a deliberate choice. Restricting agents to domain-sliced features would require the orchestrator to perform feature classification before the agents run, creating a chicken-and-egg dependency. Full-feature routing also lets each agent apply its domain lens to borderline features (e.g., the RRC Agent can see T-wave amplitudes to detect TdP pause-dependent patterns even though T-wave morphology is nominally the MR Agent's domain). The agents are scoped by their *reasoning mission*, not by feature access.

The second major design decision is the **cross_domain_hint controlled vocabulary**. Rather than freetext strings (which the RRC review identified as an unvalidated communication channel), this document defines a typed vocabulary of 17 hint categories. Each hint has a defined `hint_type` enum, a structured `payload`, and a required CDS response. This makes inter-agent communication testable and deterministic.

The third decision is **CDS raw feature selection**. CDS always receives a minimal base set (12 fields) plus any features named in Phase 1 `raw_feature_flags` arrays, subject to a 2,000-token budget. The selection mechanism is explicit — Phase 1 agents flag by feature name, not by value. This closes the "safety net is undefined" issue identified by the RRC review.

What this document does NOT specify: exact prompt text (that is implementation work for Node 2.2's execution phase), the orchestration runtime (Node 2.3), and the API integration layer (Node 2.6). Section 9 lists the specific requirements this design places on those downstream nodes.

## 1. Feature Input Specification

### 1.1 SDA-1 Feature Set Required

The SDA-1 pipeline (Ingest → Preprocess → Quality → Fiducials → Features) must produce a structured feature object for each ECG. ECGdeli provides a 13-column Fiducial Point Table (FPT): `Pon, Ppeak, Poff, QRSon, Q, R, S, QRSoff, L, Ton, Tpeak, Toff, class` per beat per lead. From this, the Feature Extraction node (1.5) computes derived features.

**Required fields:**

*Per-beat fields (array of N beats):*
- `rr_intervals_ms[]` — RR interval preceding each beat, milliseconds
- `pp_intervals_ms[]` — PP interval preceding each P-wave
- `pr_intervals_ms[]` — PR interval per beat (QRSon − Ppeak), ms
- `qrs_duration_ms[]` — QRS duration per beat (QRSoff − QRSon), ms
- `qt_intervals_ms[]` — QT interval per beat (Toff − QRSon), ms
- `beat_class[]` — ECGdeli class column: 0=normal, 1=LBBB, 2=RBBB, 3=PVC, 4=noise
- `p_wave_present[]` — boolean per beat
- `p_wave_morphology_lead_ii[]` — peak amplitude of P in lead II, mV
- `p_wave_duration_ms[]` — P-wave duration per beat (Poff − Pon from FPT), ms; null if P-wave absent
- `fusion_beat_flag[]` — boolean (intermediate QRS morphology vs dominant)

*Per-beat per-lead fields (N beats × 12 leads):*
- `st_amplitude_j_point_mv[beat][lead]` — ST amplitude at J+60ms, mV
- `st_amplitude_j80_mv[beat][lead]` — ST amplitude at J+80ms (backup measurement)
- `st_morphology[beat][lead]` — enum: `horizontal|upsloping|downsloping|concave|convex|saddle|coved`
- `t_wave_amplitude_mv[beat][lead]` — T-wave peak amplitude, mV
- `t_wave_symmetry[beat][lead]` — float 0-1: `min(Tpeak−Ton, Toff−Tpeak) / max(Tpeak−Ton, Toff−Tpeak)`; 1.0=perfectly symmetric, approaches 0 as one limb dominates. Set to null for beats with beat_class=1 (LBBB) or beat_class=2 (RBBB). Set to null if Tpeak=Toff (degenerate/inverted T). Requires recording_duration_sec ≥60 for reliable TWA detection.
- `t_wave_morphology[beat][lead]` — enum: `upright|inverted|biphasic|flat|hyperacute|notched`
- `r_wave_amplitude_mv[beat][lead]` — R-wave peak amplitude, mV
- `s_wave_amplitude_mv[beat][lead]` — S-wave depth (negative value), mV
- `q_wave_duration_ms[beat][lead]` — Q-wave duration, 0 if absent
- `q_wave_amplitude_mv[beat][lead]` — Q-wave depth (negative), 0 if absent
- `rs_interval_ms[beat][lead]` — RS interval per Brugada Step 2: computed as `S_fiducial − QRSon_fiducial` (ms), matching the criterion "onset of QRS to nadir of S wave"

*Aggregate fields (computed across all beats):*
- `heart_rate_ventricular_bpm` — 60000 / mean(rr_intervals)
- `heart_rate_atrial_bpm` — 60000 / mean(pp_intervals), null if P-waves absent
- `rhythm_regularity` — float: SD(rr_intervals) / mean(rr_intervals); <0.05 = regular
- `qtc_bazett_ms` — QTc = QT / √(RR_sec), mean across beats
- `qtc_fridericia_ms` — QTc = QT / RR^(1/3), mean across beats
- `qrs_axis_degrees` — mean electrical axis from leads I and aVF
- `r_wave_progression_v1_v6[]` — R-wave amplitude sequence across V1-V6 (6 values)
- `av_ratio` — ratio of P-waves to QRS complexes detected
- `t_wave_alternans_present` — boolean (beat-to-beat alternation in T-wave amplitude >1.9μV)
- `t_wave_alternans_lead` — lead where TWA detected, null if absent
- `beat_to_beat_qrs_morphology_stable` — boolean: same morphology every beat
- `signal_quality_per_lead[]` — quality score 0-1 per lead (from Node 1.3)

*Metadata:*
- `patient_sex` — `M|F|unknown` (affects QTc thresholds and STEMI criteria)
- `patient_age_years` — integer or null
- `ecg_id` — UUID
- `lead_placement_verified` — boolean (dextrocardia detection flag from SDA-1)
- `total_beats_analyzed` — integer
- `recording_duration_sec` — float

### 1.2 Feature Routing Strategy

**Decision: All three Phase 1 agents receive the identical full feature object.**

Rationale:
1. **Avoids pre-classification chicken-and-egg**: Routing by domain requires knowing which features belong to which domain before analysis. But domain classification of a feature is itself a clinical judgment — peaked T-waves could belong to IT (hyperacute ischemia) or MR (hyperkalemia). Pre-routing would force a decision at the orchestrator level that should be made at the reasoning level.
2. **Enables cross-boundary verification**: RRC Agent needs T-wave timing (from `qt_intervals_ms`) to detect pause-dependent TdP risk. MR Agent needs `rr_intervals_ms` to compute QTc. Restricting access would require inter-agent queries that add latency and complexity.
3. **Scope is enforced by prompt, not by data**: Each agent's system prompt defines its mission boundary. An RRC Agent that can see ST-segment data but is instructed not to classify ischemia patterns is well-behaved — it simply ignores out-of-scope findings. This is validated by the fact that LLMs can follow scoped instructions reliably at temperature=0.
4. **Simplifies orchestrator**: The orchestrator sends one feature object to all three Phase 1 agents concurrently. No routing logic required.

**Token budget**: The full feature object for a 10-second ECG (~12 beats, 12 leads) is estimated at 1,800-2,600 tokens in JSON. This is within the 1,500-2,500 token Phase 1 input budget from Node 2.1 Section 6.1.

## 2. RRC Agent Design

### 2.1 Mission Statement

The RRC Agent is responsible for everything that relates to **how the heart is beating** and **how the electrical impulse travels**. It owns: rate (ventricular and atrial), rhythm classification and regularity, P-wave detection and morphology, AV relationship and dissociation, PR interval, QRS duration, QRS axis, and all conduction disorders (AV blocks, bundle branch blocks, fascicular blocks, accessory pathways).

The RRC Agent is explicitly NOT responsible for: ST-segment ischemia classification (IT Agent), T-wave morphology beyond timing (MR Agent), voltage criteria for hypertrophy (MR Agent), or QTc measurement (MR Agent). The RRC Agent measures QT interval duration but does not compute or interpret QTc — that belongs to MR.

When VT is detected, the RRC Agent is the primary agent. It applies initial AV dissociation assessment, rate and regularity analysis, and QRS morphology classification (LBBB vs RBBB pattern of the VT). The full Brugada/Vereckei multi-criterion algorithm is executed by the CDS Agent after receiving RRC + IT + MR outputs, because VT discrimination requires simultaneous assessment of morphology (MR), concordance (IT-adjacent), and rhythm (RRC).

### 2.2 System Prompt Architecture

The RRC system prompt has five sections, in order:

**Section 1 — Role and Scope** (~200 tokens): Defines the agent as the Rhythm/Rate/Conduction specialist. Explicitly states what the agent is responsible for and what it must not analyze. Sets the clinical stakes: this is a STAT ECG analysis system for emergency medicine; rhythm disorders can cause immediate death; false negatives are worse than false positives.

**Section 2 — Reasoning Protocol** (~600 tokens): The ordered reasoning steps (see Section 2.3). Written as a numbered checklist the agent must follow in order. Includes decision branch conditions: "If QRS >120ms AND rate >100 → wide complex tachycardia protocol."

**Section 3 — Criteria Reference** (~800 tokens): Key thresholds embedded directly (not RAG-dependent): AV block PR thresholds, BBB QRS duration thresholds, VT rate threshold, AV dissociation definition, Wenckebach pattern description, complete AVB definition. This prevents hallucination of criteria values. RAG provides context; criteria are in the prompt.

**Section 4 — cross_domain_hint Emission Rules** (~200 tokens): The 5 conditions under which the RRC Agent must emit a cross_domain_hint (see Section 2.5). Written as explicit IF/THEN rules.

**Section 5 — Output Schema** (~400 tokens): The exact JSON schema the agent must produce at the end of its reasoning. Includes field names, types, and one example. Instructs: "After completing your reasoning, output a valid JSON object matching this schema exactly. Do not include any text after the JSON."

**Total estimated system prompt: ~2,200 tokens** — within the 3,000-token cap.

### 2.3 Reasoning Chain Design

The RRC Agent follows this ordered reasoning protocol:

```
Step 1: RATE
  1a. Compute mean ventricular rate from rr_intervals_ms
  1b. Compute mean atrial rate from pp_intervals_ms (if P-waves present)
  1c. Classify: bradycardia (<60), normal (60-100), tachycardia (>100)
  1d. Note rate variability (rhythm_regularity_coefficient)

Step 2: P-WAVE ANALYSIS
  2a. Are P-waves present? (p_wave_present[] majority)
  2b. If present: morphology in lead II (peaked >2.5mm → RAE; notched/wide → LAE)
  2c. Is P-wave morphology consistent across beats (sinus origin)?
  2d. Count distinct P-wave morphologies (≥3 → MAT or WAP)
  2e. Measure PP interval regularity

Step 3: RHYTHM CLASSIFICATION
  3a. Is rhythm regular (regularity_coefficient <0.05)?
  3b. Irregular? → is it regularly-irregular (grouped beating) or irregularly-irregular?
  3c. Absent P-waves + irregularly-irregular → AFib
  3d. Flutter waves at 250-350bpm → AFlutter
  3e. Regular narrow tachycardia with retrograde P or no P → SVT
  3f. Wide complex tachycardia (QRS >120ms + rate >100) → wide complex tachycardia protocol (Step 6)

Step 4: AV RELATIONSHIP
  4a. Is every P followed by a QRS? Is every QRS preceded by a P?
  4b. Compute P:QRS ratio (av_ratio)
  4c. Is PR interval constant across beats?
  4d. If PR increasing with dropped beat → Wenckebach (Type I 2AVB)
  4e. If PR constant with sudden dropped QRS → Mobitz II (Type II 2AVB)
  4f. If P-waves and QRS complexes are independent (AV dissociation) → suspect VT or Complete AVB
  4g. AV dissociation + bradycardia (escape rate) → Complete AVB
  4h. AV dissociation + tachycardia (ventricular rate >100) → VT (emit cross_domain_hint)

Step 5: INTERVALS
  5a. PR interval: mean and range. >200ms → first-degree AVB. <120ms → WPW or junctional
  5b. QRS duration: mean across beats. <120ms → narrow; 100-119ms → incomplete BBB; ≥120ms → complete BBB or VT
  5c. If QRS ≥120ms → classify morphology (Step 6)
  5d. Do NOT compute QTc — that belongs to MR Agent

Step 6: QRS MORPHOLOGY (if QRS ≥120ms OR wide complex tachycardia)
  6a. In V1: is dominant deflection positive (RBBB pattern) or negative (LBBB pattern)?
  6b. RBBB criteria: rSR' or RSR' in V1 + wide S in I/V5-V6
  6c. LBBB criteria: QS or rS in V1 + broad monophasic R in I/V5-V6 + absent septal Q in I/V5
  6d. Fascicular blocks: axis deviation without QRS widening → LAFB (LAD -30 to -90) or LPFB (RAD +90 to +180)
  6e. Bifascicular: RBBB + LAFB or RBBB + LPFB
  6f. For wide complex tachycardia: note QRS morphology (LBBB-pattern or RBBB-pattern), axis, AV dissociation

Step 7: CONDUCTION FINDINGS ASSEMBLY
  7a. List all conduction abnormalities found
  7b. Assign STAT flag to: Complete AVB, VT (rate >150), VF (no organized complexes), high-degree AVB with hemodynamic compromise
  7c. Emit cross_domain_hints per Section 2.5 rules
  7d. Populate raw_feature_flags for any findings needing CDS verification
```

### 2.5 cross_domain_hint Emission Rules

The RRC Agent emits a cross_domain_hint under these conditions:

| Trigger | hint_type | Payload | Required CDS Response |
|---------|-----------|---------|----------------------|
| LBBB detected (QRS ≥120ms + LBBB morphology confirmed) | `LBBB_DETECTED` | `{ "qrs_duration_ms": float, "morphology_confirmed": true }` | Load Sgarbossa criteria RAG; apply Sgarbossa to any IT ischemic ST findings |
| AV dissociation confirmed + ventricular rate >100 | `WCT_AV_DISSOCIATION` | `{ "ventricular_rate_bpm": float, "atrial_rate_bpm": float }` | Integrate with MR morphology and IT concordance to apply Brugada/Vereckei VT criteria |
| Complete AVB detected (AV dissociation + escape rate <60) | `COMPLETE_AVB_ISCHEMIC_CONTEXT` | `{ "escape_rate_bpm": float, "escape_qrs_morphology": string }` | Check IT output for concurrent inferior STEMI — if present, flag ischemic AVB (transient, good prognosis); if anterior STEMI — permanent, poor prognosis |
| WPW pattern detected (short PR + delta wave) | `WPW_DETECTED` | `{ "pr_interval_ms": float, "delta_wave_leads": [] }` | Suppress IT pseudo-Q-wave findings in inferior leads; flag AFib+WPW risk if rhythm is AFib |
| Wide complex tachycardia detected but AV dissociation NOT confirmed | `WCT_NO_AV_DISSOCIATION` | `{ "qrs_duration_ms": float, "rate_bpm": float, "morphology": string, "rs_max_ms": float }` | Apply full Brugada + Vereckei algorithm using all three Phase 1 outputs |

### 2.6 Known Failure Modes

1. **P-wave buried in T-wave at high rates**: At ventricular rates >150 bpm, the P-wave of the next beat may merge with the preceding T-wave, causing `p_wave_present` to return false. The agent may misclassify sinus tachycardia as junctional rhythm or SVT. Mitigation: the prompt instructs the agent to note "P-waves may be obscured at this rate — classification confidence reduced."

2. **Fusion beat misclassification in VT**: A fusion beat (intermediate morphology between sinus and ventricular) is pathognomonic for VT but may be missed if `beat_to_beat_qrs_morphology_stable = true` (because fusion beats are rare in a run). The agent must check per-beat morphology when wide complex tachycardia is suspected, not rely on the aggregate stability flag.

3. **Rate-dependent BBB mistaken for fixed BBB**: If tachycardia causes rate-dependent LBBB, the RRC Agent may report LBBB without noting it is rate-dependent. This triggers Sgarbossa evaluation unnecessarily. Mitigation: the agent should note if BBB appeared only at elevated rate vs baseline.

4. **AFib + regular ventricular rate misclassified**: If a patient has AFib with a very consistent ventricular response (digoxin effect, complete AVB with junctional escape), the RRC Agent may classify the rhythm as "regular" and miss the AFib (absent P-waves with what appears to be regular RR). The agent must explicitly check P-wave morphology before assigning "regular" rhythm classification.

## 3. IT Agent Design

### 3.1 Mission Statement

The IT Agent is responsible for everything that relates to **where ischemia is occurring and which coronary territory is involved**. It owns: ST-segment elevation measurement and territory classification, ST-segment depression (horizontal and downsloping), ischemic T-wave changes in the context of territory (hyperacute T-waves, de Winter upsloping pattern, Wellens T-wave inversions), pathological Q-waves as infarction markers, reciprocal change detection, and coronary territory mapping (LAD/RCA/LCx/posterior/RV).

The IT Agent is explicitly NOT responsible for: T-wave morphology analysis outside territorial context (MR Agent), QTc measurement (MR Agent), rhythm disorders (RRC Agent), or voltage criteria (MR Agent). When LBBB is present (signalled by RRC's `LBBB_DETECTED` hint), the IT Agent must flag its ST findings as "LBBB-confounded — standard STEMI criteria not valid; Sgarbossa assessment required" and NOT produce a standalone STEMI diagnosis. The CDS Agent resolves STEMI-in-LBBB.

### 3.2 System Prompt Architecture

**Section 1 — Role and Scope** (~200 tokens): Ischemia/Territory specialist definition. Scope statement. Clinical stakes: missed STEMI = preventable death; lower threshold for flagging suspicious findings.

**Section 2 — Reasoning Protocol** (~700 tokens): Ordered steps for ST analysis, territory mapping, and ischemia pattern recognition (see Section 3.3).

**Section 3 — Criteria Reference** (~900 tokens): Embedded STEMI thresholds (sex/age-specific: ≥2.5mm in V2-V3 for men <40, ≥2mm for men ≥40, ≥1.5mm for women; ≥1mm in all other leads; ≥0.5mm in V7-V9 for posterior). De Winter criteria: upsloping ST depression ≥1mm at J-point in V1-V6 + peaked symmetric T-waves. aVR criteria: ST elevation in aVR ≥1mm with diffuse ST depression = left main or proximal LAD occlusion. Coronary territory lead groups: LAD=V1-V4; RCA=II/III/aVF; LCx=I/aVL/V5-V6; posterior=V7-V9 or V1-V3 reciprocal.

**Section 4 — LBBB Handling Rule** (~150 tokens): Explicit instruction: if `cross_domain_hint.hint_type == "LBBB_DETECTED"` in the session context, do not make standalone STEMI determinations. Flag all ST findings as LBBB-confounded and emit `LBBB_STEMI_ASSESSMENT_NEEDED` hint for CDS.

**Section 5 — cross_domain_hint Rules** (~200 tokens): 4 conditions (see Section 3.5).

**Section 6 — Output Schema** (~400 tokens).

**Total estimated system prompt: ~2,550 tokens** — within 3,000-token cap.

### 3.3 Reasoning Chain Design

```
Step 1: ST ELEVATION SURVEY (per lead)
  1a. For each lead: measure st_j60_mv (primary) and st_j80_mv (backup)
  1b. Apply sex/age-specific thresholds to classify each lead as: elevated|normal|depressed
  1c. Record morphology per lead (concave/convex/horizontal/upsloping/downsloping/saddle/coved)
  1d. Flag leads with elevation: list them in st_elevation_leads[]

Step 2: TERRITORY MAPPING (if any elevation detected)
  2a. Map elevated leads to coronary territory:
      V1-V4 → LAD (anterior); II/III/aVF → RCA or LCx (inferior);
      I/aVL/V5-V6 → LCx or diagonal (lateral); V1-V3 depression + tall R = posterior
  2b. Does elevation fit a single territory? Multiple contiguous leads in one territory → STEMI
  2c. Diffuse elevation (≥4 territories, saddle-shaped) → pericarditis pattern
  2d. Elevation only in aVR + diffuse depression → left main/proximal LAD pattern

Step 3: ST DEPRESSION SURVEY
  3a. Identify all leads with horizontal or downsloping ST depression ≥0.5mm
  3b. Are depressions territorial (matching a territory) or diffuse?
  3c. Territorial depression in V1-V3 (horizontal/downsloping) + no elevation elsewhere → flag for posterior STEMI assessment (emit cross_domain_hint)
  3d. Upsloping ST depression V1-V6 at J-point → de Winter pattern if associated with peaked T-waves

Step 4: DE WINTER PATTERN CHECK
  4a. Are there leads with upsloping ST depression ≥1mm at J-point in V1-V4?
  4b. If yes: check t_wave_morphology for symmetry and hyperacute appearance in same leads
  4c. De Winter pattern = upsloping STD + peaked symmetric T-waves in anterior leads
  4d. This pattern = STEMI equivalent → STAT flag

Step 5: WELLENS PATTERN CHECK
  5a. Are there deep (>2mm) symmetric T-wave inversions in V2-V3 (Type B) or biphasic T in V1-V3 (Type A)?
  5b. If yes: verify ABSENCE of ST elevation (no acute injury current)
  5c. Verify ABSENCE of Q-waves in V2-V3 (not completed MI)
  5d. If criteria met: flag as Wellens pattern = critical LAD stenosis, STEMI equivalent

Step 6: RECIPROCAL CHANGE ASSESSMENT
  6a. For each identified ischemic territory: check expected reciprocal leads
  6b. Inferior STEMI (II/III/aVF elevation) → expect reciprocal STD in aVL/I
  6c. Lateral STEMI → expect reciprocal in III/aVF
  6d. Record reciprocal_changes_present

Step 7: Q-WAVE ASSESSMENT
  7a. Identify pathological Q-waves: duration ≥40ms OR depth ≥25% of R-wave amplitude OR QS pattern
  7b. Map Q-waves to territory
  7c. New Q-waves in STEMI territory → evolving MI (older findings do not exclude acute STEMI)

Step 8: ASSEMBLY
  8a. List all ischemic findings with lead evidence
  8b. Assign STAT flag to: any STEMI, de Winter, Wellens, diffuse STD + aVR elevation
  8c. Emit cross_domain_hints
  8d. Populate raw_feature_flags for findings needing CDS verification
```

### 3.6 Known Failure Modes

1. **de Winter T-wave missed in tachycardia**: At high rates (>100 bpm), the upsloping ST depression of de Winter is compressed and the characteristic morphology is harder to detect. The upsloping slope may appear shorter and the T-wave may look less symmetric. This is the most dangerous failure mode of the IT Agent — a missed LAD occlusion in a young patient.

2. **Posterior STEMI missed without right-sided leads**: The IT Agent can only flag posterior STEMI based on V1-V3 mirror image. Without V7-V9 leads (which are not in the standard 12-lead), the diagnosis is inferential. The agent must emit the `POSTERIOR_STEMI_SUSPECT` hint even with moderate confidence (>0.4) — CDS verifies.

3. **Pericarditis ST elevation classified as STEMI**: Diffuse, saddle-shaped ST elevation with PR depression looks like multi-territory STEMI to a rule-based system. The IT Agent should detect the saddle morphology and diffuse distribution and explicitly note "pericarditis vs multi-vessel STEMI" rather than asserting STEMI. CDS resolves.

4. **Confounded findings in VT**: During VT, ST segments are dominated by the abnormal depolarization and are not assessable for ischemia. The IT Agent must check the RRC output for wide complex tachycardia and, if confirmed, flag all ST findings as "uninterpretable during wide complex tachycardia."

## 4. MR Agent Design

### 4.1 Mission Statement

The MR Agent is responsible for everything that relates to **how the myocardium looks electrically** beyond rate, rhythm, and acute ischemia: QRS morphology as it reflects structural disease, T-wave morphology as a repolarization signal (excluding territorial ischemic T-waves which belong to IT), QTc measurement and interpretation, R-wave progression, voltage criteria for chamber enlargement, U-waves, and special repolarization patterns (early repolarization, Brugada morphology, Osborn waves, epsilon waves, delta waves, strain patterns, T-wave alternans).

The MR Agent is NOT responsible for: ST elevation territory (IT Agent), rhythm classification (RRC Agent), or axis calculation (RRC Agent). The MR Agent does compute QTc (Bazett and Fridericia, using the aggregate fields provided) and interprets the result. It also performs T-wave morphology classification that informs but does not replace the IT Agent's territorial assessment — both agents see T-wave data, but IT classifies T-waves in the context of territory and MR classifies T-waves in the context of repolarization patterns.

### 4.2 System Prompt Architecture

**Section 1 — Role and Scope** (~200 tokens).

**Section 2 — Reasoning Protocol** (~700 tokens): Ordered morphology analysis steps (see Section 4.3).

**Section 3 — Criteria Reference** (~900 tokens): Voltage criteria (Sokolow-Lyon: SV1+RV5≥35mm; Cornell: RaVL+SV3>28mm men/20mm women; Romhilt-Estes ≥5 points); Brugada Type 1 morphology (coved: J-point elevation ≥2mm in V1-V2 with downsloping ST ending in inverted T); QTc thresholds (normal: men <440ms, women <460ms; prolonged ≥500ms = STAT TdP risk; short QT: <340ms); T-wave alternans detection threshold (>1.9μV beat-to-beat amplitude difference); Wellens Type A (biphasic T in V1-V3) and Type B (deep symmetric inversion in V2-V3).

**Section 4 — cross_domain_hint Rules** (~200 tokens): 4 conditions (Section 4.5).

**Section 5 — Output Schema** (~400 tokens).

**Total estimated system prompt: ~2,400 tokens** — within 3,000-token cap.

### 4.3 Reasoning Chain Design

```
Step 1: T-WAVE MORPHOLOGY SURVEY (per lead)
  1a. For each lead: classify t_wave_morphology using the enum
  1b. Note amplitude (absolute value), symmetry score (0-1)
  1c. Identify leads with inverted T-waves; note symmetry score (symmetric deep inversion = score near 1.0)
  1d. Identify leads with hyperacute T-waves (tall, asymmetric, broad base) — flag for IT cross-check
  1e. Identify leads with peaked narrow T-waves (tall, narrow base, symmetric) — hyperkalemia or short QT

Step 2: T-WAVE DISTRIBUTION ANALYSIS
  2a. Are T-wave abnormalities territorial (confined to one coronary region) or diffuse?
  2b. Diffuse peaked T-waves → metabolic etiology likely (hyperkalemia); emit cross_domain_hint
  2c. T-wave inversions limited to right precordial leads (V1-V4) → RVH strain, ARVC, PE right heart strain
  2d. T-wave inversions limited to V2-V3 with preserved or absent Q-waves → flag `symmetric_inversion_v2v3 = true`; do NOT classify as Wellens (Wellens diagnosis is IT's exclusive domain — MR emits raw morphology data only)
  2e. Symmetric deep T-wave inversions in V3-V6 + tall precordial R-waves → apical HCM

Step 3: QTc MEASUREMENT
  3a. Use aggregate.qtc_bazett_ms and qtc_fridericia_ms (already computed by SDA-1)
  3b. Apply sex-specific thresholds: normal (men <440, women <460), borderline (440-499, 460-499), prolonged (≥500)
  3c. Classify: normal|borderline_prolonged|prolonged|short
  3d. Short QT (<340ms): flag for Short QT syndrome; check T-wave morphology (narrow-based peaked T)
  3e. QTc ≥500ms: STAT flag — TdP risk

Step 4: T-WAVE ALTERNANS
  4a. Check t_wave_alternans_present (from aggregate)
  4b. If present: note lead(s) affected; flag STAT — imminent TdP risk if also QTc ≥500ms

Step 5: QRS MORPHOLOGY FOR STRUCTURAL DISEASE
  5a. R-wave progression V1-V6: normal (R grows from V1 to V5-V6); poor (R fails to grow before V4); reverse (R shrinks from V1)
  5b. Voltage criteria for LVH: apply Sokolow-Lyon, Cornell criteria. Note: LVH in presence of LBBB is unreliable.
  5c. Brugada morphology in V1-V2: Type 1 (coved: J≥2mm, downsloping ST, inverted T) vs Type 2 (saddle-back) vs Type 3
  5d. Epsilon waves in V1-V3: small positive deflection after QRS, best seen in right precordial leads (ARVC marker)
  5e. Osborn/J-waves: positive deflection at J-point in lateral/inferior leads (hypothermia)
  5f. Delta waves: slurred QRS onset (WPW — if not already flagged by RRC)

Step 6: VOLTAGE AND ATRIAL CRITERIA
  6a. LVH voltage criteria assessment (per Section 3 reference)
  6b. RVH: dominant R in V1 (R/S >1 in V1 + R ≥7mm), right axis deviation
  6c. LAE: notched P in II (duration ≥120ms) + terminal negative force in V1 >1mm×1ms
  6d. RAE: peaked P ≥2.5mm in II
  6e. Low voltage: QRS amplitude <5mm all limb leads AND <10mm all precordial leads

Step 7: U-WAVE ASSESSMENT
  7a. Are U-waves visible (positive deflection after T-wave)?
  7b. Amplitude >1mm or inverted → abnormal; note lead distribution
  7c. Prominent U-waves in lateral leads → hypokalemia; inverted U-waves → lateral ischemia context

Step 8: ASSEMBLY
  8a. Compile all morphology and repolarization findings
  8b. Assign STAT flag to: QTc ≥500ms + T-wave alternans, Brugada Type 1 (arrhythmic risk), critical hyperkalemia (sine wave morphology)
  8c. Emit cross_domain_hints per Section 4.5
  8d. Populate raw_feature_flags
```

### 4.6 Known Failure Modes

1. **Brugada Type 1 morphology in right bundle branch block**: RBBB can produce an rSR' pattern in V1 that visually resembles Brugada Type 1. The MR Agent may falsely flag Brugada in a patient with simple RBBB. The coved morphology (descending ST + inverted T) must be distinguished from the typical RSR' + S-wave of RBBB. CDS must verify by cross-referencing RRC's BBB classification.

2. **QTc inaccuracy during AFib**: QTc formulae assume regular RR intervals. During AFib, RR intervals are highly variable, making Bazett's formula unreliable (Bazett over-corrects at short RR, under-corrects at long RR). The MR Agent should note QTc reliability as "low" when AFib is detected in RRC output.

3. **LVH strain pattern classified as lateral ischemia**: The asymmetric ST depression + T-wave inversion in lateral leads (I/aVL/V5-V6) in LVH strain mimics lateral ischemia. The MR Agent should flag this as "strain pattern consistent with LVH" and emit the finding to CDS with a hint that IT should not classify this as primary ischemia if LVH voltage criteria are met.

4. **T-wave alternans false positive from noise**: In low signal quality leads, beat-to-beat T-wave amplitude variation may reflect noise rather than true TWA. The MR Agent must check `signal_quality_per_lead` — if the TWA-flagged lead has quality <0.7, confidence in TWA should be reduced to <0.5.

## 5. CDS Agent Design

### 5.1 Mission Statement

The CDS Agent is the final integration layer. It receives all three Phase 1 structured outputs plus selective raw features, applies dynamic RAG based on detected findings, resolves cross-domain conflicts and interactions, applies complex multi-criterion algorithms (Sgarbossa, Brugada/Vereckei), generates a differential diagnosis, assigns confidence scores, and produces the final clinical report.

The CDS Agent does NOT re-run the Phase 1 domain analyses from scratch. It trusts Phase 1 findings and either accepts, modifies, or overrides them based on cross-domain evidence. An override must always be documented in the reasoning chain with explicit evidence. The CDS Agent is the only agent that produces the final diagnosis — Phase 1 outputs are intermediate findings, not final diagnoses.

### 5.2 System Prompt Architecture

**Section 1 — Role and Constraints** (~300 tokens): Cross-domain synthesis role. Explicit constraint: trust Phase 1 findings unless cross-domain evidence contradicts them. Document all overrides. Clinical stakes: this output drives ER clinical decisions.

**Section 2 — Input Structure Description** (~300 tokens): Describes the exact JSON structure CDS will receive: `{ rrc: {...}, it: {...}, mr: {...}, raw_features: {...}, rag_context: "string" }`. Instructs the agent on how to parse and use each section.

**Section 3 — Cross_Domain_Hint Processing Protocol** (~400 tokens): For each hint_type in the vocabulary: what the CDS must do. Written as a lookup table: hint_type → response action.

**Section 4 — Cross-Domain Interaction Rules** (~500 tokens): The 12 known interactions from Node 2.1 Table 10.2, written as IF/THEN rules with explicit resolution steps.

**Section 5 — STAT Escalation Protocol** (~200 tokens): Any STAT flag from any Phase 1 agent is preserved in the final output unless CDS has cross-domain evidence to downgrade it. Downgrading a STAT flag requires explicit evidence and documentation. When in doubt: keep STAT.

**Section 6 — Output Schema** (~400 tokens).

**Total estimated system prompt: ~2,100 tokens** — within 4,000-token cap. The additional 1,900-token budget is available for dynamic RAG context loaded based on Phase 1 findings.

### 5.3 Raw Feature Selection Protocol

CDS always receives a **base feature set** (12 fields, ~150 tokens):
- `aggregate.heart_rate_ventricular_bpm`
- `aggregate.rhythm_regularity_coefficient`
- `aggregate.qtc_bazett_ms`
- `aggregate.qtc_fridericia_ms`
- `aggregate.av_ratio`
- `aggregate.t_wave_alternans_present`
- `patient_sex`
- `patient_age_years`
- `signal_quality_per_lead` (all 12 values)
- `total_beats_analyzed`

Beyond the base set, CDS receives any features named in Phase 1 `raw_feature_flags` arrays. Each Phase 1 agent names specific feature paths (e.g., `"beats[*].leads.V1.st_j60_mv"`, `"beats[*].leads.V1.r_amp_mv"`).

**Budget rule**: CDS raw features are capped at 2,000 tokens total (base + flagged). If Phase 1 flags exceed this budget, the orchestrator selects the flagged features in priority order: RRC flags first, then IT flags, then MR flags. This priority reflects the life-saving hierarchy (arrhythmia > ischemia > morphology).

**Implementation note for Node 2.3 (Orchestrator)**: The orchestrator must implement a feature extractor that takes a list of field path strings and produces a minimal JSON object with only those fields. This is a Node 2.3 responsibility.

### 5.4 Dynamic RAG Selection Protocol

After Phase 1 outputs are received, the orchestrator (Node 2.3) applies these trigger rules to select RAG chunks before sending to CDS:

| Trigger | RAG Chunk to Load | Token Estimate |
|---------|------------------|----------------|
| `LBBB_DETECTED` hint present | Sgarbossa criteria full text + Smith-modified formula | ~600 tokens |
| `WCT_AV_DISSOCIATION` or `WCT_NO_AV_DISSOCIATION` hint | Brugada algorithm (4 steps) + Vereckei aVR algorithm | ~800 tokens |
| `POSTERIOR_STEMI_SUSPECT` or `POSTERIOR_STEMI_MORPHOLOGY` hints both present | Posterior STEMI mirror-image criteria + V7-V9 recommendation + LCx vs RCA inference | ~400 tokens |
| `INFERIOR_STEMI_EXTENSIONS` hint | RV STEMI assessment (V1 STE criterion; V4R supplemental if available) + posterior coexistence rules + Complete AVB ischemic context | ~500 tokens |
| `PEAKED_T_TERRITORIAL_CHECK` hint | Hyperkalemia staging ECG criteria + territorial vs diffuse decision tree | ~400 tokens |
| `BRUGADA_MORPHOLOGY_CONFIRMED` hint | Brugada Type 1 diagnostic criteria + differential (RBBB, ARVC, hyperK pseudo-Brugada) | ~600 tokens |
| `CRITICAL_QTC_PROLONGATION` hint | TdP risk stratification table + short-long-short pause assessment + TdP treatment-altering findings | ~400 tokens |

### 5.5 Cross-Domain Resolution Protocol

For each of the 12 known interactions:

**1. LBBB masks STEMI → Sgarbossa**
- Trigger: `LBBB_DETECTED` hint from RRC + any ST findings in IT output
- Resolution: Apply Sgarbossa criteria to IT's `j_point_measurements{}`:
  - Concordant STE ≥1mm in any lead with LBBB direction = +5 points
  - Concordant STD ≥1mm in V1, V2, or V3 = +3 points
  - Smith-modified: ST/S ratio ≤ −0.25 in any lead = +2 points (compute: st_j60_mv / |s_amp_mv|)
  - Score ≥3 → STEMI-in-LBBB: STAT flag, highest confidence
  - Score 0-2 → No Sgarbossa positivity: document as "LBBB without STEMI criteria"

**2. LBBB makes Wellens unreliable**
- Trigger: `LBBB_DETECTED` + IT `wellens_pattern.present = true`
- Resolution: Downgrade Wellens to "cannot be assessed in presence of LBBB." Document in findings.

**3. LBBB makes de Winter unrecognizable**
- Trigger: `LBBB_DETECTED` + IT `de_winter_pattern.present = true`
- Resolution: Suppress de Winter finding. IT cannot assess de Winter morphology through LBBB. Document.

**4. Inferior STEMI → RV STEMI assessment**
- Trigger: IT `territory_mapping.primary_territory = "inferior"` + `INFERIOR_STEMI_EXTENSIONS` hint
- Resolution: Examine raw feature `beats[*].leads.V1.st_j60_mv` (primary criterion — V4R is not in the standard 12-lead feature schema; V1 STE ≥1mm is the standard substitute). V1 ST elevation ≥1mm in context of inferior STEMI = RV extension. Flag RV STEMI as co-diagnosis. Clinical implication: no nitroglycerin. If right-sided leads (V4R) are separately available, they take precedence over V1 for confirmation but are not required.

**5. Inferior STEMI → Posterior STEMI assessment**
- Trigger: IT `POSTERIOR_STEMI_SUSPECT` + MR `POSTERIOR_STEMI_MORPHOLOGY` both present
- Resolution: V1-V3 horizontal STD + R/S >1 in V1 + upright T in V1 = posterior STEMI. Determine culprit: III > II → RCA; II ≥ III → LCx. Recommend V7-V9 leads.

**6. Inferior STEMI + Complete AVB → Prognostic integration**
- Trigger: RRC `av_relationship.av_block_type = "complete"` + IT inferior STEMI finding
- Resolution: Ischemic AVB from RCA → typically transient (AV node ischemia), good prognosis with revascularization. Flag as ischemic AVB. Note: anterior STEMI + CHB = infra-nodal, permanent, poor prognosis.

**7. Hyperkalemia mimics Brugada**
- Trigger: MR `BRUGADA_MORPHOLOGY_CONFIRMED` + MR `peaked_diffuse_present = true`
- Resolution: Diffuse peaked T-waves + Brugada-like ST in V1-V2 may indicate hyperkalemia pseudo-Brugada. Check if QRS is widening (from RRC) and if T-wave peaks are diffuse vs isolated. If diffuse peaked T + wide QRS → hyperkalemia more likely. Recommend electrolytes.

**8. Hyperkalemia masks STEMI hyperacute T-waves**
- Trigger: MR `peaked_diffuse_present = true` + IT any territorial ST finding
- Resolution: Diffuse peaked T-waves overshadow territorial hyperacute T interpretation. CDS must resolve: if T-wave peaks are diffuse (all leads) → hyperkalemia primary; if T-wave peaks are confined to ischemic territory → STEMI T-waves. Check QRS width from RRC (hyperkalemia widens QRS).

**9. Hyperkalemia mimics Complete AVB**
- Trigger: RRC `rhythm.classification = "complete_avb"` AND MR `peaked_diffuse_present = true` AND RRC `p_waves.present = false`
- Resolution: Severe hyperkalemia → sinoventricular rhythm (absent P-waves + wide QRS) mimics CHB. CDS flags: "Complete AVB vs sinoventricular rhythm (severe hyperkalemia) — clinical correlation with K+ level essential." Both are STAT.

**10. VT prevents assessment of all other conditions**
- Trigger: RRC `wide_complex_tachycardia.present = true` AND CDS confirms VT via Brugada/Vereckei
- Resolution: All IT ischemia findings during VT are flagged "unreliable — assessed during wide complex tachycardia; reassess after cardioversion." All MR morphology findings except QTc are similarly flagged. QTc is not measurable during VT. Only RRC findings (rhythm itself) are reliable.

**11. AFib complicates QTc measurement**
- Trigger: RRC `rhythm.classification = "afib"` + MR `qtc_class != "normal"`
- Resolution: Downgrade QTc confidence to "low — AFib makes QTc unreliable." If QTc ≥500ms despite AFib → still flag, but note unreliability.

**12. Long QT + PVCs → TdP short-long-short**
- Trigger: MR `qtc_stat_flag = true` AND RRC `findings` contains PVC finding
- Resolution: Check RRC for short-long-short sequences (short coupling PVC → compensatory long pause → triggered beat on T-wave). If present: IMMINENT TdP risk — highest urgency STAT.

**13. AFib + LBBB → Wide complex tachycardia mimic (most dangerous VT differential trap)**
- Trigger: RRC `rhythm.classification = "afib"` AND RRC `conduction.bbb_type = "LBBB"`
- Resolution: AFib with LBBB produces an irregularly-irregular wide complex tachycardia at ventricular rates that can exceed 150 bpm — visually nearly identical to VT. The Brugada algorithm does NOT reliably distinguish this from VT. The key discriminating feature is rhythm regularity: true VT is regular (or near-regular); AFib+LBBB is irregularly-irregular. CDS must: (1) confirm rhythm irregularity from `rhythm_regularity_coefficient` — if >0.15 with wide QRS, AFib+LBBB is more likely than VT; (2) look for LBBB morphology consistency across beats (VT morphology may vary slightly; LBBB morphology is consistent); (3) flag as "wide complex tachycardia — AFib+LBBB vs VT cannot be excluded" if irregular. Both diagnoses are STAT (AFib with rapid ventricular response via aberrancy, or VT). Note: accessory pathway-mediated AFib (WPW+AFib) has similar presentation but is the most immediately dangerous — if delta waves were detected prior to the tachycardia, escalate to highest urgency.

```
Step 1: PARSE PHASE 1 INPUTS
  - Enumerate all findings from RRC, IT, MR

  3b. Brugada + Vereckei (if WCT + no confirmed AV dissociation)
```

### 5.8 Known Failure Modes

1. **Anchoring on first STAT finding**: If RRC flags VT with high confidence (0.85), CDS may anchor on VT and under-weight IT/MR findings. This is particularly dangerous because VT does not prevent the detection of hyperkalemia (which may be causing the VT). CDS must always process all Phase 1 findings regardless of a dominant STAT finding.

2. **Context overflow in complex coexisting conditions**: A patient with inferior STEMI + complete AVB + hyperkalemia generates 3 `INFERIOR_STEMI_EXTENSIONS` sub-checks + Sgarbossa (from LBBB if also present) + hyperkalemia pseudo-Brugada check. RAG may overflow the 2,500-token budget. The orchestrator must implement graceful RAG truncation with a warning flag in the CDS output.

3. **CDS overrides Phase 1 without sufficient evidence**: CDS might suppress a Phase 1 STAT finding without valid cross-domain evidence — effectively hiding a real finding. This is prevented by the "document all overrides" prompt instruction, but should be monitored in the regression test suite (any suppressed STAT finding from Phase 1 should appear in `suppressed_findings` with an explanation).

4. **Sgarbossa false positive at low ST changes**: The Sgarbossa score of ≥3 uses a 5-point threshold for concordant STE ≥1mm. In high signal-noise conditions, small measurement errors can push a borderline case to ≥3. The Smith-modified criterion (ST/S ratio) is more robust but requires clean S-wave measurements. CDS should note measurement uncertainty when flagging Sgarbossa positivity in noisy signals.

### 6.4 Confidence Scoring Convention

**Scale: 0.0 - 1.0 (continuous float)**

| Range | Clinical Class | Meaning | Clinical Implication |
|-------|----------------|---------|----------------------|
| 0.0 - 0.40 | `possible` | Finding cannot be excluded but evidence is weak | Include in differential only |
| 0.41 - 0.75 | `probable` | Evidence supports the finding but not conclusive | Flag for clinical attention; correlate with symptoms |
| 0.76 - 0.90 | `high confidence` | Strong evidence meets established criteria | Act on finding; clinical correlation recommended |
| 0.91 - 1.0 | `definite` | Pathognomonic finding or multiple independent criteria met | Immediate clinical action warranted |

**STAT flag threshold**: Any finding with confidence ≥0.50 AND clinical criticality (life-threatening if missed) receives STAT flag. The asymmetric threshold (0.50 for STAT vs 0.76 for normal findings) reflects the false-negative cost asymmetry for emergency diagnoses.

**Per-finding vs overall confidence**: Each finding has its own confidence score. The `confidence_overall` field in Phase 1 outputs is the mean confidence of all findings. The `overall_confidence` in the CDS output is the weighted mean (STAT findings weighted 2x). Do not use overall_confidence for clinical decisions — use per-finding confidence.

**CDS confidence inheritance**: When CDS accepts a Phase 1 finding without modification, it inherits the Phase 1 confidence. When CDS modifies a finding (via cross-domain integration), it assigns a new confidence based on the integration evidence. Cross-domain confirmation (two agents agree) typically increases confidence by 0.10-0.20. Contradiction resolution assigns confidence based on the dominant evidence.

## 7. Prompt Engineering Principles

### 7.1 DeepSeek R1 Reasoner-Specific Design Rules

1. **Provide reasoning structure, not reasoning answers**: The system prompt should specify WHAT to analyze (rate first, then rhythm, etc.) and HOW to analyze it (apply Brugada Step 1: check concordance) but never pre-state the expected conclusion. A prompt that says "if you see X, conclude Y" produces sycophantic confirmation rather than genuine reasoning.

2. **Embed decision trees for complex algorithms**: For multi-step algorithms (Brugada 4-step, Sgarbossa scoring), embed the complete decision tree in the prompt's criteria reference section. Do not rely on the reasoner to reconstruct these from memory — LLMs occasionally hallucinate criterion values. The prompt is the ground truth.

3. **Place output schema at the END of the system prompt**: DeepSeek R1 tends to respect end-of-prompt formatting instructions more consistently than mid-prompt ones. The output schema and the instruction "output valid JSON matching this schema exactly after completing your reasoning" should be the last section.

4. **Avoid negative space in scope definitions**: Saying "do not analyze T-waves" is less effective than "your scope is rhythm/rate/conduction." Agents should be scoped positively (what to do) rather than negatively (what not to do), with the negative scope as a secondary reminder.

5. **Short system prompts outperform long ones for reasoning quality**: Per Node 2.1 Section 6.2, system prompts should not exceed 3,000 tokens for Phase 1 agents. Each token added to the system prompt consumes reasoning capacity. Criteria embedded in the prompt should be the minimum required to avoid hallucination — not comprehensive textbook summaries.

6. **Do not instruct the reasoner to format its chain-of-thought**: DeepSeek R1's chain-of-thought is a private reasoning space. Attempting to format it (e.g., "reason step by step and label each step") interferes with the model's natural reasoning. Let the CoT be unstructured; enforce structure only on the final JSON output.

### 7.2 Medical Safety Asymmetry

For a STAT ECG system, false negatives (missed life-threatening findings) are far more costly than false positives (unnecessary alerts). This asymmetry must be explicitly encoded in each agent's system prompt.

**Recommended prompt language** (to be included in all four agents' Section 1 — Role and Scope):

> "This system is used in emergency medicine for ECG analysis of potentially life-threatening conditions. For any finding that could represent a STAT condition (STEMI, VT, complete heart block, severe hyperkalemia, or any other immediately life-threatening arrhythmia or ischemia pattern): flag it if you have ≥50% confidence. A missed STEMI that leads to delayed PCI is a preventable death. A false positive STEMI that leads to an unnecessary cath lab activation results in unnecessary but generally reversible intervention with real but limited procedural risk. When uncertain, flag."

For non-STAT findings, standard confidence thresholds apply.

**Wellens syndrome** deserves special mention: this condition presents with T-wave inversions in a *pain-free interval* after angina. The ECG looks "non-urgent" but represents critical LAD stenosis with imminent MI risk. The IT Agent must have explicit Wellens handling with STAT flag even though there is no ST elevation.

### 7.3 Output Parsing Reliability

Phase 1 agents must produce parseable JSON as the final output. Strategies to ensure this:

1. **Include a complete example in the schema section**: Not just field definitions — a literal JSON example with real-looking values. DeepSeek R1 imitates the format of demonstrated examples more reliably than it follows abstract schema definitions.

2. **Instruct: "after your reasoning, output ONLY the JSON object with no additional text"**: The word "ONLY" is important. Without it, agents sometimes produce commentary after the JSON block.

3. **Orchestrator-level fallback**: If the output contains a JSON block (detectable by `{` and `}` delimiters) but also text, the orchestrator should attempt to extract and parse the JSON block. If parsing fails, the finding should be flagged as `parse_error` and CDS should note that one Phase 1 agent's output is unavailable — CDS proceeds with two agents and a lower overall confidence.

1. **Concurrent API calls**: Support launching 3 simultaneous requests to DeepSeek API. Handle rate limiting with exponential backoff (not immediate failure).
2. **Temperature=0 parameter**: All calls must use `temperature=0`. The API integration layer must enforce this — it is not optional.
3. **Model pinning**: Pin to a specific DeepSeek R1 model version. Do not use `latest` — model regression risk (identified in RRC review). Model version must be configurable without code change.
4. **Reasoning token tracking**: Log both the visible output token count and the billed output token count per call, for cost monitoring and billing verification.
5. **Response streaming**: Phase 1 agents can stream responses (the orchestrator waits for completion before assembling Phase 2 input). Phase 2 CDS response can be streamed to the Report layer as it arrives.
6. **Fallback mode**: If DeepSeek API returns 503 or rate limit error after 3 retries: (a) flag the ECG as `llm_unavailable`, (b) apply the rule-based fallback covering the 12 documented cross-condition interactions (minimum: flag any STAT conditions detectable purely from feature thresholds without LLM reasoning), (c) queue for retry when API recovers.

### 9.3 Requirements for SDA-1 (Signal Pipeline)

Node 2.2 places these specific feature requirements on SDA-1 (primarily Node 1.5 — Feature Extraction):

1. **Per-beat per-lead fields**: All 11 per-lead fields defined in Section 1.1 must be computed for every beat and every lead. This requires the Feature Extraction node to process ECGdeli FPT columns into the derived schema — raw FPT columns alone are insufficient.
2. **ST measurement at J+60ms and J+80ms**: Both measurement points are required. J+60ms is the primary STEMI criterion measurement; J+80ms is the backup for high-rate cases where J+60ms is influenced by the late QRS.
3. **ST morphology classification**: The `st_morphology` enum (horizontal|upsloping|downsloping|concave|convex|saddle|coved) must be computed algorithmically by SDA-1. This requires a morphology classifier, not just amplitude measurement.
4. **T-wave symmetry score**: The `t_symmetry` field (0-1 float) is computed as: `min(Tpeak−Ton, Toff−Tpeak) / max(Tpeak−Ton, Toff−Tpeak)`. This produces 1.0 for a perfectly symmetric T-wave regardless of which limb is longer, and approaches 0 as asymmetry increases. Edge cases: set to null if Tpeak=Toff (degenerate/flat/inverted T-wave where ECGdeli cannot distinguish peak from offset); set to null for beats with beat_class=1 (LBBB) or beat_class=2 (RBBB) where T-wave shape is expected to be abnormal and should not trigger Wellens or hyperkalemia detection.
5. **RS interval per lead**: `rs_interval_ms` per the Brugada Step 2 criterion is defined as: `S_fiducial − QRSon_fiducial` (ms) — the interval from onset of QRS to the nadir of S wave. Both `QRSon` and `S` are columns in the ECGdeli FPT. This matches the Brugada algorithm's criterion ("RS interval >100ms in any precordial lead"). Note: "onset of R" is not a clean FPT value — QRSon is used instead as the unambiguous start of ventricular activation.

6. **P-wave duration per beat**: `p_wave_duration_ms[]` = `Poff − Pon` from ECGdeli FPT. Required for LAE detection (duration ≥120ms) in the RRC Agent (Step 2b) and MR Agent (Step 6c). Set to null if P-wave absent (`p_wave_present = false`).
6. **T-wave alternans detection**: The `t_wave_alternans_present` aggregate field requires a time-domain beat-to-beat TWA detection algorithm. Important limitation: the 1.9μV threshold from the Microvolt TWA literature applies to sustained recordings (≥2-3 minutes at stable rate). On short 12-lead strips (<60 seconds), the noise floor may exceed 1.9μV, generating false positives. **Implementation rule**: set `t_wave_alternans_present = null` and add a `t_wave_alternans_reliability = "low"` flag when `recording_duration_sec < 60`. Only assert `t_wave_alternans_present = true` on recordings ≥60 seconds. The CDS STAT flag for TWA should include the reliability flag in its output.
7. **Signal quality per lead**: The `signal_quality_per_lead` array (0-1 per lead) must be produced by Node 1.3 (Quality Assessment) and passed through to the feature object. This is used by agents to reduce confidence for low-quality leads.
8. **P-wave presence per beat**: `p_wave_present[]` per beat is derivable from ECGdeli FPT — a beat has a P-wave if Pon and Ppeak are non-null and within expected timing range.
