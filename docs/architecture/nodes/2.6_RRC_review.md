# RRC Adversarial Review — Node 2.6 DeepSeek API Integration (FD-FPRE)

**Reviewed by:** Research Review Committee (RRC)
**Document under review:** `docs/architecture/nodes/2.6_deepseek_api_integration.md`
**Review date:** 2026-03-28
**Verdict:** GATE CONDITIONAL

---

## Preamble

This review is conducted adversarially. Node 2.6 specifies the Python layer connecting the orchestrator to the DeepSeek API — a layer that handles PHI exclusion, clinical error propagation, cost metering, and audit trail generation. Failures here propagate upward into clinical decisions. The bar for correctness is correspondingly high.

Ten issues are raised. Four are CRITICAL (block implementation), four are REQUIRED (must be resolved before first production deployment), and two are RECOMMENDED.

---

## Issue 1 — Timeout Race Condition: `asyncio.wait_for()` vs `httpx.AsyncClient` Internal Timeout

**Severity: CRITICAL**

**Location:** Section 2.2, `call_agent()`, lines:

```python
async with httpx.AsyncClient(timeout=None) as client:
    ...
    response = await asyncio.wait_for(
        client.post(DEEPSEEK_API_URL, json=payload, headers=headers),
        timeout=timeout_s,
    )
```

**The flaw:**

`dispatch_phase1()` creates the client with `timeout=None`, which correctly disables httpx's internal timeout and cedes full control to `asyncio.wait_for()`. This part is sound.

However, the `call_agent()` signature accepts an externally-supplied `client: httpx.AsyncClient` — and the document makes no requirement on how that client must be configured. Any caller that passes a client constructed with a default timeout (httpx default is 5s connect / 5s read) will produce a race condition:

- If the httpx internal timeout fires first, it raises `httpx.ReadTimeout` (a subclass of `httpx.TimeoutException`, which is a subclass of `httpx.RequestError`).
- This is caught by the `except httpx.RequestError` branch and raises `AgentAPIError(0, "Network error: ...", latency)` — NOT `AgentTimeoutError`.
- The orchestrator's retry logic branches on error type. A timeout that masquerades as `AgentAPIError` will be handled as a network error, not as a timeout. The Node 2.3 contract is violated.

`dispatch_phase1()` correctly uses `timeout=None`, but since `call_agent()` is a public function the orchestrator is also directed to call directly (Section 2.4 note), the contract is incomplete. The spec must either (a) require that any client passed to `call_agent()` be constructed with `timeout=None`, or (b) make `call_agent()` assert this at entry, or (c) remove the `client` parameter and own client creation internally.

A secondary concern: `asyncio.wait_for()` cancels the coroutine on timeout, which cancels the underlying httpx request. Cancellation of an `httpx.AsyncClient` request mid-flight may leave the connection pool in a degraded state if the client is shared across concurrent calls. The document does not address this.

**Required action:** Specify that `call_agent()` must validate or enforce `client.timeout == httpx.Timeout(None)`, or document this as a hard precondition with a runtime assertion. Clarify connection pool state after task cancellation.

## Issue 2 — `dispatch_phase1()` Does Not Provide `as_completed()` Semantics

**Severity: CRITICAL**

**Location:** Section 2.4, `dispatch_phase1()`:

```python
results = {}
for agent_id, task in tasks.items():
    try:
        results[agent_id] = await task
    except ...:
        results[agent_id] = e
```

**The flaw:**

The three tasks are created concurrently (correct — `asyncio.create_task()` starts them all). But the result-collection loop `await task` iterates in insertion order: `rrc`, then `it`, then `mr`. This means:

- If `it` finishes first with a STEMI flag, the calling code does not see `it`'s result until `rrc`'s `await task` completes.
- If `rrc` is slow (near its 15s timeout), the STAT fast-path fires 15 seconds late regardless of how fast `it` responded.

The document acknowledges this in Section 2.4's note: "the orchestrator may call `call_agent()` directly with `asyncio.as_completed()` for STAT monitoring." But this note is insufficient: it presents a misleading helper (`dispatch_phase1()`) that appears to be the primary dispatch path, then footnotes that it lacks the timing property that matters most for ER clinical workflow. An ER clinician waiting for a STEMI alert should not wait for `rrc` to finish reasoning about rate-related criteria before `it`'s STEMI finding is surfaced.

Either `dispatch_phase1()` must be rewritten to use `asyncio.as_completed()` internally and emit results as they arrive, or the function must be removed and the document must provide a concrete `as_completed()` dispatch pattern for the orchestrator to implement. Leaving the STAT fast-path to the caller as a footnote is insufficient specification for a clinical-grade system.

**Required action:** Remove `dispatch_phase1()` as specified or replace it with an `as_completed()`-based generator. Provide an explicit STAT fast-path dispatch pattern in the spec.

## Issue 3 — `_extract_json()` Strategy 3 Is Fragile Under Reasoning Bleed

**Severity: CRITICAL**

**Location:** Section 2.3, `_extract_json()`:

```python
match = re.search(r"\{[\s\S]+\}", content)
```

**Two distinct flaws:**

**Flaw 3a — Reasoning bleed into `content`:**
DeepSeek R1 returns the thinking trace in `reasoning_content` and the final answer in `content`. However, empirically, the R1 model occasionally prefixes `content` with fragments of its reasoning before producing the JSON — particularly when prompted with a strict output schema. If `content` contains prose like:

```
Let me analyze the ST changes. The findings suggest... {"stat_flag": true, ...}
```

Strategy 1 (direct parse) fails. Strategy 2 (markdown fences) fails. Strategy 3 applies `r"\{[\s\S]+\}"`, which correctly finds `{...}` in this case. But if the prose itself contains a `{` character (e.g., "changes in {V1, V2, V3}"), the regex matches from that first `{` to the last `}` in the string — capturing both the prose fragment and the JSON, producing unparseable text.

**Flaw 3b — Greedy `[\s\S]+` on nested JSON:**
`r"\{[\s\S]+\}"` is greedy and matches from the first `{` to the LAST `}` in the string. In a well-formed response this is usually the outermost JSON object. But if there is any trailing text after the closing `}` of the JSON (e.g., a closing code fence not caught by strategy 2, or a trailing newline followed by a stray `}`), the regex extends past the JSON's closing brace and `json.loads()` fails. This falls through to `AgentSchemaError`, which is correct behavior — but the failure mode is non-obvious and hard to debug.

The fix is to use `re.search(r"\{[^{}]*(?:\{[^{}]*\}[^{}]*)*\}", content)` (balanced-brace matching) or to use `json.JSONDecoder().raw_decode()` which parses exactly one JSON value from a string and returns the position, allowing correct extraction regardless of surrounding text.

**Required action:** Replace strategy 3 with `json.JSONDecoder().raw_decode()` applied at each `{` position in `content`, returning the first successfully decoded object. Document the reasoning-bleed scenario explicitly.

## Issue 4 — PHI Validation Uses a Denylist; Should Use an Allowlist

**Severity: CRITICAL**

**Location:** Section 4.1, `validate_no_phi()`:

```python
PHI_FIELDS = {"patient_name", "patient_mrn", "patient_dob", "patient_address", "patient_ssn"}

def validate_no_phi(feature_obj: dict) -> None:
    found_phi = PHI_FIELDS.intersection(feature_obj.keys())
    if found_phi:
        raise ValueError(...)
```

**The flaw:**

This is a denylist. It blocks known bad field names. Any PHI field not on the list passes silently. SDA-1 is a separate module that produces the feature object — if it adds a field named `patient_identifier`, `subject_id`, `visit_id`, `patient_birth_year`, `encounter_id`, or any other quasi-identifier not enumerated in `PHI_FIELDS`, that field is transmitted to the DeepSeek API without error.

The document acknowledges `ALLOWED_PATIENT_FIELDS = {"patient_sex", "patient_age_years"}` but does not use it in the validation function. The safe design is to invert the check: define the complete set of permitted patient-linked fields (`ALLOWED_PATIENT_FIELDS`) and raise if any key in the feature object matches a `patient_*` or similar prefix pattern and is NOT in the allowlist.

The existing `ALLOWED_PATIENT_FIELDS` constant is defined but unused. This is not a documentation oversight — it is an implemented-but-not-wired safety control, which is worse than no control because it creates a false impression of safety.

**Required action:** Rewrite `validate_no_phi()` to use `ALLOWED_PATIENT_FIELDS` as an allowlist: reject any key matching a configurable set of sensitive prefixes (at minimum `patient_`, `subject_`, `visit_`, `encounter_`, `mrn`, `dob`, `ssn`, `name`) unless the key is in `ALLOWED_PATIENT_FIELDS`. Wire the existing constant.

## Issue 5 — Cost Estimates Are 3–5× Too Low for Complex Cases

**Severity: REQUIRED**

**Location:** Section 5.2, cost table:

```
RRC Agent: ~3,000 reasoning tokens → ~$0.009
IT Agent:  ~4,000 reasoning tokens → ~$0.011
...
Total per ECG: ~$0.046
```

**The flaw:**

DeepSeek R1 produces reasoning tokens proportional to problem complexity. The 3,000–5,000 reasoning token estimates are appropriate for straightforward cases (normal sinus rhythm, simple rate abnormality). For clinically complex cases — STEMI with atypical presentation, polymorphic VT, Brugada pattern, LBBB masking ischemia — empirical measurements show R1 generating 10,000–40,000 reasoning tokens per agent call. This is not a theoretical upper bound; it is routine behavior for hard diagnostic cases.

At $0.55/1M:
- 20,000 reasoning tokens per agent = $0.011 per agent vs. the ~$0.003 implied in the table
- 4 agents at 20,000 reasoning tokens each = $0.044 in reasoning alone, plus output and prompt tokens
- Total per complex ECG: ~$0.10–$0.15, not $0.046

The table's "Total per ECG: ~$0.046" figure will be cited by operations and clinical governance as the deployment cost basis. If it is 3–5× wrong for the cases that matter most (STAT cases — exactly the complex ones), the project's cost model is misleading for the scenarios it is designed to handle.

VT-2.1-03 is listed as a verification task, but its success criterion ("inspect `usage` response for `reasoning_tokens` field") does not require running against diagnostically complex ECGs. The verification task is incomplete.

**Required action:** (1) Update the cost table to show a range: typical case vs. complex case. (2) Add a `max_cost_per_ecg` circuit breaker to `_log_usage()` that alerts if any single ECG run exceeds $0.20. (3) Expand VT-2.1-03 to include at least 3 PTB-XL ECGs with confirmed complex diagnoses (STEMI, polymorphic VT, Brugada) and record actual reasoning token counts.

## Issue 6 — `_round_floats` Serializer Is Never Invoked

**Severity: REQUIRED**

**Location:** Section 3.2, `serialize_feature_object()`:

```python
def _round_floats(obj):
    if isinstance(obj, float):
        return round(obj, 4)
    raise TypeError(f"Not JSON serializable: {type(obj)}")

return json.dumps(feature_obj, separators=(",", ":"), default=_round_floats)
```

**The flaw:**

`json.dumps()` calls the `default` handler only for objects it cannot serialize natively. Python `float` is natively JSON-serializable — `json.dumps` converts it directly to a JSON number without calling `default`. Therefore `_round_floats` is never invoked for any float value.

The claimed 15–20% token reduction from float rounding does not occur. The full-precision floats (e.g., `0.12345678901234567`) are transmitted to the API. This is both a correctness issue (the spec states floats are rounded to 4 decimal places) and a cost issue (un-rounded floats in beat arrays expand token counts significantly).

The fix is to traverse the feature object recursively and round floats in-place before calling `json.dumps`, or to use a custom `json.JSONEncoder` subclass that overrides `iterencode`.

**Required action:** Replace `_round_floats` as a `default` handler with a recursive pre-processing step or a `JSONEncoder` subclass that intercepts float encoding. Add a unit test that verifies a float value `0.123456789` serializes as `0.1235` in the output string.

## Issue 7 — `_current_model_version` Global Is Unsafe Across Sessions and Concurrent Calls

**Severity: REQUIRED**

**Location:** Section 6.2:

```python
_current_model_version: str | None = None

def _check_model_version(response_model: str) -> None:
    global _current_model_version
    if _current_model_version is None:
        _current_model_version = response_model
    elif response_model != _current_model_version:
        pass  # log alert
```

**Two flaws:**

**Flaw 7a — No cross-session persistence:**
`_current_model_version` is a module-level variable. It is `None` on every application start. Every new session sets it from the first response received. If the DeepSeek API silently updates the model between application restarts (which is the exact scenario this check is intended to catch), the check resets and does not fire. A model version change that occurs between a Tuesday and Wednesday ER shift will never be detected. The expected model version must be persisted to the database and compared against the first response of each session.

**Flaw 7b — No async safety on concurrent Phase 1 calls:**
Three `call_agent()` coroutines run concurrently. All three may call `_check_model_version()` at nearly the same time after receiving their responses. In CPython, the GIL makes the read-modify-write of a simple global assignment effectively atomic for the `None` check. However, this is an implementation detail of CPython, not a language guarantee, and the code is documented as async-safe in the spec. More importantly, if two agents return different model versions (possible during a rolling API deployment), the second one to call `_check_model_version()` will see the first's version already set and log an alert — but only if they return in the right order. This is a TOCTOU problem.

**Required action:** (1) Persist the expected model version to the local database on first observation. (2) On session start, load the stored version and compare all responses against it rather than against an in-memory global. (3) If no stored version exists (first ever run), store it and log "model version baseline established."

## Issue 8 — Audit Record Destination Is Unspecified

**Severity: REQUIRED**

**Location:** Section 6.3:

> "This audit record is written to the local database — not to any external system."

**The flaw:**

For a clinical audit trail, "the local database" is not a specification. The document must answer:

- Which database, which table/collection, which schema?
- What is the write failure behavior? If the audit write fails, does the pipeline continue (the finding is surfaced to the clinician but not audited) or abort (the finding is suppressed to ensure auditability)?
- What is the retention period? Clinical audit records in most jurisdictions have mandatory minimum retention (typically 7–10 years for medical records).
- Who has read access? The audit record contains `ecg_id` linked to a clinical finding — this is effectively a medical record.
- Is the audit write synchronous or async? An async write that fails silently defeats the purpose.

These are not implementation details to be deferred — they are design requirements that must be specified before the audit layer is implemented. A post-hoc audit schema designed around existing code is a compliance risk.

**Required action:** Add a Section 6.4 specifying: database table name and schema, write failure behavior (fail-open vs. fail-closed), retention period requirement, access control requirement, and whether the write is synchronous within the pipeline execution or deferred.

## Issue 9 — No Connection Pooling; TLS Overhead Per ECG

**Severity: RECOMMENDED**

**Location:** Section 2.4, `dispatch_phase1()`:

```python
async with httpx.AsyncClient(timeout=None) as client:
    ...
```

**The issue:**

`dispatch_phase1()` creates and destroys an `httpx.AsyncClient` on every ECG analysis. Each client creation triggers TLS handshake to `api.deepseek.com` — approximately 100–300ms of additional latency per ECG, before the first token is sent.

For an ER system processing concurrent ECGs during a busy shift, this compounds: if 5 ECGs arrive within the same second (multi-bay monitoring scenario), 5 × 3 = 15 TLS handshakes are initiated simultaneously. Beyond the latency cost, this creates a burst connection pattern to the DeepSeek API that may trigger rate limiting at the connection level before the per-request 429 threshold.

The fix is an application-scoped `httpx.AsyncClient` created at startup, passed into `call_agent()` from above, and closed on graceful shutdown. `dispatch_phase1()` in its current form should not own client lifecycle.

**Required action:** Specify that a persistent `httpx.AsyncClient` must be created at application startup and injected. Document the startup/shutdown lifecycle. (Note: this also partially addresses Issue 1 — centralized client creation allows enforcing `timeout=None` in one place.)

## Issue 10 — No Local Stub Strategy for Integration Testing

**Severity: RECOMMENDED**

**Location:** Section 9.3:

```
DEEPSEEK_API_URL=<override>     # Optional — for API proxy / local testing
```

**The issue:**

The YouOwnECG project prohibits mocking (per project rules: "never mock the database or external data sources in tests — use real data or skip the test"). The document provides an override env var but no specification for what a local test server should return.

Without this, any test that exercises the `call_agent()` code path either (a) calls the real DeepSeek API (expensive, requires network, flaky in CI), or (b) cannot be run at all. Neither is acceptable for a clinical system that requires regression testing before every prompt change (Section 9.2).

The correct approach consistent with the project's no-mock rule is a **local replay server**: a lightweight HTTP server that receives the same request the real API would receive, and returns a pre-recorded real API response for a known input. This is not mocking — it is caching real responses and replaying them deterministically. The spec should define: the replay server interface contract, the response file format, and how to record new responses (by running against the real API and capturing output).

**Required action:** Add a Section 9.4 specifying a local replay proxy: its interface (same as DeepSeek API), the format for stored response fixtures (real API responses captured from VT runs), and instructions for recording new fixtures. Reference VT-2.6-01 and VT-2.6-02 as the initial fixture capture events.

## Required Actions Before Gate Passage

1. **[Critical — Issue 1]** Specify that any `httpx.AsyncClient` passed to `call_agent()` must have `timeout=httpx.Timeout(None)`. Add a runtime assertion at the start of `call_agent()`. Clarify connection pool behavior after `asyncio.wait_for()` cancellation.

2. **[Critical — Issue 2]** Remove or replace `dispatch_phase1()`. Provide an explicit `asyncio.as_completed()`-based dispatch pattern in the specification. The STAT fast-path timing guarantee must be built into the dispatch layer, not left to the caller.

3. **[Critical — Issue 3]** Replace `_extract_json()` strategy 3 with `json.JSONDecoder().raw_decode()` applied at each `{` offset. Document the reasoning-bleed scenario as a known DeepSeek R1 behavior.

4. **[Critical — Issue 4]** Rewrite `validate_no_phi()` to use `ALLOWED_PATIENT_FIELDS` as an allowlist. Add pattern-based detection for unrecognized quasi-identifier prefixes. Wire the existing constant that is currently dead code.

5. **[Required — Issue 5]** Update the cost table to show typical vs. complex case ranges. Add a per-ECG cost circuit breaker. Expand VT-2.1-03 to include complex diagnostic ECGs.

6. **[Required — Issue 6]** Fix float rounding by replacing the `default=` approach with recursive pre-processing or a `JSONEncoder` subclass. Add a unit test for float precision in serialized output.

7. **[Required — Issue 7]** Persist the model version baseline to the local database. Load it on session start and compare all call responses against the stored version. Remove the in-memory global as the primary version-tracking mechanism.

8. **[Required — Issue 8]** Add Section 6.4 specifying audit record destination, write failure mode, retention period, and access control. This must be specified before implementation begins.

## Verdict

**GATE CONDITIONAL**

Node 2.6 is well-structured and covers the right ground. The async architecture, error taxonomy, and PHI logging rules are sound. The cost model and reproducibility sections show careful thinking.

However, four CRITICAL issues must be resolved before implementation begins. Issues 1 and 2 affect the orchestrator's error-handling and STAT-path correctness. Issues 3 and 4 represent clinical safety risks: unreliable JSON extraction fails silently into `AgentSchemaError` (degrading mode, acceptable), but PHI denylist bypass transmits patient data to a third-party API — this is a HIPAA violation that cannot be caught in retrospect.

The gate opens when a revised document addresses all eight required actions above. Recommended actions (9 and 10) may be deferred to a follow-on implementation task with an explicit tracking item.
