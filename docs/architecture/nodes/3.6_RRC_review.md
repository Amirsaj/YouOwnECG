## Issues Found

### REQUIRED-1: P8 (clinical vs technical language) has no enforcement mechanism

**Location:** Node 3.6, Section 4, Principle P8

P8 states: "Clinical language for ER display; technical language for cardiologist display." The principle is correctly stated but the spec does not define HOW the language switch is implemented. There are two possible interpretations:

1. The LLM generates two versions of each finding at inference time (clinical + technical) — doubles latency and cost
2. The diagnostic result schema carries both a `clinical_label` and a `technical_label` field — generated once, displayed contextually

If interpretation 1 is implemented, the < 3 s first-token target (Node 3.5) becomes difficult to meet for both displays simultaneously.

**Required fix:** Specify that the `DiagnosticFinding` schema (Node 2.3) carries BOTH `clinical_summary` (plain language, ≤ 15 words) and `technical_detail` (clinical terminology, full precision). The agent generates both in a single pass as part of structured output. The ER dashboard renders `clinical_summary`; the cardiologist extension renders `technical_detail`.

### REQUIRED-2: "Show more" for findings beyond 7 has no specification of what triggers the threshold

**Location:** Node 3.6, Section 1.2 (Miller's Law application)

The spec states "never present more than 5–7 findings simultaneously; additional findings are accessible via 'show more.'" But it does not define what counts as a "finding" for this limit. If PVCs are listed per-beat (10 PVCs = 10 findings), the initial view is overwhelmed. If PVCs are grouped into one "Occasional PVCs (10 beats)" finding, the limit works correctly.

**Required fix:** The finding count limit applies to DISTINCT CONDITION findings, not beat-level events. Beat-level events are pre-aggregated by BeatSummary (Node 2.5) into single entries (e.g., "10 PVCs (16.7% of beats)"). The show-more threshold is applied to the `DiagnosticResult.findings` list, which contains one entry per condition (not one per beat).
