# RRC Adversarial Review — Node 1.4 Fiducial Detection (FD-FPRE)

**Reviewed by:** Research Review Committee (RRC)
**Document under review:** `docs/architecture/nodes/1.4_fiducial_detection.md`
**Review date:** 2026-03-29
**Verdict:** GATE CONDITIONAL

---

## Preamble

The FPT is the single most load-bearing artifact in the entire pipeline. Every interval measurement, every voltage measurement, every ST-elevation finding derives from these sample indices. A 20 ms error in QRSoff propagates directly to a 0.1 mV ST measurement error — enough to flip a STEMI diagnosis. Two issues are raised.

---

## Issue 1 — Safe Analysis Window Applied to Signal but FPT Index Offset Not Propagated to `FiducialTable`

**Severity: CRITICAL**

**Location:** Section 3, `run_ecgdeli()`:

```python
lead_signal_windowed = lead_signal[start:end]
fpt = qrs_detection(lead_signal_windowed, samplerate=500)
fpt = mastermind_delineate(lead_signal_windowed, 500, fpt, ...)
# Offset indices back to full-signal coordinates
fpt_full = fpt.copy()
for col in range(12):
    mask = fpt_full[:, col] != -1
    fpt_full[mask, col] += start
fpts[lead_name] = fpt_full
```

**The flaw:**

The index offset restoration (`+= start`) is applied to the FPT inside `run_ecgdeli()`. This is correct — the FPT indices now reference the full `preprocessed_signal` sample grid. However, `FiducialTable` stores `rr_intervals_ms` and `mean_heart_rate_bpm` computed from the reference lead FPT.

The RR interval computation is not shown, but the document states:
> "All beat timing (RR intervals, heart rate, rhythm analysis) is derived from the reference lead FPT."

If `rr_intervals_ms` is computed from the offset-corrected FPT (after `+= start`), it is correct — RR intervals are differences between R-peak indices, so the offset cancels out. But if computed from the windowed FPT before offset correction (e.g., inside `qrs_detection()` return processing), the intervals are still correct (differences cancel offset). The document does not show where `rr_intervals_ms` is computed, creating ambiguity.

The real problem is more subtle: `FiducialTable.n_beats` counts beats detected within the safe analysis window only. But Node 1.5 and the orchestrator need to know how many total beats were in the full recording, not just the analyzed window. A 10-second recording at 60 bpm has ~10 beats; the safe analysis window (3 seconds excluded from each end) leaves only ~4 beats. `n_beats = 4` is technically correct but misleading — the orchestrator reports "4 beats analyzed" when the recording contained 10.

**Required action:** (1) Add `total_recording_beats: int` to `FiducialTable` — beats detected in the full signal (including excluded edge zones) for reporting. (2) Clarify that `n_beats` is "beats within safe analysis window." (3) Show explicitly where `rr_intervals_ms` is computed (from the offset-corrected full FPT).

## Issue 2 — AFib Post-Processing ("set P-wave columns to -1") Is Listed as Mitigation but Not Specified

**Severity: REQUIRED**

**Location:** Section 6, failure modes table:

> "AFib — no P waves: Check `afib_suspected`; set all P-wave columns to -1 in post-processing"

**The flaw:**

This mitigation is stated in a table row but there is no code or function specifying where, when, or how this post-processing happens. ECGdeli's `condition_detection` may set `afib_suspected` internally, but the document does not show how this flag reaches `FiducialTable.afib_suspected` or how it triggers P-wave column zeroing.

In practice, if AFib is missed by condition detection (a known failure mode for coarse f-waves), ECGdeli may populate P-wave columns with f-wave peaks. The IT agent's AFib detection (Node 2.2) relies partly on the absence of organized P waves in the FPT — if spurious P-wave fiducials are present, the agent receives contradictory evidence.

The mitigation must be an explicit post-processing step with specified logic:
- If `afib_suspected = True`: set `fpt[:, 0:3] = -1` (Pon, Ppeak, Poff) for all beats
- If `rhythm_regularity_cv > 0.2` (highly irregular): warn that P-wave reliability is reduced
- This step must happen before `FiducialTable` is constructed

**Required action:** Add an explicit `_apply_condition_corrections(fpts, condition_flags)` function that performs the AFib P-wave zeroing and any other condition-specific corrections. Call it before constructing `FiducialTable`. Document that this is a mandatory post-processing step, not an optional mitigation.

## Verdict

**GATE CONDITIONAL**

Node 1.4 is thorough. The wave genesis explanation, ECGdeli phase-by-phase analysis, multi-lead alignment protocol, and validation framework are all well-specified. The failure mode catalog is valuable. The CRITICAL issue is a metadata gap (missing total beat count) that affects reporting accuracy; the REQUIRED issue is a code-path gap that could let spurious AFib P-wave fiducials reach the agent layer. Both are small fixes. The gate opens when they are addressed.
