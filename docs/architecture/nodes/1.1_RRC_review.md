## Preamble

Node 1.1 is the first node in the sacred pipeline. Any error introduced here — wrong units, wrong lead order, wrong sampling rate, uncleaned PHI — propagates through every downstream node with no recovery path. The RRC reviews with corresponding severity.

Three issues are raised. One is CRITICAL (blocks implementation), one is REQUIRED, and one is RECOMMENDED.

---

## Issue 1 — `ECGRecord.signal` Shape Inconsistency with ECGdeli Input Contract

**Severity: CRITICAL**

**Location:** Section 4.2, `ECGRecord`:

```python
signal: np.ndarray  # shape: (n_samples, 12), dtype: float64, units: mV
```

**And Section 7:**

> "ECGdeli also expects `(n_samples, n_leads)` layout."

**The flaw:**

This claim must be verified against the actual ECGdeli Python API, not assumed. ECGdeli is a MATLAB-origin library ported to Python. MATLAB convention is column-major with signals as rows (shape `(n_leads, n_samples)`). Many Python wrappers of MATLAB signal processing tools default to the MATLAB convention.

If ECGdeli expects `(n_leads, n_samples)` and the ingestion layer produces `(n_samples, n_leads)`, the fiducial detector will silently transpose time and lead axes — treating each time point as a "lead" and each lead as a "time series". The ECGdeli output (FPT table) would be nonsense. This would not raise an exception — it would produce plausible-looking but clinically wrong fiducial positions.

The document states the shape "matches numpy broadcasting conventions" and "ECGdeli also expects this layout" — but provides no citation or verification. This is an unverified assumption about an external library's interface contract.

**Required action:** Verify the exact input shape expected by `ecgdeli`'s detection function by reading the ECGdeli source or documentation. If ECGdeli expects `(n_leads, n_samples)`, document the transposition step at the Node 1.4 interface boundary. Do not assume — cite.

## Issue 2 — EDF PHI Stripping Is Mentioned but Not Specified

**Severity: REQUIRED**

**Location:** Section 3.2:

> "The local patient ID and recording ID fields in the EDF header may contain patient name, date of birth, and hospital ID in plaintext. The ingestion layer must strip these fields before any further processing — they must never enter the internal representation or be logged."

**The flaw:**

The requirement is correctly stated but the mechanism is not specified. The document says "must strip" without specifying:

1. **When**: Is stripping done before `ECGRecord` construction, or does `ECGRecord` include a `raw_header` field that gets cleared?
2. **What exactly**: The EDF `local_patient_id` field is 80 characters of free-text. Does the ingestion layer set it to an empty string, replace it with the UUID `ecg_id`, or parse it and extract only the non-PHI subfields?
3. **Logging**: If the EDF file path contains a patient name (e.g., `SMITH_JOHN_2026-03-29.edf`), does the ingestion layer log the filepath? If so, PHI appears in logs.
4. **Verification**: How does the pipeline verify that stripping worked? A unit test that checks `ECGRecord` has no PHI fields is easy to specify.

For a system with HIPAA obligations, "must strip" is not sufficient — the stripping procedure must be specified precisely. Vague requirements produce gaps at implementation time.

**Required action:** Add a Section 4.6 (or expand Section 3.2) specifying: (a) the exact EDF header fields to strip, (b) what value to replace them with, (c) whether the source filepath is logged (and if so, how to sanitize it), (d) a unit test requirement that verifies no PHI fields are present in the constructed `ECGRecord`.

## Issue 3 — Lead Augmentation Applied Unconditionally; Computed Leads Are Clinically Distinct
**Severity: RECOMMENDED**

**Location:** Section 4.4:

> "If I and II are present: compute III, aVR, aVL, aVF. Mark computed leads in ECGRecord."

**The issue:**

Computing augmented leads from I and II is mathematically exact (it is how the AHA defines them). However, some downstream clinical applications distinguish between directly measured leads and derived leads:

- The MR agent's T-wave morphology analysis in augmented leads (aVR, aVL) has different clinical significance when the leads are measured vs. derived. Specifically, subtle ST-T changes in aVL that are used for high lateral STEMI detection must be from the measured lead — a derived aVL introduces no additional error but should be flagged so the agent can weight its confidence accordingly.
- For research/validation, comparing measured vs. derived leads quantifies the accuracy of the measuring equipment (deviations from the theoretical relationship indicate electrode impedance mismatch or measurement noise).

The document marks computed leads in provenance (`augmented_leads` field) — this is correct. The recommendation is to ensure that `ECGRecord` exposes this information in a way that is explicitly accessible to downstream consumers, not just stored as a free-form provenance string.

**Recommended action:** Add `computed_leads: list[str]` as a typed field on `ECGRecord` (not buried in a provenance string). Downstream agents can check `"aVL" in ecg_record.computed_leads` to adjust confidence weighting if needed.

## Verdict

**GATE CONDITIONAL**

Node 1.1 is thorough and well-grounded. The first-principles coverage (ion channels → lead systems → ADC → calibration) is complete. The WFDB loading specification is correct and matches the actual PTB-XL format. The `ECGRecord` design is sound — typed, format-agnostic, PHI-limited by construction.

The CRITICAL issue (ECGdeli shape) must be resolved before gate passage — an unverified assumption about an external library's input convention is exactly the kind of oversight that produces silent, hard-to-debug errors in clinical pipelines.

The gate opens when Issues 1 and 2 are addressed.
