## Executive Summary

Node 2.3 is structurally sound in its high-level architecture and its handling of the STAT fast-path timing problem. The core design decisions — binary STAT flag monitoring, provisional-then-confirmed alert lifecycle, graceful degradation table — are clinically defensible. However, the document has six issues that must be resolved before implementation begins: two are CRITICAL (will cause system failure or incorrect clinical data), two are REQUIRED (will reduce reliability in production), and two are RECOMMENDED improvements. Additionally, there are four untested edge cases of clinical significance.

## Issue 1 — Concurrency Model Correctness

**Rating: REQUIRED**

### 1.1 `asyncio.as_completed()` is correct in principle, but the document does not specify how blocking I/O is handled

Section 2.2 and 4.3 specify `asyncio.as_completed()` semantics for Phase 1 collection. This is the correct approach for concurrent API calls. However, the document explicitly defers "Python framework / async library implementation details" to the implementation phase. This creates a gap: if the DeepSeek API call implementation uses a synchronous HTTP client (e.g., `requests`) wrapped in `asyncio.to_thread()`, rather than a native async client (e.g., `httpx` with `async with`), the event loop will not be fully non-blocking and STAT fast-path processing for arriving agents will be delayed.

**Specific risk:** If Agent A finishes and the orchestrator's STAT check callback is blocked waiting for Agent B's synchronous HTTP call to release the thread pool, the STAT alert for Agent A's output is delayed until the thread pool slot is freed. This can add hundreds of milliseconds to the fast-path fire time — small but measurable, and the document claims tight latency targets (2.5–4s for STAT).

**Required action:** Add a requirement to Section 11.1 (Requirements on Node 2.6): "The `call_agent()` function MUST be implemented using a native async HTTP client (e.g., httpx AsyncClient). Synchronous HTTP clients wrapped in `asyncio.to_thread()` are not acceptable for Phase 1 concurrent dispatch."

### 1.2 Race condition in STAT fast-path: two agents completing near-simultaneously

The `check_stat_fast_path()` function in Section 3.1 is called per-agent as each output arrives. If RRC and IT both complete within milliseconds of each other (common under low API load), both may set `stat_flag = true`. The document does not specify whether both STAT alerts fire, or only the first.

**The gap:** If both fire, the UI receives two `STAT_PROVISIONAL` events for the same ECG within milliseconds. The document says the `STAT_CONFIRMED` update arrives later — but what is the alert update protocol when two provisional alerts were fired? Section 3.3 describes a single-alert update lifecycle. There is no `alerts[]` array — just a single `stat_alert` object in the report (Section 7.1). This implies either (a) only the first alert fires and the second is swallowed, which would suppress a legitimate second STAT condition, or (b) both fire and the UI receives two separate provisional alerts that both need individual `STAT_CONFIRMED` updates.

Neither behavior is specified. This must be resolved explicitly.

**Required action:** Define whether the STAT fast-path is "first-one-wins" or "all-fire." If "all-fire," define how the UI receives and reconciles multiple provisional alerts for the same ECG. If "first-one-wins," explicitly state that subsequent Phase 1 STAT conditions are included in the `STAT_CONFIRMED` update (not silently dropped). The validation plan must add a test for this case (see Issue 10).

## Issue 2 — STAT Fast-Path Safety: The Downgrade Problem

**Rating: CRITICAL**

Section 3.3 defines `STAT_DOWNGRADED` as the update status when CDS resolves a STAT finding as a known mimic. The document states this is a defined outcome and "the UI must clearly communicate the downgrade reason to the clinician."

**This is not safe as specified.** The document places the entire downgrade burden on the UI with no constraint on what CDS is permitted to downgrade. The following scenario is not handled:

> IT Agent fires STAT for STEMI (anterior territory, high confidence). Clinician is notified within 3 seconds. Cath lab activation protocol is triggered by hospital workflow. CDS then issues `STAT_DOWNGRADED` because LBBB is present and Sgarbossa criteria are not met. The STEMI diagnosis is withdrawn. The cath lab stands down.

This is clinically acceptable if Sgarbossa evaluation is rigorous. But:

1. **The downgrade is still a clinical act.** The orchestrator executes the downgrade mechanically based on a binary CDS field. There is no human review step between CDS issuing `STAT_DOWNGRADED` and the UI updating the alert. In the STEMI-to-mimic scenario, this means an AI system is autonomously retracting a cardiac catheterization alert with no clinician confirmation that the downgrade reasoning is sound.

2. **The document acknowledges this problem only partially.** Section 3.4 marks Brugada Type 1 and QTc ≥500ms as "Required" CDS confirmation before acting — but the STAT alert has already fired and the fast-path fired **before** CDS ran. The "Required" column in Table 3.4 only describes CDS's role in adding context, not in preventing a false alert from propagating.

3. **False alarm cost in ER.** The medical literature is unambiguous that alert fatigue in ED settings directly reduces response to subsequent alerts. A STAT alert that fires and is then electronically downgraded without explanation teaches clinicians to dismiss provisional alerts. If this happens 3 times for a single clinician in a shift (VT mimic, Brugada mimic, LBBB-confounded STEMI), the fourth STAT alert for a real VF may be ignored.

**Required actions:**

(a) Add a clinical governance note to Section 3.3: `STAT_DOWNGRADED` is a high-stakes state transition. The document must specify whether a human confirmation step is required before the UI executes the downgrade visual update, or explicitly accept the autonomous downgrade with a documented clinical risk statement.

(b) Define a minimum CDS confidence threshold for downgrade. A CDS `STAT_DOWNGRADED` at confidence 0.51 should not silently retract a Phase 1 STAT alert at confidence 0.95. The downgrade should require CDS confidence ≥ 0.80 for the mimic diagnosis, otherwise the UI should display `STAT_UNRESOLVED` ("Conflicting assessments — manual review required") rather than withdrawing the alert.

(c) Add to Section 11.3 (Requirements on SDA-3): The UI must display `STAT_DOWNGRADED` with the original STAT finding and the downgrade reason displayed side-by-side. It must not simply remove the alert or change it to a normal finding without visual distinction. The clinician must explicitly acknowledge the downgrade.

## Issue 3 — Partial Failure Logic: Immediate Retry on Timeout

**Rating: REQUIRED**

Section 4.2 states: "API error (5xx): Retry once (immediate, no backoff)."

Section 11.1 requirement 4 states: "No automatic retries on timeout — timeout is orchestrator-managed."

These two are consistent. However, the failure mode table mixes timeouts and API errors in a way that creates an ambiguity in implementation:

The table row for `Agent timeout (>15s)` says "Mark agent as `TIMED_OUT`; use empty output." No retry. Correct.

The table row for `API error (5xx)` says "Retry once (immediate, no backoff)."

**The problem:** A 5xx response from DeepSeek after 14.9 seconds of processing is functionally identical to a timeout — the server gave up. An immediate retry will send the same request to the same overloaded server and has a high probability of also returning 5xx (or timing out at 15s). The document does not distinguish between:
- A fast 5xx (e.g., 500ms — server rejected the request immediately, likely a transient error, immediate retry is reasonable)
- A slow 5xx (e.g., 14s of processing then a 503 — server timed out internally, immediate retry will likely also fail or timeout)

**No backoff** is explicitly specified, which compounds this. Under API degradation (the scenario where retries matter most), immediate-no-backoff retry adds 15 seconds to the Phase 1 latency for the retried agent (original attempt + retry timeout), pushing the pipeline past the 36-second absolute hard limit if two agents fail this way.

**Required action:** Add to Section 4.2 (or Section 11.1 requirements on Node 2.6): "For API errors (5xx) received after >5 seconds of processing time, the orchestrator should treat the failure as equivalent to a timeout (no retry). Immediate retry is only applied for fast failures (5xx received within <2 seconds of the API call)." This requires Node 2.6 to expose the response latency in `AgentAPIError`.

## Issue 4 — CDS Input Assembly: Feature Path Lookup Semantics Are Undefined

**Rating: CRITICAL**

Section 5.2 states: "Orchestrator collects all `raw_feature_flags` arrays from all Phase 1 findings, deduplicates feature paths, looks up each feature path in the feature object and adds the value."

Node 2.2 Section 2.4 shows `raw_feature_flags` contains feature path strings. The Node 2.2 feature schema uses a nested structure: `beats[N].leads["V2"].st_j60_mv`. A path like `beats[*].leads.V2.st_j60_mv` is a glob, not a direct key lookup.

**The undefined behaviors:**

(a) **If the feature path is `beats[*].leads.V2.st_j60_mv`** — does the orchestrator extract all beats' V2 J-point values as an array? A single representative value (mean)? The first beat? This is completely undefined. The token cost of all beats' V2 ST values for a 12-beat ECG is 12 floats — trivial. But for all leads' all beats' all ST fields, it compounds.

(b) **What is the actual path format?** Node 2.2 does not specify the path format for `raw_feature_flags`. The schema shows `"feature_path strings"` as the type but gives no example paths and no specification of whether they use dot notation, JSONPath (`$.beats[*].leads.V2.st_j60_mv`), or a custom format.

(c) **Lookup failure handling:** If a path in `raw_feature_flags` does not resolve to a valid key in the feature object (e.g., because an agent emitted a hallucinated path), the orchestrator silently drops it or crashes. Neither behavior is specified.

(d) **Token cost of array paths:** If `raw_feature_flags` contains per-lead per-beat fields (which is likely for ischemia findings — the IT Agent needs raw ST values), the resolved JSON could easily be 500-1,000 tokens per flagged field across 12 beats × 12 leads. The 2,000-token budget for raw features could be consumed by a single feature path pointing to a per-beat-per-lead array. The priority truncation logic in Section 5.2 does not account for the difference in token cost between a scalar field (1 token) and an array path (potentially 144 values).

**Required actions:**

(a) Define the `raw_feature_flags` path format explicitly. Specify whether paths use JSONPath, dot notation, or a custom format. Add this specification to Node 2.2 Section 2.4 and to Node 2.3 Section 5.2.

(b) Specify the resolution semantics for array paths: when a path resolves to an array, define what is included in the CDS raw feature payload — the full array, a statistical summary, or a trimmed version.

(c) Add a lookup failure handling rule: if a path does not resolve, log it and continue (do not crash); add the unresolved path to `raw_feature_fields_dropped`.

(d) Add per-path token estimation to the budget enforcement logic — scalar paths and array paths must be budgeted differently.

## Issue 5 — Token Budget: 1,200 Tokens per Phase 1 Output Is Likely Too Low

**Rating: REQUIRED**

Section 6.2 estimates Phase 1 structured outputs at "3 agents × ~1,200 tokens = ~3,600 tokens" for the CDS input.

The Node 2.2 RRC output schema (Section 2.4) contains:
- Top-level fields: `agent_id`, `schema_version`, `processing_time_ms`, `confidence_overall`
- `rhythm` object: 3 fields
- `rate` object: 3 fields
- `p_waves` object: 5 fields
- `av_relationship` object: 6 fields
- `conduction` object: 11 fields
- `wide_complex_tachycardia` object: 7 fields
- `findings[]` array: each finding has `finding_id` (UUID, 36 chars), `category`, `description` (free text), `evidence` (leads array + measurements dict + criteria_met array), `confidence`, `stat_flag`, `stat_reason`, `cross_domain_hint` (with payload), `raw_feature_flags`

For a complex ECG (e.g., VT with multiple findings), the `findings` array alone can contain 3-5 findings, each with a `description` string (30-50 words), `criteria_met` list (3-5 items), and `evidence.measurements` dict (5-10 key-value pairs). A single finding with full evidence can reach 150-250 tokens.

**A realistic RRC output for a complex case (VT + complete AVB + fascicular block): 2,500-3,500 tokens.** Three agents at this size yields 7,500-10,500 tokens for Phase 1 outputs alone — putting the CDS input at 12,000-15,000 tokens, not 9,000.

This is still within the 64K context window and is not a hard failure. But:

1. The 1,200 token estimate in Section 6.2 is optimistic by 2-3x for complex cases, and is likely accurate only for simple ECGs with 1-2 findings. Token cost projections (for API cost estimation) will be significantly underestimated.

2. More importantly: combined with the raw feature budget overflow scenario (Issue 4), the CDS context could reach 15,000-18,000 tokens for a complex case with many flagged features. This must be empirically measured (VT-2.2-01 is already flagged, but the VT should also cover Phase 1 output size, not just feature object size).

**Required action:** Add VT-2.3-01: Measure actual Phase 1 output token size across 10 PTB-XL ECGs spanning normal, VT, STEMI, and LBBB+STEMI cases. Update Section 6.2 with empirical range. If P95 Phase 1 output per agent exceeds 2,000 tokens, add a Phase 1 output token cap per agent (analogous to the raw feature budget cap) and define truncation rules (which fields to drop first — evidence details before diagnosis fields, details before STAT findings).

## Issue 6 — Report Finding Ordering: Flat Confidence Thresholds Are Clinically Unsound

**Rating: RECOMMENDED**

Section 7.2 defines priority ordering with flat confidence thresholds: ≥0.8 = "Critical," 0.5–0.8 = "Significant." This applies across all finding types uniformly.

**The problem:** A CDS confidence of 0.8 for "Isolated LVH voltage criteria — borderline" and a CDS confidence of 0.8 for "De Winter T-wave pattern suggesting proximal LAD occlusion" are not clinically equivalent. Under the current scheme, both appear in the "Critical" tier and are ordered only by confidence descending within that tier. The de Winter finding (minutes-to-cath urgency) may sort below the LVH finding if the LVH confidence is 0.85 and de Winter is 0.80.

Section 7.2 does define a secondary ordering by CDS category: "ischemia > rhythm > conduction > morphology > structural > metabolic > normal variants." This partially addresses the problem — de Winter would be in "ischemia" and LVH in "structural," so ischemia wins the tiebreak. However:

1. The category ordering only applies **within a confidence tier**. If two findings are in different tiers (LVH at 0.81 = Critical, de Winter at 0.79 = Significant), the LVH finding appears first regardless of category.

2. The 0.80 threshold boundary is an arbitrary cutoff applied identically to all clinical categories. A better design would use category-specific thresholds or a pre-computed clinical urgency score.

**Recommended action:** Add a `clinical_urgency_rank` field to each finding in the CDS output schema (defined in Node 2.2 Section 5.x). CDS assigns this rank (1=most urgent, N=least urgent) based on its clinical reasoning, not on confidence alone. The orchestrator uses `clinical_urgency_rank` as the primary sort key, with confidence as a tiebreaker. This delegates clinical priority ordering to the agent that understands clinical context (CDS), rather than to the orchestrator that is explicitly prohibited from clinical reasoning.

## Issue 7 — Idempotency Claim Is False Under Retry

**Rating: REQUIRED**

Section 10 states the orchestrator is "stateless" and each pipeline run is independent.

This is true for the orchestrator's internal state. However, the pipeline is **not idempotent with respect to the UI** because of the STAT fast-path.

**Scenario:** A pipeline run fires a STAT alert (Phase 1 complete) and then CDS times out. The caller (SDA-1→SDA-2 interface) decides to retry the pipeline for the same ECG. The second run dispatches Phase 1 agents again. If any Phase 1 agent again returns `stat_flag = true` (likely — temperature=0, same input), the STAT fast-path fires **again**. The UI receives a second `STAT_PROVISIONAL` for the same `ecg_id`.

The document does not specify whether:
- The retry uses the same `ecg_id` (which would cause the second STAT alert to collide with the first)
- The retry uses a new `pipeline_run_id` but the same `ecg_id` — and the UI must deduplicate by `ecg_id`
- The retry is prohibited if a STAT alert has already been fired for this `ecg_id`

Section 9.2 says "the system should queue the ECG for retry when the API recovers" — this implies retries are expected. But the STAT alert deduplication contract between the orchestrator and the UI is never defined.

**Required action:** Add to Section 10: "If a STAT alert was fired in a prior pipeline run for the same `ecg_id`, the orchestrator must suppress duplicate STAT alerts on retry. The mechanism for this is: (a) the caller passes a `prior_stat_fired: bool` flag in the pipeline invocation, and the orchestrator skips the fast-path on retry; OR (b) the orchestrator queries a short-lived deduplication store keyed by `ecg_id` before firing a STAT alert." Option (a) is simpler and keeps the orchestrator stateless.

## Issue 8 — Missing Edge Cases

### 8a — `recording_duration_sec = 0` passes Phase 0 validation

**Rating: REQUIRED**

Phase 0 validation (Section 2.1) checks for field presence but not semantic validity. Node 2.3 Section 11.2 requirement 3 says `recording_duration_sec` is "required for TWA reliability check" — but only as a presence requirement. A value of 0.0 would pass presence validation.

If `recording_duration_sec = 0`, then:
- Node 2.2 Section 1.1 specifies `t_wave_symmetry` requires `recording_duration_sec ≥60` for reliable TWA detection. The agents receive this field and must handle it — but whether a 0-duration recording causes a division-by-zero in upstream feature computation (SDA-1) or just produces garbage values is unspecified.
- More dangerously: the MR Agent prompt includes the TWA reliability rule in its criteria. If `recording_duration_sec = 0`, the agent should flag TWA as unreliable — but this requires the agent to check this field. If the prompt does not explicitly handle 0.0 as distinct from a missing value, the agent may produce a false TWA finding on garbage data.

**Required action:** Add to Phase 0 validation: `recording_duration_sec` must be `> 0`. If `= 0`, return `PIPELINE_FAILED` with specific error. Add to Section 11.2 requirement 3: the value must be `> 0`, not merely present.

### 8b — All Phase 1 agents return empty `findings` arrays

**Rating: RECOMMENDED**

This is the normal ECG case. The document handles it implicitly (no STAT, no CDS hints, report assembles with empty findings). But two behaviors are unspecified:

1. Does CDS run on an empty Phase 1 output? It should — a normal ECG still requires CDS to synthesize "no significant findings" as a positive assertion, not as an absence. Otherwise the CDS output for a normal ECG is undefined.

2. What does the report look like for a normal ECG? Section 7.2 defines priority ordering of findings — but if there are no findings, what does `synthesis` contain in the report? The UI (SDA-3) must be able to render the affirmative normal conclusion.

**Recommended action:** Add a test case ORC-11: Normal ECG (all findings arrays empty). Expected: CDS still runs; CDS output includes an explicit `no_significant_findings: true` flag or equivalent; report renders a normal conclusion.

### 8c — CDS outputs a STAT finding that no Phase 1 agent flagged

**Rating: REQUIRED**

Section 3.5 identifies cross-domain STAT conditions (left main stenosis, AFib+WPW, borderline VT) that cannot be detected from a single Phase 1 agent alone. For these, the fast-path does not fire. CDS is expected to synthesize a new STAT finding.

When CDS issues a new STAT finding (`stat_flag = true` in CDS output, not present in any Phase 1 finding), the orchestrator's Section 7.1 report structure has `stat_alert.fired = false` (fast-path never fired) and then must update to `fired = true` with `status = "STAT_CONFIRMED"` from CDS output. But:

1. No alert was sent to the UI during Phase 1. The first alert is the CDS output — which arrives 6–10 seconds after ECG submission, not 2.5–4 seconds.
2. The `STAT_PROVISIONAL` → `STAT_CONFIRMED` lifecycle defined in Section 3.2/3.3 does not apply here — there was no provisional alert. The alert lifecycle for CDS-only STAT findings is undefined.
3. Section 12.1 test ORC-02 covers Phase 1 STAT → CDS confirmation. There is no test for CDS-only STAT.

**Required action:** Add an explicit CDS-only STAT alert lifecycle to Section 3. When `stat_fast_path_fired = false` and CDS output contains `stat_flag = true`, the orchestrator fires a `STAT_CONFIRMED` (not `STAT_PROVISIONAL`) alert directly — skipping the provisional state. Add test ORC-12: CDS issues new STAT (left main pattern) not flagged by any Phase 1 agent — verify STAT_CONFIRMED fires, timing reflects CDS completion, report status is correct.

### 8d — `beat_class` contains only class 4 (all noise)

**Rating: REQUIRED**

If all beats in the ECG have `beat_class = 4` (noise), the feature object is semantically empty. The Phase 0 validation does not check for this — it only checks field presence. The agents would receive a feature object with all beat-level measurements populated (possibly with garbage values from attempted noise beat analysis) but with `beat_class = [4, 4, 4, ...]`.

The agents may produce findings on garbage data — which is worse than no findings, because the findings appear legitimate in the report.

**Required action:** Add to Phase 0 validation: if `beat_class` array has no beats with class 0, 1, 2, or 3 (i.e., `total_beats_analyzed = 0` or all beats are class 4), return `PIPELINE_FAILED` with error `"NO_ANALYZABLE_BEATS"`. This is a SDA-1 failure, not a clinical finding. The UI should display "Signal too noisy for automated analysis — manual review required."

## Issue 9 — CDS Base Fields Do Not Match Feature Schema

**Rating: CRITICAL**

Section 5.2 defines the 12 CDS base fields "always included" in the raw feature selection. Several of these field names do not exist in the Node 2.2 feature schema (Section 1.3).

| Field in Node 2.3 Section 5.2 | Actual field name in Node 2.2 Section 1.3 |
|-------------------------------|-------------------------------------------|
| `aggregate.heart_rate_bpm` | `aggregate.heart_rate_ventricular_bpm` |
| `aggregate.rr_variability_ms` | `aggregate.rhythm_regularity_coefficient` (dimensionless ratio, not milliseconds) |
| `aggregate.dominant_rhythm` | **DOES NOT EXIST** — no `dominant_rhythm` field in the Node 2.2 aggregate schema |
| `aggregate.mean_qrs_axis_degrees` | `aggregate.qrs_axis_degrees` |
| `aggregate.lbbb_present` | **DOES NOT EXIST** — LBBB detection is in RRC Agent output (`conduction.bbb_type`), not in the SDA-1 feature object |
| `aggregate.rbbb_present` | **DOES NOT EXIST** — same as above |

Additionally, there is a key-casing inconsistency between Node 2.3 (which uses uppercase agent keys: `"RRC"`, `"IT"`, `"MR"` in Section 5.4) and Node 2.2 Section 6.3 (which uses lowercase keys: `"rrc"`, `"it"`, `"mr"` in the Phase 2 handoff payload). This will cause key lookup failures at runtime if not resolved.

**This is a CRITICAL defect.** The orchestrator will attempt to look up `aggregate.lbbb_present` in the feature object at runtime and find nothing. The base feature section will either silently omit these fields (undetected), include `null` values without the CDS prompt being designed for nulls, or crash. For `dominant_rhythm` in particular: CDS is likely using this field in its rhythm classification reasoning. If it arrives as null, CDS may produce incorrect rhythm-dependent cross-domain inferences.

**Required actions:**

(a) Update Section 5.2 base fields to match the exact field names in Node 2.2 Section 1.3:
- Replace `aggregate.heart_rate_bpm` → `aggregate.heart_rate_ventricular_bpm`
- Replace `aggregate.rr_variability_ms` → `aggregate.rhythm_regularity_coefficient` (and note it is a dimensionless coefficient, not ms)
- Remove `aggregate.dominant_rhythm` — this field does not exist; add `aggregate.av_ratio` and `aggregate.beat_to_beat_qrs_morphology_stable` instead if CDS needs rhythm context
- Replace `aggregate.mean_qrs_axis_degrees` → `aggregate.qrs_axis_degrees`
- Remove `aggregate.lbbb_present` and `aggregate.rbbb_present` — these are agent outputs, not SDA-1 features; CDS gets BBB status from the RRC Phase 1 output directly

(b) Resolve key casing: pick uppercase (`"RRC"`, `"IT"`, `"MR"`) or lowercase (`"rrc"`, `"it"`, `"mr"`) and apply it consistently in both Node 2.2 Section 6.3 and Node 2.3 Sections 5.2 and 5.4. This requires an amendment to Node 2.2.

(c) Add a Phase 0 validation step to verify all 12 base fields (with corrected names) are present before agent dispatch.
