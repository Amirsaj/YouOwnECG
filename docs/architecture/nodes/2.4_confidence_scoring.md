## Executive Summary

Node 2.4 defines the confidence scoring system for all diagnostic findings produced by the pipeline. The problem being solved: a raw confidence label from an LLM agent ("HIGH confidence: STEMI") is not a calibrated probability — it is a verbalized estimate that may be systematically overconfident or underconfident depending on pattern, context, and model behavior. This node specifies how to compute, combine, and communicate confidence in a clinically defensible way.

Three invariants govern this design:

1. **Confidence is per-finding, not per-ECG.** An ECG can have HIGH confidence on its rhythm finding and LOW confidence on its ST interpretation simultaneously.
2. **Confidence scores are descriptive, not thresholds for action.** The pipeline never withholds a finding because confidence is low — it reports the finding with its confidence. The clinician decides what to act on.
3. **When confidence is insufficient to make a finding, `INSUFFICIENT_EVIDENCE` is reported instead of a low-confidence finding.** This distinction matters: LOW confidence = "I think X but I'm not sure." INSUFFICIENT_EVIDENCE = "I don't have enough data to comment on X."

## 1. What Does Confidence Mean in Medical AI?

### 1.1 Calibration vs Accuracy

A classifier that says "90% confidence" on every prediction is well-calibrated if 90% of those predictions are correct. If 70% are correct, the classifier is overconfident. LLM verbalized confidence is frequently poorly calibrated — particularly for rare conditions and edge cases.

**The pipeline's confidence problem:**
The DeepSeek reasoner agents produce verbalized confidence as part of their structured JSON output: `"confidence": "HIGH" | "MEDIUM" | "LOW" | "INSUFFICIENT_DATA"`. This is a qualitative estimate derived from the reasoning trace. It is:
- Sensitive to how features are presented (confident presentation → overconfident LLM)
- Biased toward HIGH for common conditions, LOW for rare ones
- Not anchored to an empirical accuracy baseline

**Solution: Three-source confidence fusion.** The final per-finding confidence is computed from three independent sources, each providing a different signal:
1. **Signal confidence** — How clear is the signal evidence for this finding?
2. **Agent confidence** — How confident is the reasoning agent?
3. **Corroboration confidence** — Do multiple independent sources agree?

### 1.2 LLM Confidence Estimation Literature (2024–2026)

Three methods are documented for eliciting calibrated confidence from LLMs:

**Verbalized confidence** (used here): Ask the model to state HIGH/MEDIUM/LOW. Simple, consistent, but tends toward overconfidence. Mitigated by requiring the model to also state `confidence_rationale` — models that are forced to articulate why they're confident become more conservative.

**Logprob-based confidence**: Use the log-probability of the output tokens as a proxy for confidence. More principled but requires access to token log-probabilities. DeepSeek API provides `logprobs` in the response — usable. However, for long structured outputs, the relevant logprobs are buried in the JSON field values, requiring extraction. Not used as primary method but noted as a validation tool for VT-2.4-01.

**Consistency-based confidence** (temperature sampling): Run the same prompt N times with temperature > 0, count how often the same finding is produced. Expensive (N × latency, N × cost) and not appropriate for a real-time ER pipeline. Excluded.

**Conclusion:** Verbalized confidence with calibration modifiers is the chosen approach. The modifiers are the three confidence sources defined below.

## 2. Signal Confidence

Signal confidence measures how strongly the raw signal supports a finding, independent of any LLM reasoning.

### 2.1 Signal Confidence Components

**Lead quality modifier:**
```python
def lead_quality_modifier(lead: str, quality_report: QualityReport) -> float:
    """
    Returns a multiplier [0.0, 1.0] based on the quality of the lead relevant
    to the finding. Poor quality leads reduce confidence in findings that depend on them.
    """
    label = quality_report.lead_labels.get(lead, "UNUSABLE")
    return {"GOOD": 1.0, "ACCEPTABLE": 0.85, "POOR": 0.5, "UNUSABLE": 0.0}[label]
```

**Signal magnitude margin:**
For threshold-based findings (ST elevation, QRS duration, QTc), the margin between the measured value and the threshold determines signal confidence:

```python
def st_elevation_signal_confidence(
    st_mv: float,
    threshold_mv: float,
    lead_quality: float,
) -> float:
    """
    Returns [0.0, 1.0]. The further above threshold, the higher the signal confidence.
    """
    if st_mv <= 0:
        return 0.0
    margin = st_mv - threshold_mv
    # Normalize: 0 at threshold, 1.0 at 2× threshold margin
    raw = min(1.0, max(0.0, margin / threshold_mv))
    return raw * lead_quality
```

**Multi-lead corroboration (signal-level):**
STEMI requires elevation in ≥ 2 contiguous leads. The signal confidence increases with the number of contiguous leads showing the finding:

```python
CONTIGUOUS_LEAD_GROUPS = {
    "anterior":  ["V1", "V2", "V3", "V4"],
    "inferior":  ["II", "III", "aVF"],
    "lateral":   ["I", "aVL", "V5", "V6"],
    "posterior": ["V1", "V2"],  # depression pattern
}

def contiguous_lead_count_bonus(affected_leads: list[str], territory: str) -> float:
    group = CONTIGUOUS_LEAD_GROUPS.get(territory, [])
    n_affected = sum(1 for l in affected_leads if l in group)
    if n_affected == 0: return 0.0
    if n_affected == 1: return 0.4    # single lead — possible but weak
    if n_affected == 2: return 0.75   # minimum criterion met
    return 1.0                         # 3+ leads — strong signal confidence
```

### 2.2 Signal Confidence Levels

Signal confidence is bucketed into the same HIGH/MEDIUM/LOW/INSUFFICIENT_EVIDENCE scale used throughout:

```
signal_confidence_score → level
>= 0.75  → HIGH
0.40–0.74 → MEDIUM
0.15–0.39 → LOW
< 0.15   → INSUFFICIENT_EVIDENCE
```

## 3. Agent Confidence

Agent confidence is the verbalized confidence from the reasoning agent's JSON output, post-processed through a calibration modifier.

### 3.1 Agent Verbalized Confidence → Score Mapping

The raw verbalized confidence from the agent is mapped to a numeric score:

```python
AGENT_CONFIDENCE_SCORE = {
    "HIGH":               0.85,
    "MEDIUM":             0.60,
    "LOW":                0.35,
    "INSUFFICIENT_DATA":  0.10,
}
```

### 3.2 Calibration Modifier

The numeric score is adjusted based on the finding type. LLMs are known to be better calibrated for some finding types than others:

```python
FINDING_CALIBRATION_FACTOR = {
    # Well-calibrated: large visual signatures, unambiguous criteria
    "lbbb":             0.95,
    "rbbb":             0.95,
    "afib":             0.90,
    "st_elevation":     0.90,
    "qrs_wide":         0.90,
    "complete_avb":     0.90,
    # Moderate calibration: threshold-sensitive, some variability
    "lvh":              0.75,
    "qtc_prolonged":    0.80,
    "first_degree_avb": 0.80,
    # Lower calibration: subtle patterns, rare conditions
    "brugada_type2":    0.65,
    "de_winter":        0.70,
    "wellens":          0.70,
    "epsilon_wave":     0.60,
    "metabolic":        0.65,
    # Default
    "_default":         0.75,
}

def calibrated_agent_score(raw_verbal: str, finding_type: str) -> float:
    base = AGENT_CONFIDENCE_SCORE[raw_verbal]
    factor = FINDING_CALIBRATION_FACTOR.get(finding_type,
                                             FINDING_CALIBRATION_FACTOR["_default"])
    return base * factor
```

**Calibration factor rationale:** A LBBB diagnosis from an LLM given `qrs_duration_global_mean_ms = 138, broad_notched_R_in_I_V6 = True, rS_in_V1_V2 = True` is nearly algorithmic — the LLM is just applying a checklist. A Brugada Type 2 diagnosis from borderline features is inherently subjective even for cardiologists. The calibration factor corrects for this known variance.

## 4. Corroboration Confidence

Corroboration confidence measures agreement across independent sources. Agreement increases confidence; disagreement reduces it.

### 4.1 Sources of Corroboration

For each finding, the following independent sources are checked:

| Source | Available when |
|--------|---------------|
| Signal pipeline (Node 1.5) | Always |
| VL2 visual (Node 1.6) | When `vl2_call_succeeded = True` |
| Cross-agent agreement | When > 1 agent reports the same finding |
| RAG textbook evidence | When RAG retrieval returns relevant citations |

### 4.2 Corroboration Score

```python
def corroboration_score(
    signal_flagged: bool,
    vl2_flagged: bool | None,       # None = VL2 unavailable or CANNOT_DETERMINE
    cross_agent_agreement: bool | None,  # True if another agent independently flags same finding
    rag_strong_support: bool,       # True if RAG returned high-relevance citation supporting finding
) -> float:
    """
    Returns corroboration bonus [0.0, 0.3] to be added to combined confidence.
    No single source provides more than 0.1 bonus to prevent over-weighting.
    """
    bonus = 0.0
    if vl2_flagged is True:          bonus += 0.10
    if cross_agent_agreement is True: bonus += 0.10
    if rag_strong_support:            bonus += 0.10
    # Penalize if VL2 is HIGH confidence and DISAGREES
    if vl2_flagged is False:         bonus -= 0.10
    return max(-0.15, min(0.30, bonus))
```

**Cross-agent agreement example:** IT agent detects STEMI anterior. RRC agent independently reports "regular tachycardia — possible STEMI" in `cross_domain_hints`. This is cross-agent agreement → +0.10 corroboration.

## 5. Combined Confidence Score and Level

### 5.1 Combination Formula

```python
def combine_confidence(
    signal_score: float,
    agent_score_calibrated: float,
    corroboration_bonus: float,
    lead_quality: float,
) -> float:
    """
    Weighted combination: signal and agent are equal partners;
    corroboration is a bonus; lead quality is a hard cap.
    """
    raw = 0.50 * agent_score_calibrated + 0.50 * signal_score + corroboration_bonus
    # Lead quality is a hard cap — no matter how good the reasoning,
    # a finding in an UNUSABLE lead cannot be HIGH confidence
    return min(raw, lead_quality)
```

### 5.2 Combined Score → Level

```python
def score_to_level(score: float) -> str:
    if score >= 0.75: return "HIGH"
    if score >= 0.45: return "MEDIUM"
    if score >= 0.20: return "LOW"
    return "INSUFFICIENT_EVIDENCE"
```

### 5.3 STAT Override

For STAT-level findings (STEMI, Brugada Type 1, VT, hyperK sine wave, complete AVB), the pipeline applies a **conservative STAT override rule**:

```
If stat_threshold_met AND combined_confidence == "LOW":
    → Report as "LOW" confidence STAT finding, not suppressed.
    → stat_alerts[] still fires.
    → UI shows: "POSSIBLE STEMI — LOW CONFIDENCE — IMMEDIATE REVIEW REQUIRED"
```

**ABSOLUTE INVARIANT: STAT findings are NEVER suppressed regardless of confidence level — including INSUFFICIENT_EVIDENCE.** A low-confidence STEMI alert in a noisy ECG is infinitely preferable to a missed STEMI. The confidence level informs the clinician's response; it is NEVER used as a suppression criterion.

```python
if stat_threshold_met:
    # Fires on ALL confidence levels including INSUFFICIENT_EVIDENCE
    stat_alert_fires = True
    if combined_confidence in ("LOW", "INSUFFICIENT_EVIDENCE"):
        alert_message = (
            f"POSSIBLE {finding_type.upper()} — {combined_confidence} CONFIDENCE"
            + (" — SIGNAL QUALITY IMPAIRED" if lead_quality_cap < 0.5 else "")
            + " — IMMEDIATE REVIEW REQUIRED"
        )
    # combined_confidence level is preserved for display — not changed to HIGH
```

Signal quality impairment is communicated explicitly in the alert message. The confidence level is never upgraded to suppress the impairment information.

## 6. Per-Finding Confidence Schema

Every finding in the final diagnostic output carries:

```python
@dataclass
class FindingConfidence:
    level: str                       # "HIGH" | "MEDIUM" | "LOW" | "INSUFFICIENT_EVIDENCE"
    score: float                     # numeric [0.0, 1.0]
    signal_score: float              # signal pipeline contribution
    agent_score: float               # calibrated agent contribution
    corroboration_bonus: float       # cross-source agreement contribution
    lead_quality_cap: float          # effective quality cap applied
    sources_agree: list[str]         # which independent sources confirmed this finding
    sources_disagree: list[str]      # which sources did not confirm (or contradicted)
    confidence_rationale: str        # verbatim from agent reasoning output
    insufficient_evidence_reason: str | None  # why INSUFFICIENT_EVIDENCE, if applicable
```

## 7. Uncertainty Communication

### 7.1 ER Nurse / Triage Display

Confidence is shown as a color-coded badge only:
- HIGH → green badge: "HIGH CONFIDENCE"
- MEDIUM → yellow badge: "MODERATE CONFIDENCE"
- LOW → orange badge: "LOW CONFIDENCE — VERIFY"
- INSUFFICIENT_EVIDENCE → grey: "INSUFFICIENT DATA"

Numeric scores are NOT shown to ER nurses. A number like "0.63 confidence" communicates nothing clinically; "MODERATE CONFIDENCE" does.

### 7.2 Cardiologist / Advanced Display

Full breakdown:
- Numeric score (e.g., 0.71)
- Breakdown by source (signal: 0.80, agent: 0.62, corroboration: +0.10)
- Sources that agreed (e.g., "VL2 confirmed; RRC agent cross-domain hint")
- Sources that disagreed (e.g., "VL2 did not flag ST elevation in V2")
- Agent confidence rationale (verbatim from reasoning trace)

### 7.3 When to Report INSUFFICIENT_EVIDENCE

INSUFFICIENT_EVIDENCE is returned (instead of a LOW finding) when:

| Condition | Reason |
|-----------|--------|
| The lead(s) required for this finding are UNUSABLE | `lead_quality_cap = 0.0` |
| Combined score < 0.20 | No meaningful signal from any source |
| Agent returns `"confidence": "INSUFFICIENT_DATA"` with rationale "cannot assess X" | Agent explicitly declined |
| Feature is `null` in the feature object (not computed) | Node 1.5 could not compute the required measurement |

INSUFFICIENT_EVIDENCE findings are included in the output but displayed separately from positive/negative findings. They represent known gaps, not missed findings.
