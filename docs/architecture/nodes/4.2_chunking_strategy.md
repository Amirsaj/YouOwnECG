## 1. Chunking Goal

Chunks must be:
1. **Self-contained** — retrievable as a standalone answer to a clinical question without requiring the adjacent chunk to make sense
2. **Semantically bounded** — not split mid-sentence or mid-paragraph
3. **Appropriately sized** — large enough to carry context, small enough to stay within embedding model token limits and to rank precisely

The failure mode of over-chunking: "ST elevation" retrieved from a sentence stripped of its subject ("In anterior STEMI..."). The failure mode of under-chunking: 4,000-word chapter sections retrieved when only one paragraph is relevant.

## 2. Chunk Unit: Section-Level with Overflow Split

### 2.1 Primary Unit

The primary chunk unit is a **book section** (a SECTION_HEADER block and all following BODY blocks until the next SECTION_HEADER or CHAPTER_HEADER). This preserves clinical reasoning — a section discussing "ST elevation in STEMI" contains the clinical criteria, caveats, and differential all together.

Target chunk size: **300–800 tokens** (GPT-4 tokenizer estimate; DeepSeek uses similar BPE).

### 2.2 Overflow Split

When a section exceeds 800 tokens, it is split at paragraph boundaries (blank line or new BODY block following a sentence-ending period):

```python
MAX_CHUNK_TOKENS = 800
MIN_CHUNK_TOKENS = 150  # below this, merge with adjacent chunk

def split_section_into_chunks(
    section_text: str,
    metadata: dict,
    tokenizer,
) -> list[dict]:
    """
    Splits a section into chunks of MAX_CHUNK_TOKENS or fewer,
    splitting only at paragraph boundaries.

    Returns list of chunk dicts with text + metadata.
    """
    paragraphs = section_text.split("\n\n")
    chunks = []
    current_text = ""
    current_tokens = 0

    for para in paragraphs:
        para_tokens = len(tokenizer.encode(para))
        if current_tokens + para_tokens > MAX_CHUNK_TOKENS and current_tokens > 0:
            if current_tokens >= MIN_CHUNK_TOKENS:
                chunks.append({**metadata, "text": current_text.strip()})
            current_text = para
            current_tokens = para_tokens
        else:
            current_text = current_text + "\n\n" + para if current_text else para
            current_tokens += para_tokens

    if current_text.strip():
        if current_tokens < MIN_CHUNK_TOKENS and chunks:
            # Merge undersized tail into last chunk
            chunks[-1]["text"] += "\n\n" + current_text.strip()
        else:
            chunks.append({**metadata, "text": current_text.strip()})

    return chunks
```

### 2.3 Overlap

**No sliding window overlap.** Section-level chunking inherently includes context through metadata (book, chapter, section). Sliding overlap would duplicate sentences across chunks and inflate the vector store without improving recall — retrieval tests on medical text show metadata-anchored chunking outperforms overlap-based chunking for clinical Q&A (Xiong et al., 2024).

Instead, overlap is handled at retrieval time by also returning the adjacent chunk (±1) when a chunk scores above a secondary threshold (see Node 4.4).

## 3. Figure Chunk Handling

Each classified figure becomes its own chunk with the following structure:

```python
{
    "chunk_type": "FIGURE",
    "book": "Chou's Electrocardiography",
    "book_short": "chous",
    "chapter": "Chapter 5: Myocardial Infarction",
    "section": "5.2 ST Segment Changes",
    "page_number": 142,
    "figure_type": "ECG_STRIP",      # ECG_STRIP / DIAGRAM / TABLE / PHOTO
    "caption": "Figure 5.3. Anterior STEMI with hyperacute T waves in V2–V4...",
    "in_text_refs": ["Fig. 5.3"],    # paragraphs that cite this figure
    "image_path": "figures/chous/p142_fig0.png",  # path to extracted image
    "text": "Figure 5.3. Anterior STEMI with hyperacute T waves in V2–V4...",  # caption as text for embedding
    "isolated_figure": False,
}
```

**ECG_STRIP figures** are embedded using caption text only (vision embedding in Node 4.3 handles image embedding separately). Other figure types use caption text only.

**Isolated figures** (no caption, no adjacent text): embedded with `"text": f"Uncaptioned {figure_type} figure from {book}, page {page_number}"`. These are low-relevance candidates but retained.

## 4. Metadata Schema per Chunk

Every chunk (text or figure) carries:

```python
{
    # Identity
    "chunk_id": "chous_ch05_s5.2_p142_c0",   # book_short + chapter_idx + section_idx + page + chunk_idx
    "chunk_type": "TEXT",                      # TEXT or FIGURE

    # Provenance
    "book": "Chou's Electrocardiography in Clinical Practice",
    "book_short": "chous",
    "chapter": "Chapter 5: Myocardial Infarction",
    "chapter_number": 5,
    "section": "5.2 ST Segment Changes",
    "section_number": "5.2",
    "page_number": 142,

    # Content
    "text": "...",                             # text to embed
    "token_count": 412,

    # Figure-specific (TEXT chunks: null)
    "figure_type": None,
    "image_path": None,
    "caption": None,
    "isolated_figure": None,

    # Adjacent chunks (for context expansion at retrieval)
    "prev_chunk_id": "chous_ch05_s5.2_p141_c0",
    "next_chunk_id": "chous_ch05_s5.2_p143_c0",
}
```

`chunk_id` is deterministic (no UUID): same book + chapter + section + page + chunk index always produces the same ID. This enables deduplication when books are re-ingested.
