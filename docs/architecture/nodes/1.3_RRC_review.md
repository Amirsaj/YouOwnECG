# RRC Adversarial Review — Node 1.3 Quality Assessment (FD-FPRE)

**Reviewed by:** Research Review Committee (RRC)
**Document under review:** `docs/architecture/nodes/1.3_quality_assessment.md`
**Review date:** 2026-03-29
**Verdict:** GATE CONDITIONAL

---

## Preamble

Node 1.3 is the gatekeeper of the pipeline. A quality module that rejects good recordings wastes clinical time; one that passes bad recordings causes wrong diagnoses. The RRC raises two issues — one CRITICAL and one REQUIRED.

---

## Issue 1 — SNR Estimation Requires R-Peak Detection, Which Requires Quality Assessment: Circular Dependency

**Severity: CRITICAL**

**Location:** Section 1.1, `estimate_snr_db()`:

```python
def estimate_snr_db(signal_1d, fs, r_peaks):
    ...
```

And the pipeline entry in Section 1.3:

> "Beat-level quality (populated after rough R-peak detection)"

**The flaw:**

The SNR estimation method (Section 1.1) requires R-peak positions as input. R-peak detection requires a preprocessed signal of sufficient quality. But quality assessment is supposed to precede R-peak detection (Node 1.4) in the pipeline order. The document states this is "rough detection" — but rough detection still fails on flatlined or extremely noisy leads, and the function signature places the burden of R-peak supply on the caller without specifying where those peaks come from.

Concretely: who calls `estimate_snr_db()`, and what provides `r_peaks`? If Node 1.3 performs its own rough R-peak detection to bootstrap SNR, then Node 1.3 has an embedded signal processing step that is not specified and not validated. If it uses R-peaks from Node 1.4 (ECGdeli), then Node 1.4 must run before Node 1.3, which violates the pipeline order (Node 1.3 gates Node 1.4).

The document does not resolve this. The quality assessment framework cannot depend on ECGdeli output, because ECGdeli is downstream. Node 1.3 must either use an internal rough detector or use a quality metric that does not require beat positions.

**Required action:** (1) Specify the bootstrap R-peak detection method used internally by Node 1.3 (e.g., simple threshold on smoothed rectified signal — Pan-Tompkins derivative step only). This internal detector is not clinical-grade — it is only used for SNR estimation and per-beat quality scoring. (2) Explicitly state this detector is internal to Node 1.3 and its output is NOT the `fpt` fed to ECGdeli. ECGdeli performs its own full detection from scratch. (3) Add the internal bootstrap detector specification to the document.

## Issue 2 — `REQUIRED_LEADS["ischemia"]` Omits Inferior Leads Required for Inferior STEMI Detection

**Severity: REQUIRED**

**Location:** Section 2.2:

```python
REQUIRED_LEADS = {
    ...
    "ischemia": ["II", "V1", "V2", "V3", "V4", "V5", "V6"],
    ...
}
```

**The flaw:**

STEMI territory assignment (per the IT agent's disease knowledge base) includes:
- Inferior STEMI (RCA or LCx): ST elevation in II, III, aVF
- Lateral STEMI (LCx): ST elevation in I, aVL
- Posterior STEMI: ST depression in V1–V4

The `ischemia` required lead set includes `II` but omits `III` and `aVF`. These three leads together define the inferior territory — detecting ST elevation in II alone while `III` and `aVF` are missing means inferior STEMI cannot be confirmed or localized. Similarly, `I` and `aVL` are absent from the ischemia set, making high-lateral STEMI detection impossible even if all limb leads are available.

The IT agent (Node 2.2) requires all 12 leads for complete ischemia assessment. The minimum required set for an ischemia-capable analysis should include all leads where STEMI patterns are expected: II, III, aVF (inferior), I, aVL (lateral), V1–V6 (anterior/posterior).

**Required action:** Update `REQUIRED_LEADS["ischemia"]` to include `["I", "II", "III", "aVL", "aVF", "V1", "V2", "V3", "V4", "V5", "V6"]` (all 12 leads). If any of these are unavailable, the ischemia domain proceeds with degraded coverage and the IT agent is notified which territories cannot be assessed.

## Verdict

**GATE CONDITIONAL**

Node 1.3 is well-structured. The fail-loud principle, per-lead granularity, and per-beat quality scoring are the right design. The `QualityReport` dataclass is a clean interface. The monitoring-mode detection is a valuable safety feature not seen in most open-source pipelines.

The CRITICAL issue (circular dependency with R-peak detection) must be resolved — it is a specification gap that will cause confusion at implementation time. The gate opens when both issues are addressed.
